# Domain Resolution Services Methodology
**Date:** October 10, 2025  
**Component:** Domain Resolution System  
**Version:** 2.0  

---

## üìã Overview

This document provides comprehensive methodology and reference documentation for the cryptocurrency domain resolution system. The system enables resolution of human-readable domain names to blockchain addresses and reverse lookup of addresses to domain names across multiple blockchain naming services.

---

## üéØ Supported Services

### ENS (Ethereum Name Service)
- **File**: `src/components/domains/ens.ts`
- **Supported Domains**: `.eth`
- **Supported Currencies**: ETH (Ethereum only)
- **Features**:
  - Forward resolution: domain ‚Üí address
  - Reverse resolution: address ‚Üí domain
  - Address validation

### UNS (Unstoppable Domains)
- **File**: `src/components/domains/uns.ts`
- **Supported Domains**: `.crypto`, `.nft`, `.blockchain`, `.bitcoin`, `.coin`, `.wallet`, `.888`, `.dao`, `.x`, `.zil`
- **Supported Currencies**: Multiple cryptocurrencies (ETH, BTC, etc.)
- **Features**:
  - Forward resolution: domain ‚Üí address
  - Multi-chain address resolution
  - ERC20 token support

### Tezos Domains
- **File**: `src/components/domains/tez.ts`
- **Supported Domains**: `.tez`
- **Supported Currencies**: XTZ (Tezos)
- **Features**:
  - Forward resolution: domain ‚Üí address
  - Reverse resolution: address ‚Üí domain
  - GraphQL API integration

### Solana Name Service (SNS)
- **File**: `src/components/domains/sol.ts`
- **Supported Domains**: `.sol`
- **Supported Currencies**: SOL (Solana)
- **Features**:
  - Forward resolution: domain ‚Üí address
  - Reverse resolution: address ‚Üí domain (favorite domain)
  - HTTP API + RPC fallback
  - Multiple domain ownership handling

### Algorand NF Domains (NFD)
- **File**: `src/components/domains/algo.ts`
- **Supported Domains**: `.algo`
- **Supported Currencies**: ALGO (Algorand)
- **Features**:
  - Forward resolution: domain ‚Üí address
  - Reverse resolution: address ‚Üí domain
  - Multi-endpoint fallback strategy
  - Mainnet + Testnet support

## üìö Usage Guide

### Basic Import

```typescript
import { 
  ensResolver, 
  unsResolver, 
  tezosDomainsResolver,
  solanaDomainsResolver,
  algoDomainsResolver 
} from '@/components/domains';
```

### Forward Resolution (Domain ‚Üí Address)

```typescript
// ENS Example
const ensAddress = await ensResolver.getAddress({
  domain: 'vitalik.eth',
  coinTicker: 'ETH'
});

// Unstoppable Domains Example
const unsAddress = await unsResolver.getAddress({
  domain: 'example.crypto',
  coinTicker: 'ETH'
});

// Tezos Domains Example
const tezAddress = await tezosDomainsResolver.getAddress({
  domain: 'alice.tez',
  coinTicker: 'XTZ'
});

// Solana Name Service Example
const solAddress = await solanaDomainsResolver.getAddress({
  domain: 'bonfida.sol',
  coinTicker: 'SOL'
});

// Algorand NF Domains Example
const algoAddress = await algoDomainsResolver.getAddress({
  domain: 'myname.algo',
  coinTicker: 'ALGO'
});
```

### Reverse Resolution (Address ‚Üí Domain)

```typescript
// ENS Reverse Lookup
const ensDomain = await ensResolver.resolveDomainForAddress(
  '0xd8dA6BF26964aF9D7eEd9e03E53415D37aA96045'
);
// Returns: 'vitalik.eth' or null

// Tezos Domains Reverse Lookup
const tezDomain = await tezosDomainsResolver.resolveDomainForAddress(
  'tz1VSUr8wwNhLAzempoch5d6hLRiTh8Cjcjb'
);
// Returns: 'alice.tez' or null

// Solana Name Service Reverse Lookup
const solDomain = await solanaDomainsResolver.resolveDomainForAddress(
  'HN7cABqLq46Es1jh92dQQisAq662SmxELLLsHHe4YWrH'
);
// Returns: 'bonfida.sol' or null

// Algorand NF Domains Reverse Lookup
const algoDomain = await algoDomainsResolver.resolveDomainForAddress(
  'OGAS35GSNV4C6PS4YOZJWCT4VSAVCJKZZWU2XONPLM5BVYHYJR5IW3E5ZM'
);
// Returns: 'myname.algo' or null
```

### Combined Resolution (Try Multiple Services)

```typescript
import { resolveAddress, resolveDomainForAddress } from '@/components/domains';

// Try multiple services automatically
const result = await resolveAddress('example.crypto', 'ETH');
console.log(result.address); // '0x...'
console.log(result.service); // 'ENS' | 'UNS' | 'TEZ' | 'SOL' | 'ALGO'

// Reverse lookup with multiple services
const reverseResult = await resolveDomainForAddress('0x...', 'ETH');
console.log(reverseResult?.domain); // 'example.eth'
console.log(reverseResult?.service); // 'ENS' | 'UNS' | 'TEZ' | 'SOL' | 'ALGO'
```

### Domain Validation

```typescript
// Check if a string is an ENS domain
if (ensResolver.isEnsDomain('vitalik.eth')) {
  // It's an ENS domain

// Check if a string is an Unstoppable Domain

  // It's a Tezos Domain
}

// Check if a string is a Solana Name Service domain

  // It's a Solana domain
}

// Check if a string is an Algorand NF Domain
if (algoDomainsResolver.isAlgoDomain('myname.algo')) {
  // It's an Algorand NFD
}

// Check if a string is any supported domain

  // It's a supported domain from any service
}
```

## üîß Integration with TemporaryKeyModal

The domain resolution is automatically integrated into the wallet import flow:

1. User imports a private key
2. Wallet address is derived from the private key
3. **Automatic domain resolution** attempts to find associated domain names
4. If found, displays domain name with service badge (ENS/UNS/TEZ/SOL/ALGO)
5. User can copy both address and domain name
6. User can click domain icon to view on respective domain service

## üåê Environment Variables

### Infura Project ID (Optional)
The ENS resolver uses ethers' default provider (multiple public endpoints) by default. For better performance and higher rate limits, you can optionally set your own Infura key:

```bash
VITE_INFURA_PROJECT_ID=your_project_id_here
```

### Alchemy API Key (Optional)
Alternative to Infura for ENS resolution:

```bash
VITE_ALCHEMY_API_KEY=your_alchemy_api_key_here
```

**Default behavior**: Uses `ethers.getDefaultProvider('mainnet')` which automatically tries multiple public RPC endpoints:
- Cloudflare Ethereum Gateway
- Ankr
- Public Infura nodes
- And more...

This provides better reliability than a single endpoint.

### Development Proxies
For services requiring CORS handling (NFD, Tezos Domains), Vite proxies are configured:

```typescript
// vite.config.ts
server: {
  proxy: {
    '/api/nfd': {
      target: 'https://api.nf.domains',
      changeOrigin: true,
      rewrite: (path) => path.replace(/^\/api\/nfd/, '/nfd'),
    },
    '/api/testnet-nfd': {
      target: 'https://api.testnet.nf.domains',
      changeOrigin: true,
      rewrite: (path) => path.replace(/^\/api\/testnet-nfd/, '/nfd'),
    },
  },
}
```

## ‚ö†Ô∏è Error Handling

All resolvers throw descriptive errors that can be caught and handled:

```typescript
try {
  const address = await ensResolver.getAddress({
    domain: 'invalid.eth',
    coinTicker: 'ETH'
  });
} catch (error) {
  console.error('Resolution failed:', error.message);
  // Handle error gracefully
}
```

### Error Handling Strategy

1. **Validation Errors**: Immediate rejection with localized message
2. **API Errors**: Try next endpoint in fallback chain (where applicable)
3. **Complete Failure**: Return `null` with informative console log
4. **Non-blocking**: Domain resolution never prevents wallet import

## ‚ö° Performance Considerations

### Caching
- **Unstoppable Domains**: 5-minute cache (TTL: 300s)
- **Tezos Domains**: GraphQL response caching
- **Future**: Local cache implementation planned

### Rate Limiting
- **ENS**: Public Infura endpoints are rate-limited
- **Solution**: Use custom Infura/Alchemy keys for higher limits
- **Fallback**: Default provider tries multiple endpoints automatically

### Non-blocking Resolution
- Domain resolution in TemporaryKeyModal is **non-blocking**
- Won't prevent wallet import if resolution fails
- Gracefully handles network errors and timeouts

### Multi-Endpoint Fallback (NFD)
NFD resolver tries multiple endpoints for reliability:
1. Mainnet `/v2/address` (primary)
2. Testnet `/v2/address` (fallback)
3. Mainnet `/v2/search` (alternative)
4. Testnet `/v2/search` (alternative)
5. Legacy `/lookup` (compatibility)

## üîç Service-Specific Details

### ENS (Ethereum Name Service)

**Resolution Method**: Direct ethers.js integration

```typescript
const provider = ethers.getDefaultProvider('mainnet');
const address = await provider.resolveName('vitalik.eth');
const name = await provider.lookupAddress(address);
```

**Validation**: Checks for `.eth` extension

**Limitations**: Ethereum addresses only

### Unstoppable Domains

**Resolution Method**: Official Resolution library

```typescript
const resolution = new Resolution();
const address = await resolution.addr('example.crypto', 'ETH');
```

**Multi-Chain Support**: Yes (60+ cryptocurrencies)

**Reverse Lookup**: Not currently supported by API

### Tezos Domains

**Resolution Method**: GraphQL API

```typescript
const query = `
  query {
    domain(name: "${domain}") {
      address
    }
  }
`;
```

**API Endpoint**: `https://api.tezos.domains/graphql`

**Features**: Clean GraphQL interface, fast response

### Solana Name Service

**Resolution Method**: HTTP API + RPC fallback

```typescript
// Primary: HTTP API for favorite domain
const response = await fetch(
  `https://sns-sdk-proxy.bonfida.workers.dev/favorite-domain/${address}`
);

// Fallback: Direct RPC calls
const domainKey = await getFavoriteDomain(connection, publicKey);
const { registry } = await NameRegistryState.retrieve(connection, domainKey);
```

**Multi-Domain Handling**: Returns favorite domain (user-selected)

**Fallback Strategy**: HTTP API ‚Üí RPC favorite ‚Üí RPC getAllDomains

### Algorand NF Domains

**Resolution Method**: REST API with multi-endpoint fallback

```typescript
const candidates = [
  `${NFD_API_BASE}/v2/address?address=${addr}`,
  `${NFD_TESTNET_API_BASE}/v2/address?address=${addr}`,
  `${NFD_API_BASE}/v2/search?owner=${addr}`,
  `${NFD_TESTNET_API_BASE}/v2/search?owner=${addr}`,
  `${NFD_API_BASE}/lookup?address=${addr}`,
];
```

**Address Normalization**: Converts to uppercase before validation

**Response Parsing**: Flexible parser handles multiple API formats

## üêõ Known Issues & Limitations

### Unstoppable Domains
- **Limitation**: Reverse lookup (address ‚Üí domain) not supported by API
- **Reason**: UNS architecture doesn't provide efficient reverse lookup
- **Workaround**: None currently available

### Solana Name Service
- **Limitation**: Only returns favorite domain (not all owned domains)
- **Reason**: API optimization for performance
- **Note**: If address owns multiple `.sol` domains, only favorite is shown

### Rate Limiting
- **Issue**: Public RPC endpoints have rate limits
- **Solution**: Use custom Infura/Alchemy keys
- **Impact**: Occasional resolution failures during high traffic

### CORS Restrictions
- **Issue**: Some APIs (NFD, Tezos) have CORS restrictions
- **Solution**: Vite proxy in development, direct calls in production
- **Note**: Production builds work via browser-to-API HTTPS

## üìû Support & Troubleshooting

### Common Issues

**Issue**: ENS resolution returns null
- **Check**: Environment variables (VITE_INFURA_PROJECT_ID)
- **Check**: Network connectivity
- **Check**: Domain is registered on mainnet

**Issue**: NFD returns 404
- **Expected**: Address doesn't have an NFD
- **Not a Bug**: This is normal behavior

**Issue**: CORS errors in development
- **Check**: Vite dev server is running
- **Check**: Proxy configuration in `vite.config.ts`
- **Solution**: Restart dev server

**Issue**: Translations not showing
- **Check**: `language` prop is passed to components
- **Check**: Translation keys exist in all locale files
- **Check**: `getTranslation()` is imported correctly

## üéì Best Practices

### For Developers

1. **Always validate input**: Use service-specific validation functions
2. **Handle null gracefully**: Not all addresses have domains
3. **Don't block UX**: Domain resolution should be non-blocking
4. **Use appropriate service**: Check ticker before calling resolver
5. **Log for debugging**: Comprehensive console logging helps troubleshooting
6. **Translate everything**: Use `getTranslation()` for all messages

### For Users

1. **Domain ownership**: Only registered domains will resolve
2. **Network requirements**: Domain resolution requires internet connectivity
3. **Service selection**: Different blockchains use different services
4. **Privacy**: Domain resolution may expose address ownership
5. **Favorite domains**: Solana shows favorite domain if multiple owned
