# Address Book Subsystem — Architecture & Reference

> **Date:** 2026-04-28 (filename: `04282026`)
> **Scope:** `src/views/AddressBook.vue`, `src/components/pageTabs/addressbook/`,
> `src/components/modals/addressbook/`, `src/lib/cores/displayStandard/`,
> `src/lib/cores/exportStandard/`, `src/lib/cores/importStandard/`
> **Audience:** Engineers extending the address book, debugging exports/imports,
> or onboarding to the codebase.

This document is a developer reference for the **Address Book** feature of
Sparkplate.fresh. It maps the directory layout to runtime behavior, names the
contracts between layers, and documents the conventions any new code in this
subsystem is expected to follow.

---

## 1. High-level architecture

The Address Book is a four-tab view backed by a small set of services and a
single shared modal stack:

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  src/views/AddressBook.vue   (parent view — owns lists, search, paging)     │
└──┬───────────────────────┬─────────────────────────┬───────────────────────┬┘
   │                       │                         │                       │
   ▼                       ▼                         ▼                       ▼
 Contacts               Exchanges                  Wallets                Companies
 (tab + modal)          (tab + modal)              (tab + modal)          (tab + modal)
   │
   ▼
 Per-row actions   ──►  exportStandard/* (CSV / VCF / JSON / MD / QR)
                   ─►  displayStandard/gravatar.ts (avatar)
                   ─►  displayStandard/generateContactVCardQrCode.ts (QR)
                   ─►  importStandard/contacts/fileImports.addressBook.contacts.csv.ts
                       (and importWallet.json.ts) for bulk file imports
```

Three layering rules are enforced by convention:

1. **Views own state, tabs are presentational.** `AddressBook.vue` holds the
   search query, pagination, sort key, selected-id sets, and load/persist
   wiring. The tab components are dumb tables over those props.
2. **Modals own their own edit drafts.** Each `modal.details.*` keeps its own
   `editedContact`/`draft` state and only emits `*-updated`/`*-deleted`
   events; the parent view re-loads from the service after the event.
3. **Imports/exports are pure modules.** Anything in
   `src/lib/cores/{import,export,display}Standard/` returns data or calls
   `download` directly — they never reach into Vue state.

## 2. `src/views/AddressBook.vue`

### Responsibilities

- Render the tab strip (Contacts / Exchanges / Wallets / Companies) and the
  toolbar (Add, Import, Export, Search).
- Own the **canonical lists**: `contacts`, `exchanges`, `wallets` (Companies
  is derived inside its tab from `getCompanies`).
- Own **selection state** per tab: `selectedContacts`, `selectedExchangeIds`,
  `selectedWalletIds`, `selectedCompanyIds`.
- Own **pagination/search/sort** for the Contacts tab (the only tab that
  paginates at the view level).
- Mount the modal stack (`modal.add.entry`, `modal.details.Contact`,
  `subModal.add.Currency`, `modal.confirm.delete.general`).
- Re-load lists after mutations by calling the matching `loadXxx()`.

### Add/Import/Export buttons

Defined by `addButton`, `importButton`, `exportButton` computed labels that
swap based on the active tab. Imports are funneled through a single
`<ImportButton>` that emits one of `contacts-imported`,
`companies-imported`, `wallets-imported`, `exchanges-imported`. Both
`contacts-imported` and `companies-imported` go through `addContacts()`
because companies are derived from contacts (see §6).

### `addContacts(newContacts)` contract

Critical method that ingests rows from
`fileImports.addressBook.contacts.csv.ts` (and from VCF/XLSX in
`useContactParser.ts`). It **conditionally spreads** optional fields so
phone, website, relationship, and social handles aren't silently dropped:

```ts
await addContact({
  type: 'regular',
  firstname, lastname, company, email, notes,
  ...(c.phone        ? { phone:        c.phone        } : {}),
  ...(c.website      ? { website:      c.website      } : {}),
  ...(c.relationship ? { relationship: c.relationship } : {}),
  ...(c.twitter      ? { twitter:      c.twitter      } : {}),
  ...(c.linkedin     ? { linkedin:     c.linkedin     } : {}),
  ...(c.instagram    ? { instagram:    c.instagram    } : {}),
  ...(c.facebook     ? { facebook:     c.facebook     } : {}),
})
```

If a future field is added to `Contact`, **add it here too** or the import
will appear to lose that column.

### Modal events

```
showAddContactModal       → AddContactModal (`modal.add.entry.vue`)
showContactDetailsModal   → ContactDetailsModal (`modal.details.Contact.vue`)
showAddCurrencyModal      → AddCurrencyModal (`subModal.add.Currency.vue`)
showConfirmModal          → ModalConfirmDeleteGeneral
```

All four are mounted unconditionally (with `:show=""`) so transitions work
correctly and we don't churn the DOM.

## 3. `src/components/pageTabs/addressbook/`

Four sibling files, all using the new naming convention:

| File | Purpose |
|---|---|
| `tab.addressBook.Contact.vue` | Paginated table of contacts; emits per-row export/QR events up to the view. |
| `tab.addressBook.Exchange.vue` | Standalone exchange records (own table, own delete confirm). |
| `tab.addressBook.Wallet.vue`   | Standalone wallet records (own table, own delete confirm, **own export pipelines** — does *not* emit to view). |
| `tab.addressBook.Company.vue`  | Companies table — companies are **derived** from contacts via `getCompanies()`, refreshed when contacts change. |

### Naming convention

```
tab.addressBook.<EntitySingular>.vue       (file)
TabAddressBook<EntitySingular>             (component name in defineOptions)
```

When renaming a legacy `*Tab.vue`, also update:
1. The `import` site in `AddressBook.vue` and `00.backup/AddressBookPage.Alt*.vue`.
2. The `defineOptions({ name })` value.
3. Any service/test referencing the old name.

### Two patterns for per-row actions

There are deliberately two patterns; pick by ownership of the data:

- **Bubble-up pattern (Contacts):** the tab `emit`s
  `generate-qrcode-png`, `export-csv`, etc. The view binds the handler and
  calls the appropriate exporter. Use this when the **view** knows enough
  about the record to do the export.
- **Local-handle pattern (Wallets, Companies):** the tab imports the
  exporter modules directly and calls them inline. Use this when the
  records live in the tab's local store (Wallets) or are derived in the tab
  itself (Companies).

The Company tab's `defineEmits` is intentionally trimmed to just
`update:selectedCompanyIds` and `select-all` because the bubble-up events
are no longer relevant.

### `ContactTableRow`

Exported by `tab.addressBook.Contact.vue`:

```ts
export interface ContactTableRow extends Contact {
  wallets: number   // count, not the array — the table only displays a count
}
```

The view widens `Contact` into `ContactTableRow` in `loadContacts()` by
calling `getWalletCountForContact(contact.id)` per row.

### Disabled columns

The Contacts table currently has the **Company** column commented out (both
the `<th>` at `tab.addressBook.Contact.vue:44–53` and the matching
`<td>{{ contact.company }}</td>`). The `colspan` on the empty-state row was
reduced from `8` → `7` to compensate. The column is deliberately disabled
in the UI; the underlying field is still imported, edited, persisted, and
exported. Re-enable by uncommenting both blocks and bumping `colspan` back
to `8`.

### `aspect.*` vs `subModal.*` vs `tab.details.*`

- **`aspect.*`** — non-modal sub-pieces composed into a parent modal. Always
  presentational with thin emits. Example: `aspect.socialMedia.vue` shows a
  read-only list and emits `edit-requested` when clicked.
- **`subModal.*`** — full-screen modals that pop *over* a parent modal.
  Always close back to their parent without unmounting it.
- **`tab.details.*`** — Radix Tabs panels rendered inside a details modal
  (e.g. the Notes / Wallets / GPG tabs of a contact).
- **`transformsFor.add.Entry/`** — alternative forms inside the single
  `modal.add.entry.vue`, picked by the active "entity" radio.

### Naming conventions inside `modals/addressbook/`

```
modal.<role>.<Entity>.vue         # primary modal
subModal.<purpose>.<Entity?>.vue  # nested modal
aspect.<purpose>.vue              # composable read-only aspect
tab.details.<Entity>.<Section>.vue
form.addEntry.<entity>.vue
```

Existing files use `submodal.*` (lowercase) and `subModal.*` (camelCase) —
both are accepted but new code should prefer `subModal.*` (camelCase) to
match the dominant pattern.

### Editable fields contract (`subModal.socialMedia.vue`)

`subModal.socialMedia.vue` exposes a `ContactInfoFields` interface:

```ts
interface ContactInfoFields {
  bio?: string
  company?: string   // populates the Company column on tab.addressBook.Contact
  phone?: string
  location?: string
  email?: string
  website?: string
  github?: string
  twitter?: string
  linkedin?: string
  instagram?: string
  bluesky?: string
  telegram?: string
}
```

It emits `save (fields)` and the parent (`modal.details.Contact.vue` /
`modal.details.Company.vue`) does
`{ ...editedContact, ...fields }` then `updateContact(...)`. **Adding a new
field**: add to the interface, add a row in the General or Social tab, add
to `syncDraft()`, and add the matching read-only row in
`aspect.socialMedia.vue` so the value displays once saved.

### `subModal.qrCode.WalletAddress.vue` ticker prefix toggle

Clicking the address text toggles `showTickerPrefix`. The QR is regenerated
from `fullAddress` (`<ticker>://<address>` vs raw `<address>`) so the QR
and visible string always match.

### `modal.details.Wallet.vue` action wiring

Header `ActionsDropdown` calls **local** export functions
(`exportWalletQrPng`, etc.) over `currentWalletRecord` — a computed
`StandaloneWalletRecord` derived from props. **Do not** add bubble-up emits
for these actions; the wallet tab does the same.

## 5. `src/lib/cores/displayStandard/`

Pure runtime utilities for the *display* layer. They produce strings/URLs
and never reach into storage.

### `gravatar.ts`

- `md5(str)` — RFC 1321 MD5 (inline, no external dep).
- `gravatarUrl(email, options?)` — builds
  `https://www.gravatar.com/avatar/<md5>?d=<defaultImg>&s=<size>&r=<rating>`.
  Returns `null` for empty/invalid emails so callers can `v-if` the
  `<img>` away cleanly.
- Used by `modal.details.Contact.vue` with a `gravatarError` ref that falls
  back to initials when the avatar 404s.

### `generateContactVCardQrCode.ts`

- `generateVCard(contact, wallets)` — vCard 3.0 string with
  `FN`/`N`/`ORG`/`EMAIL`/`TEL`/`URL`/`X-SOCIALPROFILE` and
  `X-CRYPTOCURRENCY-N` per wallet. Phone numbers are split on `,` so the
  vCard emits one `TEL;TYPE=CELL` per phone.
- `generateQRCodeDataUrl(vcard)` — `qrcode` lib wrapper at 400×400 px,
  margin 2.
- `generateContactQRCode(contact, wallets)` — composition helper used by
  the contact QR sub-modal.

### `mirrorText.ts`

- `mirrorText` / `unmirrorText` / `reverseText` — character-order reversal
  used by encrypted notes for at-rest obfuscation.
- `prepareTextForStorage(text, isMirrored)` — reverse before save unless
  already mirrored.
- `getInitialMirrorState(content)` — `true` for non-empty content (assumed
  pre-mirrored), `false` otherwise.

### Adding a new display helper

Put it in this folder if it transforms data **for the screen** (no I/O,
no storage). Use camelCase filenames matching the exported entry function.

## 6. `src/lib/cores/exportStandard/`

Pattern: each entity has **two** modules — one for QR (PNG/SVG) and one
for text (CSV/VCF/JSON/MD).

```
exportStandard/
├── filenameStructureAndContent.general.qrCode.individual.ts   # shared QR filename builder
├── filenameStructureAndContent.AddressBook.Individual.qrCode.ts # legacy per-contact QR
├── filenameStructureAndContent.Contact.notes.ts                # contact notes export
├── currencies/                                                  # currency / seed exports
│   ├── filenameStructureAndContent.individual.{text,visual}.ts
│   ├── filenameStructureAndContent.seed.{text,visual}.ts
│   ├── filenameStructureAndContent.walletAddresses.{text,visual}.ts
│   └── index.ts
└── addressBook/
    ├── filenameStructureAndContent.addressBook.Company.{qrCode,text}.ts
    ├── filenameStructureAndContent.addressBook.Exchange.{qrCode,text}.ts
    └── filenameStructureAndContent.addressBook.Wallet.{qrCode,text}.ts
```

### Filename pattern (text exports)

```
%projectName%.%YYYYMMDD%.%HHMMSS%.addressbook.<entity>.%target%.%ext%
```

Example: `sparkplate-fresh.20260428.222733.addressbook.wallet.MetaMask.json`

The previous `%entry%` segment (e.g. `.record`, `.currencies`, `.vcard`)
was removed for **wallet** exports — the file extension already conveys
the payload kind. **Exchange** and **Company** text exporters still emit
the `entry` segment; align them with Wallet on the next pass for full
consistency.

### Filename pattern (QR exports)

QR exports go through the shared
`generateQRCodeSvgFilename` / `generateQRCodePngFilename` helpers in
`filenameStructureAndContent.general.qrCode.individual.ts`. The pattern
emitted by the helper is:

```
%projectName%.%YYYYMMDD%.%HHMMSS%.%section%.%subSection%/%target%.%entry%.<ext>
```

The `/` would create a Downloads subdirectory, but Firefox flattens `/`
in `a.download` to `_`, so each address-book entity QR exporter calls
`flattenStructuredFilename()` to replace it with a `.` for cross-browser
parity.

### Wallet name target (`walletFilenameTarget`)

`StandaloneWalletRecord.name` is conventionally a composite display label
of the form `<brand> · <mnemonic-hint> · <id>` (matched-option head plus
metadata; see `matchedWalletOption` in `modal.details.Wallet.vue:515`).
The filename helper splits on ` · ` and keeps only the head, then sanitizes
to `[A-Za-z0-9_]+`. This is implemented identically in both
`Wallet.qrCode.ts` and `Wallet.text.ts`.

### vCard generation

Each entity's `qrCode.ts` exports a `<entity>ToVCard()` that:

- Uses `BEGIN:VCARD` / `VERSION:3.0` / `FN` / `ORG`.
- Maps social handles to `X-SOCIALPROFILE;TYPE=<network>`.
- Maps wallets/currencies to `X-CRYPTOCURRENCY-N;TYPE=<ticker>` with the
  payload `<ticker>://<address>` (a deterministic URI scheme that some
  wallet apps recognize).
- RFC-6350-escapes values via `escapeVCardValue` (backslash, `;`, `,`,
  newline; strips CR).

The text exporters re-import the vCard helper from the QR module rather
than duplicating it.

### Adding a new entity export

1. Create `addressBook/filenameStructureAndContent.addressBook.<Entity>.qrCode.ts`
   with `<entity>ToVCard()`, `exportXxxQrPng()`, `exportXxxQrSvg()`.
2. Create the matching `.text.ts` re-using `<entity>ToVCard()` for the VCF
   path; add `exportXxxCsv/Vcf/Json/Md`.
3. Use the same filename builder shape: build segments, join with `.`,
   no `/`. Drop the `entry` segment for new code (matches Wallet).
4. Wire the per-row dropdown in the matching `tab.addressBook.<Entity>.vue`
   to the new exporters via local handlers.

## 7. `src/lib/cores/importStandard/`

```
importStandard/
├── importWallet.json.ts          # JSON wallet-address import
├── fileImports.seed.ts           # seed/mnemonic import (JSON + CSV)
└── contacts/
    └── fileImports.addressBook.contacts.csv.ts   # contacts CSV import
```

### `contacts/fileImports.addressBook.contacts.csv.ts`

The robust Google-Contacts-compatible CSV parser. Key features:

- **Header normalization** via Papa's `transformHeader` — strips UTF-8 BOM
  and trims, so `"\uFEFFEmail 1 - Value"` matches `"Email 1 - Value"`.
- **Single-value lookup** against `HEADER_CANDIDATES` arrays in priority
  order (e.g. firstname tries `'First Name'`, `'FirstName'`,
  `'firstname'`, `'Given Name'`, `'givenname'`).
- **Multi-value collection** via regex patterns (e.g.
  `/^E-?mail \d+ - Value$/i`). All matching columns are joined with
  `', '` so a contact with three emails ends up with a single comma-list.
- **Greedy empty-line skipping** (`skipEmptyLines: 'greedy'`) drops rows
  that are pure commas/whitespace.
- **Identity gate** — rows with no firstname, no lastname, **and** no
  email are dropped (prevents Google's phone-only phantom rows from
  becoming empty contact entries).

`ImportedContact` is the canonical shape consumed by
`AddressBook.vue#addContacts`. To extend (e.g. adding a `birthday`
field): add the optional key, register header candidates, surface the
field in `addContacts`, and persist it in `service.addressBook.Contact`.

### `useContactParser.ts` (composables)

Delegates `.csv` to `parseContactsCsvFile` and inlines the simpler
parsers for `.vcf` and `.xlsx/.xls/.ods` (`xlsx` lib). Keep this file
*thin* — anything more complex should move to `importStandard/`.

### `importWallet.json.ts`

Accepts four shapes:

1. `{ "addresses": [ { "currency", "address", … } ] }`
2. `{ "wallets": [ { … same row shape … } ] }`
3. Root array `[ { … } ]`
4. Single object `{ "currency", "address" }` → wrapped to a one-element list

`coinFromEntry` and `addressFromEntry` accept many synonyms
(`coinTicker`, `coin`, `ticker`, `network`, `symbol`, `chain` for the
former; `walletAddress`, `wallet_address`, `terraAddress`,
`publicAddress` etc. for the latter).

**Safety rejections** (throws with a helpful message):

- `looksLikeSeedOrKeyForgeBackup` — has `mnemonicSeedPhrase` /
  `seedPhrase` or `from === 'keyForge'`. Refused.
- `looksLikePrivateKeyExport` — has `privateKey` / `secretKey` /
  `hexPrivateKey` and **no** public address. Refused.

Both prevent secret material from being silently added to the address
book.

### `fileImports.seed.ts`

Parses mnemonic seed phrases from JSON or CSV. Validates word count
(12–33) and 500-char ceiling. Accepts both `mnemonicSeedPhrase` and
`seedPhrase` keys.

### Adding a new importer

Place it under `importStandard/<area>/` with a `parseXxxFile(file)` that
returns a typed promise. Wire it into the matching button component and,
for contacts/companies, route the result through `addContacts` so the
spread-optional-fields pattern (§2) is honored.

## 8. Cross-cutting conventions

### Storage

Every service in `src/services/addressBook/` uses `localStorage` with a
versioned key (e.g. `sparkplate.addressbook.standaloneWallets.v1`). When
the schema changes incompatibly, **bump the version** in the key and
either migrate or accept the implicit reset.

### Selection ID lists

Every tab maintains its selection as a plain `number[]` of record IDs
exposed via `v-model:selectedXxxIds`. The footer's bulk-delete count
reads from the **active** tab's list via the
`footerBulkDeleteCount` computed in `AddressBook.vue`.

### Error surfacing in exports

All `exportXxx*` helpers wrap their body in `try/catch`,
`console.error` the failure, and `alert(...)` a human-readable message.
This is intentional — the export is a user-initiated terminal action; we
prefer the loud failure to a silent one. Keep this pattern for new
exporters.

### Companies are derived

There is **no** Company storage. `getCompanies()` aggregates over
contacts, grouping by `contact.company`. This is why renaming a contact's
company through `modal.details.Contact` will rename or move the company
in the Companies tab on the next refresh. The
`tab.addressBook.Company.vue` calls `companiesTabRef.value?.loadCompanies?.()`
from `AddressBook.vue#loadContacts` to keep the two views in sync.

### Confirmation dialog colors

`modal.confirm.delete.general.vue` sets explicit
`color: #111827` on its `.modal-content` and `.btn-secondary` to override
the global near-white default from `src/style.css`. **Don't rely on
inheritance** for text color in modals built on the global stylesheet —
state it explicitly.

## 10. Open follow-ups

These are known gaps for the next maintenance pass — not bugs.

- **Filename `entry` segment:** Wallet text exports drop it
  (`…wallet.MetaMask.json`); Exchange and Company text exports still
  emit it (`…exchange.X.record.json`, `…company.X.record.json`). Decide
  on a project-wide rule and align.
- **QR `vcard` segment:** still appears in QR filenames
  (`…wallet.MetaMask.vcard.png`). Reasonable to keep (the extension
  doesn't disambiguate vCard vs raw URI), but should be a conscious
  decision documented in the helper.
- **Company-level "add currency":** `tab.addressBook.Company.vue#openAddCurrencyForCompany`
  is a `console.log` placeholder. Companies are derived, so any company
  currency would have to attach to one of the member contacts. Pick a
  convention (primary contact? prompt-to-pick?) before wiring.
- **`subModal.*` vs `submodal.*`:** mixed casing in the subModals folder.
  Migrate to camelCase on the next rename.
- **Contact table `Company` column:** currently commented out at
  `tab.addressBook.Contact.vue:44–53` and `:99`. Re-enable when the
  column is wanted again (also bump empty-state `colspan` 7 → 8).
- **`aspect.socialMedia.vue` Bluesky → social tab:** Bluesky and
  Telegram are in the read-only aspect under "social", but the editable
  submodal puts them under the **Social** tab — verify visual order
  parity if you reshuffle either.
