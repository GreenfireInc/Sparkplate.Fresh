## 1. Reframing: V2 is **not** a 1:1 Greenery route port

The methodology's Phase 2/3 store/view lists were derived from **Greenery's** modules and routes. V2's actual
surface differs, so we must port each store **only when a V2 feature needs it**, not mechanically per the
Greenery list.

- **Greenery route views** (About, Calculator, Dashboard, Exchange, History, Home, PaperWallet, QuickExchange,
  Settings, Storage, Transactions, Web3Connect) vs. **V2 route views** (`src/views/`): Home, Dashboard, AddressBook,
  Cryptocurrency, Cryptography, KeyFiles, Networking, Build, Package, Publish, Repurposing, Games, Sandbox, QRCode,
  Techstack, Settings. Several Greenery views are **absent** (History, Exchange, PaperWallet, QuickExchange,
  Transactions, Web3Connect, About-as-route) or exist only as **tabs/components** (e.g. Calculator lives in the
  Cryptocurrency view).
- **No V2 services or UI exist yet** for the Phase 2 domains: invoices, activities, quickExchange,
  mnemonicPasswords, exchanges. Only `services/addressBook/*` and `services/account/*` exist.

**Implication:** do not create speculative stores for modules that have no V2 feature. Each Phase 2/4 store is
gated on its feature + a backing service landing first.

## 3. What to do next — prioritized

### 3.1 Near-term, no Electron/IPC (available right now)

**A. Wire `useCoinsStore` into the views that already fetch prices (Phase 3 slice).**
Two components fetch market data independently today and would each become a thin consumer of the one store
(single cache + persisted warm-start + dedup):
- `src/components/partials/marqueeTicker/MarqueeTicker.vue` — currently its own CoinGecko fetch.
- `src/components/pageTabs/cryptocurrency/tab.cryptocurrency.Indices.vue` — currently calls `fetchCurrencyPrices` directly.
- (Stretch) the Calculator tab (`tab.cryptocurrency.Calculator.vue` / `aspect.Calculator/*`) for rate/price reads.
Outcome: exercises the Phase 1 `useCoinsStore` for the first time; removes duplicate fetching; instant warm
start from `sparkplate_coins`.

**B. Fix the stray Vuex logout in `KeyboardShortcuts.vue` (§2).**
Swap `this.$store.dispatch('accounts/logout')` → `useAccountsStore().logout()`. One-line behavioral fix now
that the store exists; removes a broken call.

**C. Logout/reset hardening (partial Phase 5).**
Add a single `resetAllStores()` (or per-store `$reset()`) invoked from the logout flow
(`dropdown.authentication.vue` → `alerts.logout.vue`) so settings/coins/wallets/contacts session state is
cleared consistently. Decide whether wallet/coins caches survive logout (per-device cache) or clear.

**D. Persistence + cross-window audit (partial Phase 5).**
Verify `sparkplate_settings`, `sparkplate_coins`, `sparkplate_wallets` round-trip and sync across windows via
the persistedstate `storage` event (the mechanism that replaced `vuex-shared-mutations`). Produce the
persisted-keys table and confirm no double-persistence with service-owned data (contacts, accounts).

### 3.1.x As-built — §3.1-C/D (implemented 20260611)

**C. Logout reset — `src/stores/resetStores.ts`.**
Added `resetStoresOnLogout(options?)`, the single place the logout flow touches. Wired into both logout
entry points: `dropdown.authentication.vue` `confirmLogout()` (replaced the bare `useAuth().logout()`) and
`KeyboardShortcuts.vue` `Ctrl+Shift+L`. Data policy (resolves the §5 open decision):
- **Always cleared (session-scoped):** `useAccountsStore.reset()` (active user + `authenticated` + `ip`;
  superset of `logout()`) and `useContactsStore.reset()` (in-memory list only — on-disk service data is
  untouched and reloads on next mount).
- **Retained by default (device-level), opt-in via options:** `useSettingsStore` (app-wide UI prefs),
  `useCoinsStore` (market cache, not user-specific), and `useWalletsStore` — the latter **deliberately**
  retained because its public addresses are derived from **discarded** mnemonics, so clearing is
  irreversible. `LogoutResetOptions.{clearWallets,clearCoins,clearSettings}` flip these on for
  shared-device privacy; proper per-user partitioning remains a Phase 4 concern.

**D. Persistence + cross-window audit (findings).**

| Key | Owner | Persists via | Cross-window live sync |
|-----|-------|--------------|------------------------|
| `sparkplate_settings` | `useSettingsStore` | persistedstate (`pick` menu/visibility/network) | **Yes** — explicit `window 'storage'` listener in `ensureInitialized()` |
| `sparkplate_coins` | `useCoinsStore` | persistedstate (`pick` prices/lastFetched) | No live listener — rehydrates on window open only |
| `sparkplate_wallets` | `useWalletsStore` | persistedstate (`pick` byTicker) | No live listener — rehydrates on window open only |
| `sparkplate.accounts.users.v1` | `service.account.User` | service `localStorage` | n/a (store holds no persist; reloads via `loadUsers()`) |
| `addressBook` service keys | `services/addressBook/*` | service `localStorage` | n/a (service-owned; finding 06032026) |

- **No double-persistence:** accounts and contacts stores enable no `persist` (service-owned), confirmed
  against `stores/index.ts` policy and the address-book finding.
- **Cross-window gap (intentional, low priority):** only `settings` live-syncs across already-open windows;
  `coins`/`wallets` pick up changes on next open. Acceptable today — coins refresh on a timer and wallets
  rarely change with the Dashboard typically single-window. If needed later, replicate the settings
  `storage`-listener pattern (the methodology's "built-in storage event" claim is only true with such a
  listener — the plugin does not auto-sync open windows by default).

### 3.2 Phase 2 — domain stores, **only as features land**

Prerequisite chain for each store: **(1) a V2 feature/UI surface exists → (2) a `localStorage`/SQLite
service is added (mirror `services/addressBook/*` + `services/account/*`) → (3) a `use*Store` wraps the
service (service-owned persistence; no `persist` block).**

**Surface audit (20260611).** Searched `src/` for each methodology-named domain:

| Domain (methodology) | V2 UI surface? | V2 service? | Status |
|----------------------|----------------|-------------|--------|
| invoices | none | none | **Blocked** — feature not in V2 |
| activities | none | none | **Blocked** — feature not in V2 |
| quickExchange | none (backend lib only: `src/lib/cores/currencyCore/quickExchanges/*`, ~20 providers, zero UI consumers) | none (no order-state service) | **Blocked** — build the swap UI + order service first |
| mnemonicPasswords | `tab.cryptocurrency.MnemonicSeedPhrase.vue` is a **stateless generator** (no persistence) | none | **Blocked** — not a saved-passphrase manager |
| exchanges (markets) | none (no markets-exchange view) | none | **Blocked** — feature not in V2 |

**Unblocked opportunity — the rest of the address book.** The Greenery `contactModule` (mapped to
`useContactsStore`) actually covered the whole address book. In V2 only **Contact** has a store; the other
address-book entities have **live tabs that call their `localStorage`-backed services directly**, with no
Pinia layer — exactly the pre-`useContactsStore` situation, same finding (06032026), same pattern:

| Entity | Tab | Service | Persistence |
|--------|-----|---------|-------------|
| Standalone wallets | `tab.addressBook.Wallet.vue` | `service.addressBook.StandaloneWallet` (+ per-contact `…Wallet`) | own `localStorage` |
| Exchange accounts | `tab.addressBook.Exchange.vue` | `service.addressBook.Exchange` (`updateExchange`/`deleteExchange`/`ExchangeRecord`) | own `localStorage` |
| Notes | (note UI) | `service.addressBook.Note` | own `localStorage` |
| Companies | `tab.addressBook.Company.vue` | `service.addressBook.Company` (`getCompanies`/`deleteCompany`) | **derived** from contacts (no own store) → a getter, not a persisted slice |

**Recommendation:** the next genuinely-unblocked Phase 2 store is an **address-book companion store** —
either per-entity (`useExchangeContactsStore`, `useStandaloneWalletsStore`, `useNotesStore`, with companies
as a derived getter) or a single `useAddressBookStore` — thin wrappers over the existing services, mirroring
`useContactsStore` (no `persist`; service owns the data). It has a real V2 surface today and needs no IPC.
The five methodology-named domains remain blocked until their features are built.

**As-built — `src/stores/useAddressBookStore.ts` (implemented 20260611).** Built the consolidated variant.
Thin wrapper over the existing services, no `persist` (service-owned, per finding 06032026):
- **Exchanges** (`exchanges` ref): `loadExchanges` / `saveExchange` (insert when `id` omitted, else update) /
  `removeExchange`, over `service.addressBook.Exchange`.
- **Standalone wallets** (`wallets` ref): `loadWallets` / `saveWallet` / `removeWallet`, over
  `service.addressBook.StandaloneWallet`.
- **Companies** (`companies` ref, **derived** from contacts): `loadCompanies` / `removeCompany` (the latter
  deletes the underlying company contacts via the service; callers showing contacts should also refresh
  `useContactsStore`).
- **Notes** (owner-scoped, not a flat slice): thin passthroughs `notesFor` / `addNote` / `updateNote` /
  `deleteNote` plus the re-exported reactive `notesRevision` signal.
- `loadAll()` hydrates exchanges + wallets + companies; `reset()` clears the in-memory lists only and is
  now called from `resetStoresOnLogout()` (session-scoped group, alongside contacts).
- Verified: `vue-tsc` 0 errors on the new store + `resetStores.ts`; no lint errors.

**Wiring (Phase 3, implemented 20260611).** The address-book UI now reads/writes through the store:
- `AddressBook.vue`: `exchanges` / `wallets` are now `storeToRefs(useAddressBookStore())` (local refs
  removed); `loadExchanges()` / `loadStandaloneWallets()` delegate to `loadExchanges()` / `loadWallets()`.
  Bulk import/delete loops still call the services then refresh via the store (same hybrid as the contacts
  wiring). Dropped the now-unused `getExchanges` / `getStandaloneWallets` imports.
- `tab.addressBook.Exchange.vue`: `onExchangeDetailSaved` → `saveExchange`, `onConfirmDelete` →
  `removeExchange` (dropped direct `updateExchange` / `deleteExchange`).
- `tab.addressBook.Wallet.vue`: currency add / bulk-import / remove → `saveWallet` (reads the target from
  `addressBookStore.wallets`); delete → `removeWallet` (dropped `getStandaloneWallets` /
  `updateStandaloneWallet` / `deleteStandaloneWallet`).
- `tab.addressBook.Company.vue`: `companies` is now `storeToRefs(...).companies`; `loadCompanies` →
  `addressBookStore.loadCompanies`, delete → `removeCompany` (dropped direct `getCompanies` /
  `deleteCompany`). `defineExpose({ loadCompanies })` retained so `AddressBook.vue`'s post-contact-change
  refresh still works.
- Verified: `vue-tsc` 0 errors across `AddressBook.vue` + the three tabs; no lint errors.
