## 1. Reframing: V2 is **not** a 1:1 Greenery route port

The methodology's Phase 2/3 store/view lists were derived from **Greenery's** modules and routes. V2's actual
surface differs, so we must port each store **only when a V2 feature needs it**, not mechanically per the
Greenery list.

- **Greenery route views** (About, Calculator, Dashboard, Exchange, History, Home, PaperWallet, QuickExchange,
  Settings, Storage, Transactions, Web3Connect) vs. **V2 route views** (`src/views/`): Home, Dashboard, AddressBook,
  Cryptocurrency, Cryptography, KeyFiles, Networking, Build, Package, Publish, Repurposing, Games, Sandbox, QRCode,
  Techstack, Settings. Several Greenery views are **absent** (History, Exchange, PaperWallet, QuickExchange,
  Transactions, Web3Connect, About-as-route) or exist only as **tabs/components** (e.g. Calculator lives in the
  Cryptocurrency view).
- **No V2 services or UI exist yet** for the Phase 2 domains: invoices, activities, quickExchange,
  mnemonicPasswords, exchanges. Only `services/addressBook/*` and `services/account/*` exist.

**Implication:** do not create speculative stores for modules that have no V2 feature. Each Phase 2/4 store is
gated on its feature + a backing service landing first.

| Key | Owner | Persists via | Cross-window live sync |
