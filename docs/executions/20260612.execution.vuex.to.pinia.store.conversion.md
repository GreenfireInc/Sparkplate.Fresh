## 4. IPC — Sparkplate today vs Greenery `background/`

### 4.1 The gap

Greenery's renderer reached the OS/crypto layer through a **wide `contextBridge` surface**
(`00.references/from.Greenery/background/preload.js`) backed by per-domain `ipcMain.handle` files
(`background/ipcMain/*`). Sparkplate exposes only a **narrow** surface today.

| `window.*` namespace | Greenery (V1) | Sparkplate (V2) | Phase 4 plan |
|----------------------|---------------|------------------|--------------|
| `ipcRenderer` | ✅ `invoke/send/on/once` | ✅ `invoke/send/on/off` | keep |
| `app` | ✅ `fetchIP`, `getPath`, `getReleaseInfo`, `openExternal`, `relaunch`, … | 🟡 `getPreloadData`, `getGPUInfo`, `getUsbDrives`, `getNetworkAdapters` | extend (`fetchIP` if kept) |
| `cryptos` | ✅ full wallet/tx/fee/send surface | ❌ **missing** | **P4.6–P4.9** (largest) |
| `fs` | ✅ read/write/exists/createZip/streams | ❌ **missing** | **P4.1** |
| `storage` | ✅ `encryptBuffer`/`decryptFile`/`encryptFile` | ❌ **missing** | **P4.4** |
| `notification` | ✅ `newNotification` | ❌ **missing** | **P4.2** |
| `geoip` | ✅ `lookup` | ❌ **missing** | optional (signup geo) |
| `authenticator` | ✅ OTP `check`/`generateSecret`/`keyuri` | ❌ **missing** | MFA (`TwoFactorAuth`) — later |
| `steganography` | ✅ `createUserSteg`/`createStegImage` | ❌ **missing** | recovery-image custody — optional |
| `dialog` / `archiver` / `minizip` / `json2csv` / `domains` / `exchanges` / `mailchimp` / `wert` / `qrscanner` / `greeneryId` | ✅ various | ❌ mostly missing | per-feature, as those features land |

> Note: V2 already has `background/functions/utils/blockchainApi/*` (infura/alchemy/blockchair/…) and
> `background/functions/utils/domains/*` (ens/uns/tezos) in main — the **data sources** for `cryptos`/`domains`
> exist; only the **IPC handlers + preload bridges + renderer types** are missing.

### 4.2 Greenery's pattern (what to mirror)

```
background/ipcMain/index.js        → imports every per-domain handler module
background/ipcMain/cryptos.js      → ipcMain.handle('cryptosGenerateWallet'|'cryptosGetBalances'|'cryptosSendToAddress'|…)
background/ipcMain/fs.js           → ipcMain.handle('fsReadFile'|'fsWriteFile'|'fsCheckFileExists'|…)
background/ipcMain/storage.js      → ipcMain.handle('storageEncrypt'|'storageDecrypt'|…)
background/utils/cryptos/index.js  → per-coin libs (btc/eth/sol/…) the handlers delegate to
background/preload.js              → contextBridge.exposeInMainWorld('cryptos'|'fs'|'storage'|'notification'|…)
```

**Sparkplate target** (per `06112026…ipc.crypto.bridge.md`, adapted to V2's modular `background/`):

```
background/main/ipc/<namespace>.ts   → registerCryptosIpc() / registerFsIpc() / … (channels 'ns:verb')
background/main/index.ts             → calls register*Ipc() once on startup
background/preload/index.ts          → contextBridge.exposeInMainWorld('<ns>', { verb: (a)=>ipcRenderer.invoke('ns:verb', a) })
src/types/background.d.ts            → declare global { interface Window { <ns>: { … } } }
```

**Conventions to standardize (differs from Greenery's flat channel names):**
- Namespaced channels `'<ns>:<verb>'` (Greenery used flat `cryptosGenerateWallet`; V2 uses `cryptos:generateWallet`).
- **Serializable boundary only** — no `Buffer`/streams across IPC (Greenery's `fsCreateWriteStream` returning a
  stream handle is an anti-pattern; return base64/hex strings instead). This also avoids the
  `Uint8Array`↔`Buffer` class of type errors.
- **Uniform `{ ok, data | error }` envelope** unwrapped in preload (Greenery threw raw or returned `{err,data}`
  inconsistently).
- `contextIsolation` stays **on**, `nodeIntegration` **off** (already correct in `background/main/index.ts`).

## 5. State vs Persistence vs Database — and IndexedDB → SQLite3

This is the conceptual core of the request: **Pinia state is not a database.** There are three distinct
layers, and conflating them is the main source of confusion.

### 5.1 The three layers

| Layer | What it is | Lifetime | In Sparkplate |
|-------|------------|----------|---------------|
| **1. State (Pinia)** | In-memory reactive refs/computeds that the UI renders | **Cleared on reload** unless rehydrated | `useXxxStore()` |
| **2. Persistence (key-value)** | A flat blob serialized to disk so state survives restart | Survives restart; **not queryable/relational** | `pinia-plugin-persistedstate` (`sparkplate_*`) **or** a `localStorage` service |
| **3. Database (records)** | Durable, **queryable, relational, transactional** record store | Survives restart; supports joins, indexes, migrations | **today:** `localStorage` JSON services; **target:** **SQLite3** |

Pinia (layer 1) is **always** just the reactive cache the components read. It gets its data *from* layer 2 or
3 and writes *back* to them. The migration's recurring decision — "does a service already own this data?" —
is exactly the choice between **layer 2 in the store** (`persist`) vs **layer 2/3 behind a service** (no
`persist`).

### 5.2 What each layer holds today

| Data | Layer 1 (store) | Layer 2/3 (durable) |
|------|-----------------|---------------------|
| Settings, coins cache, public wallets | the three `persist` stores | **layer 2** key-value (`sparkplate_settings/coins/wallets`) |
| Contacts, exchanges, standalone wallets, notes | `useContactsStore` / `useAddressBookStore` | **layer 2** `localStorage` services (`services/addressBook/*`) |
| Accounts (profile + password hash) | `useAccountsStore` (session only) | **layer 2** `localStorage` service (`sparkplate.accounts.users.v1`) |
| Wallet **secrets**, transactions, invoices, activities | — (not built) | **layer 3 target: SQLite** |

### 5.3 How Greenery did it (IndexedDB)

Greenery's durable layer (layer 3) was **IndexedDB** (browser-embedded object DB), wrapped by `*Service`
classes (`UserService`, `WalletService`, `TransactionService`, …) and protected for secrets via SecureLS
(`vuex-persistedstate`) + `window.storage.encryptBuffer`. On login it **rehydrated Vuex** from IndexedDB
(`fetchDBWallets(userId)`, etc.). So Greenery's flow was:

```
IndexedDB (records)  →  *Service  →  Vuex (state)  →  components
                         ↑ encrypt/decrypt secrets via window.storage
```

IndexedDB is a renderer-side, schemaless, object store — convenient but **not relational**, awkward for
multi-table queries (balances-by-account, tx-by-wallet), and its encryption was bolted on.

### 5.4 Why Sparkplate moves to SQLite3 (TypeScript) — and where it lives

Per `10192025…sqlite3…` and the login-auth finding, V2 replaces IndexedDB with **SQLite3 in the Electron
main process**, exposed through IPC (a `window.database` bridge), with `*Service` modules in main running the
SQL:

```
SQLite (main, relational)  →  db service (main)  →  window.database (IPC)  →  Pinia store  →  components
                              ↑ field encryption for secrets (window.storage / cryptographyCore)
```

| Dimension | Greenery IndexedDB | Sparkplate SQLite3 (TS) |
|-----------|--------------------|--------------------------|
| Location | renderer (browser) | **Electron main** (Node), via IPC |
| Model | schemaless object store | **relational** tables + foreign keys + indexes |
| Queries | key/cursor scans | **SQL** (joins, aggregates: balances, tx history) |
| Migrations | ad-hoc | versioned schema migrations |
| Secrets | SecureLS + `window.storage` | **column-level encryption** (decided in P4.3/§5.5) |
| Access from renderer | direct (same context) | **`window.database` IPC bridge** (`db:user:create`, `db:wallet:list`, …) |
| Pinia's role | rehydrate Vuex on login | rehydrate Pinia store on login — **identical mental model** |

**Crucial point:** SQLite does **not** replace Pinia. Pinia stays the reactive layer the components read;
SQLite replaces the *durable record* layer (IndexedDB) that the stores hydrate from and persist to. A store
backed by SQLite simply enables **no `persist`** (the DB owns the data) and calls `window.database.*` in its
actions — the same service-owned pattern `useContactsStore` already uses for `localStorage`, just with a
relational backend reached over IPC.

### 5.5 Migration boundary (what goes to SQLite, and when)

Not everything should move. Recommended split:

| Keep in layer 2 (key-value) | Move to layer 3 (SQLite) |
|------------------------------|--------------------------|
| UI prefs (`settings`), coin cache (`coins`), **public** wallet addresses | Accounts (profile + password hash) |
| Active-ticker / ephemeral UI flags | **Wallet secrets** (encrypted `privateKey`/`wif`) |
| | Transactions, invoices, activities (relational, queryable) |
| | Address-book entities once relational queries are needed (optional) |

**Open decisions (resolve in a short decision record before building the DB):**
1. **Driver** — `better-sqlite3` (sync, fast, native) vs `node:sqlite`/`sql.js`. *Recommend `better-sqlite3`
   in main.*
2. **Encryption** — column-level (encrypt secret fields) vs whole-file (SQLCipher). *Recommend column-level
   via the `window.storage` bridge (P4.4), keeping the DB file plain for portability.*
3. **First table to land** — accounts (lowest risk, already has a localStorage service to migrate) before
   wallet secrets / transactions.
4. **Coexistence** — localStorage services stay for UI/public data; SQLite is introduced **only** for the
   durable/relational/secret set, migrating one entity at a time (mirror the `useContactsStore` rollout).
