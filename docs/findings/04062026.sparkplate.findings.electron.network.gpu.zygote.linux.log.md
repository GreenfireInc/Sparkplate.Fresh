# Electron / Chromium on Linux — network service, GPU process, and zygote log lines

**Date:** April 6, 2026  
**Source:** Console / stderr lines observed while running an Electron-based app (e.g. Sparkplate.Fresh) on Linux.

## Raw messages (example)

```
[38646:0406/191301.472046:ERROR:content/browser/network_service_instance_impl.cc:597] Network service crashed, restarting service.
[38646:0406/191301.524584:ERROR:content/browser/gpu/gpu_process_host.cc:957] GPU process exited unexpectedly: exit_code=15
[38646:0406/191301.550894:ERROR:content/common/zygote/zygote_communication_linux.cc:296] Failed to send GetTerminationStatus message to zygote
```

## What each line means (plain language)

### 1. `Network service crashed, restarting service`

- **What it is:** Chromium (inside Electron) splits work across **multiple processes**. One of them is the **network service** (handles HTTP, WebSockets, cookies, etc. in modern Chromium).
- **What happened:** That subprocess **died or was terminated**. The browser process detected the failure and **tried to start a replacement** network service.
- **Does it always mean a bug in your app?** No. It can follow a **GPU crash**, **OOM** (out of memory), **manual kill** of a child process, or **shutting down** the app while something was still cleaning up.

### 2. `GPU process exited unexpectedly: exit_code=15`

- **What it is:** Another subprocess, the **GPU process**, handles **OpenGL/Vulkan/ANGLE** work and talks to your graphics drivers.
- **exit_code=15:** On Linux, signal number **15** is **SIGTERM** — “please terminate.” So the process **exited after SIGTERM**, not necessarily a random “segfault” (which would often be **11** = SIGSEGV, or **6** = SIGABRT).
- **Implications:** Something **asked the GPU process to stop** (normal during app exit, session logout, or parent teardown), or the system **killed** it as part of a cascade. It is still logged as “unexpected” from Chromium’s point of view if the parent did not shut down in the usual orderly sequence.

### 3. `Failed to send GetTerminationStatus message to zygote`

- **What it is:** On Linux, Chromium historically uses a **zygote** model (like Chrome): a parent process forks renderers. The browser asks the zygote for **termination status** of children.
- **What happened:** IPC to the zygote **failed** — often because the **zygote or a related process was already dead**, **stopping**, or the **socket/pipe was broken**.
- **Implications:** Usually a **follow-on error** after other processes (e.g. GPU or network service) have already crashed or been signaled, not a root cause by itself.

## How these three relate

They typically appear **in quick succession** (milliseconds apart in your example). That pattern usually means:

1. One critical subprocess **stopped** (often **GPU** first if graphics tear down or crash).
2. Other services (**network**) then fail or get restarted.
3. Internal **IPC** (zygote) breaks because the process tree is **already tearing down** or **unstable**.

So they often indicate **one underlying event** (shutdown, crash, kill, OOM, driver reset) rather than three independent bugs.

## Common benign causes

- **Closing the Electron app** or **stopping** `electron` / `vite` / debugger while the window is still doing work.
- **Hot reload** or **devtools** triggering a reload that tears down processes quickly.
- **Session lock / sleep / resume** confusing the GPU stack on some drivers.
- **Wayland / XWayland / compositor** quirks (see `04062026.sparkplate.findings.wayland.linux.electron.md`).

## When to investigate further

- **Reproducible** while performing a specific action (open tab, video, WebGL, etc.).
- **Blank window**, **permanent** network failure, or **instant** exit every launch.
- **exit_code** other than **15** during normal use (e.g. **139** often hints at SIGSEGV).

**Practical checks:** GPU drivers, `dmesg` for OOM or GPU reset, try `--disable-gpu` temporarily, compare X11 vs Wayland session, and note whether logs only appear on **exit** (often harmless).
