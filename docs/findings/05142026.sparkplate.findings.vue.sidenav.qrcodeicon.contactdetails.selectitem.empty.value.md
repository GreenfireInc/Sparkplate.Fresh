# Vue: unresolved `QrCodeIcon` in SideNav and Radix `SelectItem` empty `value` in contact details

**Date:** May 14, 2026 (from `date -Iseconds` / local shell)  
**Category:** Vue 3 / component resolution / Radix Vue (Select)  
**Status:** Runtime warnings and hard error when opening contact details from Address Book

## Overview

The console output mixes **normal startup logs**, a **Vue resolve warning** for an icon component, and a **Radix Vue guard** that throws when a `<SelectItem>` is rendered with `value=""`. The stack traces tie the icon issue to `SideNav.vue` (around line 149) and the select issue to opening the contact-details modal (`AddressBook.vue` → `openContactDetailsModal`, `ContactDetailsModal` with a `SelectRoot` whose `modelValue` is empty alongside at least one `SelectItem` also using an empty string).

## Issue 1: `[Vue warn]: Failed to resolve component: QrCodeIcon` — resolved

### Symptom

Vue reported that it could not resolve `QrCodeIcon` while rendering `<SideNav>`. The warning appeared after login and again on navigation (e.g. router `finalizeNavigation`), because `SideNav` re-renders when auth/route state updates.

### Root cause

`SideNav.vue` used `<QrCodeIcon />` in the template but registered **Lucide’s `QrCode`** (not `QrCodeIcon`) in the options `components` map. Vue resolves PascalCase tags to registered components by name, so **`QrCodeIcon` did not match `QrCode`**, which produced the unresolved-component warning.

### Resolution (Bootstrap Icons)

The QR nav item was switched to the **`bootstrap-icons`** dependency (`bi bi-qr-code`), with `import 'bootstrap-icons/font/bootstrap-icons.css'` in `SideNav.vue` and inline `font-size` / `color` bound to the same `pathColor('/qrcode')` logic used for Lucide icons elsewhere in the nav. Lucide’s `QrCode` import/registration was removed from this file.

## Issue 2: `Error: A <SelectItem /> must have a value prop that is not an empty string`

### Symptom

`main.ts`’s `app.config.errorHandler` logs a thrown `Error` from **radix-vue** `SelectItem` setup. Immediately after, Vue may warn that **`SelectItem` is missing template or render function**—that secondary warning is typically a **follow-on effect** of the component failing during setup, not a separate root bug.

### Meaning

Radix Vue reserves **`value=""`** for the select’s own **cleared / placeholder** state. Therefore **every** `<SelectItem>` must have a **non-empty** `value` (or the item must not be rendered as a `SelectItem`). Passing `value=""` violates the library’s contract and throws during `SelectItem` setup.

The trace shows:

- `<SelectRoot modelValue="" onUpdate:modelValue=…>` — the root value can be empty for “nothing selected”.
- `<SelectItem value="" …>` — **this** is invalid for an actual list item.

Typical causes:

- A “placeholder” or “none” row implemented as `<SelectItem value="">` instead of using the root’s placeholder and omitting that item.
- Data-driven options where one option’s key is `""` (e.g. empty string category from `ContactDetailsModal` / address book data).

### Likely fix direction

- Ensure every real option uses a **sentinel non-empty value** if you need “unset” in the list (e.g. `"__none__"`) and map it in `@update:modelValue`, **or** remove the empty `SelectItem` and rely on `SelectRoot` placeholder + empty `modelValue`.
- Audit `ContactDetailsModal` (and any shared select) for `SelectItem` bound to fields that can be `''` from props or API.
