# Why Dashboard wallet generation wasn't working: the registration recovery phrase is discarded at signup

**Date:** June 12, 2026 (`20260612`, executions-style `YYYYMMDD` schema)
**Category:** Registration / account seed custody / Dashboard wallet generation
**Status:** Root-cause finding

## Observation

Sparkplate's registration flow **does** generate a genuine BIP-39 recovery phrase: `03.registration.mnemonicHDSeedPhrase.vue` mints a 12–24-word mnemonic, shows it, computes a BIP32 root GPG fingerprint, makes the user **verify** it, and can print it. On verification it emits `confirm(mnemonic)`.

But the Dashboard generates wallet addresses that have **nothing to do with that phrase**. The reason is a single broken link in the chain: **the verified recovery phrase is thrown away at signup and never persisted to the account**. With no account seed anywhere in the app, the Dashboard's wallet store falls back to minting a throwaway mnemonic on every click. That is *why it was not working*.

This was a **deliberate Phase 4 deferral**, not an accident — but it is the precise reason downstream wallet generation can't work correctly yet.

### 3. The account layer has nowhere to put it

`useAccountsStore.signup` takes `CreateUserInput = { firstName, lastName, email, password }` — no mnemonic — and `service.account.User.ts`'s `StoredUser` has no `mnemonic`/seed column. So even if `handleSignup` forwarded the phrase, there is currently no field to persist it in. (Contrast Greenery, where `users.mnemonic` is a real, AES-encrypted column unlocked at login — see `06102026…login.auth…`.)

### 4. The Dashboard falls back to a throwaway seed

With no account seed available, `useWalletsStore.generateAndAddWallet` invents a fresh random mnemonic per click and discards it (documented in the companion finding §2.2). Consequences:

- Generated addresses are **not derived from the user's recovery phrase**, so the printed/verified phrase can never recover them.
- They are **non-deterministic** — repeating "New Wallet" yields unrelated addresses, not the next HD index.
- The secret is **discarded**, so wallets are **unspendable/unrecoverable**.
- **"From Mnemonic" and "Throwaway" are identical**, because neither path has a real account seed to use.

---

## Why this is the root cause (not the renderer derivation)

The derivation core (`@/utils/cryptoGenerator`) and the store primitive (`useWalletsStore.addFromMnemonic(ticker, mnemonic, options)`) are both capable of producing correct addresses **when given a real, stable mnemonic**. The failure is upstream of them: the one real mnemonic the app ever creates — at registration — is dropped before it can be stored, so nothing downstream can reference it. Fix the persistence link and the existing primitives start producing correct, recoverable addresses.

---

## The fix (minimal chain to make it work)

1. **Stop discarding the phrase at signup.** Change `handleSignup(_mnemonic?)` to forward the mnemonic into `accounts.signup`.
2. **Persist it (encrypted).** Add a `mnemonic` field to `CreateUserInput` / `StoredUser` and store it **encrypted** (per `06102026…login.auth…` + `10192025…sqlite3…`) — never in plaintext. The store/UI should only ever see a sanitized profile.
3. **Expose it to the wallet flow.** Add an account accessor (e.g. `getActiveMnemonic()`), as already specified in the companion finding §6.1.
4. **Derive from it in the Dashboard.** "From Mnemonic" → `walletsStore.addFromMnemonic(ticker, accountMnemonic, { index: nextHdIndex })`; keep "Throwaway" as an independent random keypair (companion finding §6.3).

After step 1–2, the recovery phrase the user wrote down at registration actually backs their wallets, making generated addresses deterministic and recoverable — matching Greenery.

## Summary

Sparkplate generates a real recovery phrase at registration and even makes the user verify it — but `01.registration.signUp.vue`'s `handleSignup` **intentionally ignores the confirmed mnemonic**, and neither `useAccountsStore.signup` nor `service.account.User.ts` has a field to store it. With no persisted account seed, the Dashboard's `useWalletsStore` mints a throwaway mnemonic per click and discards it, so generated public addresses are unrelated to the recovery phrase, non-deterministic, and unrecoverable. The fix is to forward the registration mnemonic into signup, persist it encrypted, expose it via the account store, and have the Dashboard derive from it — at which point the already-correct `addFromMnemonic` + `cryptoGenerator` primitives produce proper, recoverable addresses.
