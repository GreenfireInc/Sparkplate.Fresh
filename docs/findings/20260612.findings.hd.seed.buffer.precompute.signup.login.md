## Observation

The parent comparison doc flags a gap at registration/login parity:

| | Greenery | Sparkplate.Fresh (current) |
|---|----------|----------------------------|
| **Seed buffer precomputed** | Yes — signup/login hold `{ phrase, seed }` on the form / in `HDWalletService` | No — only the **phrase string** is kept; seed is re-derived on each wallet call via `mnemonicToSeed` in `@/utils/cryptoGenerator` |

This is marked "acceptable but slower; not a correctness bug." That is true for a **single** derivation, but it becomes a structural problem when:

1. **Every Dashboard click re-runs PBKDF2** (`bip39.mnemonicToSeed` is intentionally slow — 2048 rounds).
2. **Greenery's downstream APIs expect a seed buffer**, not a phrase — `window.cryptos.generateWallet({ seed, … })` and future Sparkplate IPC ports will need the same shape.
3. **There is no single session object** orchestrating phrase + seed + derivation counters — logic is split across `useAccountsStore.mnemonic`, `Dashboard.vue`, and `cryptoGenerator.ts`.

This document specifies exactly what to build so Sparkplate precomputes and holds the BIP-39 seed buffer once per session, matching Greenery's `HDWalletService` contract.

## 1. What Greenery does

### 1.1 Mnemonic → seed at registration mount

Before the user even finishes the form, Greenery generates `{ phrase, seed }` in the **main process**:

```5:8:greenery/background/ipcMain/cryptos.js
ipcMain.handle('generateMnemonic', async (event, mnemonicPhrase = null) => {
  const phrase = mnemonicPhrase || generateMnemonic()
  const seed = await mnemonicToSeed(phrase)
  return { phrase, seed }
})
```

`RegistrationForm.vue` stores this on `form.mnemonic = { phrase, seed }` in `beforeMount`.

### 1.2 HDWalletService holds both for the session

At signup (and again at login), Greenery constructs a service object that **keeps the seed buffer in memory**:

```6:13:greenery/src/service/HDWalletService.js
export class HDWalletService {
  constructor(mnemonic, email, password, name) {
    this.password = password
    this.email = email
    this.name = name
    this.mnemonic = mnemonic.phrase
    this.seed = mnemonic.seed
  }
```

Wallet generation passes `this.seed` directly — **no second `mnemonicToSeed` call**:

```15:24:greenery/src/service/HDWalletService.js
  async generateWallet({ coinTicker, network, displacer = 0 }) {
    const seed = this.seed
    …
    const wallet = await window.cryptos.generateWallet({
      seed,
      coinTicker,
      network,
      derivationIndex
    })
```

### 1.3 Login rebuilds `{ phrase, seed }` once

After decrypting the stored phrase from IndexedDB, login re-derives the seed **once** via the same IPC helper:

```157:160:greenery/src/store/accountsModule.js
      const mnemonic = await window.cryptos.generateMnemonic(user.mnemonic)
      const hdWallet = new HDWalletService(mnemonic, user.email)
      commit('setHDWallet', hdWallet)
```

(`generateMnemonic(existingPhrase)` validates the phrase and returns `{ phrase, seed }` without generating a new one.)

## 2. What Sparkplate does today

| Layer | Current behavior | Problem |
|-------|------------------|---------|
| **`useAccountsStore`** | Session ref `mnemonic: string \| null` — phrase only | No seed buffer |
| **`signup` / `authenticate`** | Sets `mnemonic.value` from input or `decryptMnemonic` | Never calls `mnemonicToSeed` |
| **`useWalletsStore.addFromMnemonic`** | Passes phrase string to `cryptoGenerator` | Triggers PBKDF2 on every call |
| **`cryptoGenerator.ts`** | Line 30: `const seed = await bip39.mnemonicToSeed(mnemonic)` on **every** invocation | Redundant work; derives all coin paths even when only one ticker is needed |

```30:31:Sparkplate.Fresh/src/utils/cryptoGenerator.ts
  const seed = await bip39.mnemonicToSeed(mnemonic);
  const root = bip32.BIP32Factory(ecc).fromSeed(seed);
```

There is **no** `HDWalletService` in `src/services/` — only a settings UI component named `HDWalletManagement.vue`, unrelated to session derivation.

## 3. What needs to be done (implementation plan)

### Step 1 — Add a session HD wallet service (Greenery `HDWalletService` port)

**Create:** `src/services/wallet/service.wallet.HDWallet.ts`

Responsibilities (mirror Greenery, TypeScript):

- Hold `phrase: string`, `seed: Buffer` (64-byte BIP-39 output), `email: string`.
- Expose `getSeed(): Buffer` and `getPhrase(): string`.
- Factory: `async function createHDWalletContext(phrase: string, email: string): Promise<HDWalletContext>`.
  - Validates phrase with `bip39.validateMnemonic`.
  - Calls `bip39.mnemonicToSeed(phrase)` **once**.
  - Returns immutable context object.
- Later (Phase B IPC): add `generateWallet({ coinTicker, network, derivationIndex })` delegating to `window.cryptos` with `this.seed`.
- Later: port `getDerivationIndex` / `incrementWalletCounter` from Greenery (email-coin-network localStorage keys).

**Do not persist `seed` to disk.** Session memory only — same threat model as the plaintext phrase ref today.

### Step 2 — Extend `useAccountsStore` to hold the HD context

**Modify:** `src/stores/useAccountsStore.ts`

Add session state alongside `mnemonic`:

```ts
/** Precomputed BIP-39 seed buffer + phrase — Greenery `accounts.hdWallet` equivalent. Session only. */
const hdWallet = ref<HDWalletContext | null>(null)

async function setHDWalletFromPhrase(phrase: string, email: string): Promise<void> {
  hdWallet.value = await createHDWalletContext(phrase, email)
  mnemonic.value = phrase.trim()
}

function getActiveSeed(): Buffer | null {
  return hdWallet.value?.getSeed() ?? null
}

function getActiveMnemonic(): string | null {
  return hdWallet.value?.getPhrase() ?? mnemonic.value
}
```

**Modify `signup`:**

```ts
async function signup(input: CreateUserInput): Promise<User> {
  …
  if (input.mnemonic?.trim()) {
    await setHDWalletFromPhrase(input.mnemonic, created.email)
  } else {
    mnemonic.value = null
    hdWallet.value = null
  }
  return user
}
```

**Modify `authenticate`:**

```ts
async function authenticate(email: string, password: string): Promise<boolean> {
  …
  const phrase = await decryptMnemonic(email, password)
  if (phrase) {
    await setHDWalletFromPhrase(phrase, email)
  } else {
    mnemonic.value = null
    hdWallet.value = null
  }
  return true
}
```

**Modify `logout` / `reset`:** clear `hdWallet.value = null` (alongside `mnemonic`).

Export `getActiveSeed`, `hdWallet` (read-only if needed).

### Step 3 — Split `cryptoGenerator` into seed-based derivation

**Modify:** `src/utils/cryptoGenerator.ts`

Add a new entry point that accepts a **precomputed seed** (the hot path):

```ts
/** Derive addresses from an already-computed BIP-39 seed buffer (no PBKDF2). */
export async function generateAddressesFromSeed(
  seed: Buffer,
  indices: Record<string, number> = {},
): Promise<CryptoAddress[]> {
  const root = bip32.BIP32Factory(ecc).fromSeed(seed)
  // … existing per-coin logic unchanged …
}

/** Legacy / convenience — computes seed then delegates. Prefer generateAddressesFromSeed in wallet flow. */
export async function generateAddressesFromMnemonic(
  mnemonic: string,
  indices: Record<string, number> = {},
): Promise<CryptoAddress[]> {
  const seed = await bip39.mnemonicToSeed(mnemonic)
  return generateAddressesFromSeed(seed, indices)
}
```

Refactor the body so coin loops live in `generateAddressesFromSeed` only — **one place** for path logic.

Optional optimization: add `generateAddressForTickerFromSeed(seed, ticker, index)` to derive **one** coin instead of all nine on every Dashboard click.

### Step 4 — Wire the wallet store to the session seed

**Modify:** `src/stores/useWalletsStore.ts`

Change `addFromMnemonic` to prefer the precomputed seed:

```ts
async function addFromSeed(
  ticker: string,
  seed: Buffer,
  options: GenerateWalletOptions = {},
): Promise<StoredWallet | null> { … }

async function addFromMnemonic(ticker, mnemonic, options) {
  // Prefer caller passing seed; fallback for throwaway / tests only
  const seed = await bip39.mnemonicToSeed(mnemonic)
  return addFromSeed(ticker, seed, options)
}
```

**Modify:** `src/views/Dashboard.vue` `onNewWalletFromMnemonic`:

```ts
const seed = accountsStore.getActiveSeed()
const mnemonic = accountsStore.getActiveMnemonic()
if (!seed || !mnemonic) throw new Error('…')
await walletsStore.addFromSeed(activeTicker.value, seed, { isHDWallet: true, index: nextHdIndex(…) })
```

Dashboard should **never** pass the phrase into derivation when the session seed is available.

### Step 5 — Align registration (optional but recommended)

**Modify:** `03.registration.mnemonicHDSeedPhrase.vue` (optional)

Greenery precomputes seed at form mount. Sparkplate can precompute after `generatePhrase()` for UX consistency (e.g. reuse seed for GPG fingerprint + avoid duplicate PBKDF2 in registration UI). This is **not required** for Dashboard fix if Step 2 runs at signup — but reduces duplicate work in `deterministicGPG.seed.ts` which also calls `mnemonicToSeed`.

Low priority unless profiling shows registration is slow.

### Step 6 — Prepare for IPC (Phase B, no blockers)

When `window.cryptos.generateWallet` lands, `HDWalletContext.generateWallet()` should pass `this.seed` exactly like Greenery — no API change needed in Dashboard or wallet store if Step 1–4 are done.

## 4. Security constraints (non-negotiable)

| Rule | Rationale |
|------|-----------|
| **Never persist the 64-byte seed** to `localStorage`, Pinia `persist`, or SQLite plaintext | Same as mnemonic — only the **encrypted phrase** (`mnemonicCipher`) is at rest |
| **Clear `hdWallet` on logout / `resetStoresOnLogout`** | Parity with Greenery session teardown |
| **Do not expose `getActiveSeed()` to UI components** | Only stores/services; never bind seed to Vue templates |
| **Seed buffer is not serializable** | If Pinia devtools or persistence plugins are extended, exclude `hdWallet` explicitly |

## 6. Acceptance criteria

1. **Signup:** After registration completes, `useAccountsStore.getActiveSeed()` returns a 64-byte `Buffer`; `getActiveMnemonic()` returns the verified phrase.
2. **Login:** After `authenticate`, same — seed computed **once** during login, not on each wallet click.
3. **Dashboard:** Clicking **New Wallet → From Mnemonic** three times runs `mnemonicToSeed` **zero** times (verify via breakpoint or spy on `bip39.mnemonicToSeed` in the wallet-generation path).
4. **Logout:** `getActiveSeed()` and `getActiveMnemonic()` both return `null`.
5. **Correctness:** Addresses produced before and after this change are **identical** for the same `(phrase, ticker, index)` — only the call site for PBKDF2 moves from per-click to per-session.
6. **IPC readiness:** `HDWalletContext` exposes `getSeed(): Buffer` in the shape Greenery's main process expects.

## 8. Summary

Greenery precomputes `{ phrase, seed }` at registration/login and stores it in `HDWalletService`, so every Dashboard wallet generation reuses the **same 64-byte seed buffer** and passes it to main-process crypto. Sparkplate keeps only the phrase string and calls `bip39.mnemonicToSeed` inside `cryptoGenerator` on **every** wallet click.

**Fix:** introduce `service.wallet.HDWallet.ts`, hold `hdWallet` in `useAccountsStore` (built at signup/login, cleared on logout), split `cryptoGenerator` into `generateAddressesFromSeed`, and route Dashboard HD generation through `getActiveSeed()` → `addFromSeed()`. The seed stays in session memory only; correctness is unchanged; performance and IPC alignment improve.
