# TypeScript diagnostics on `import { ref, reactive, watch } from 'vue'` in Calculator tab

**Date:** May 17, 2026 (`date +%m%d%Y` → `05172026`)  
**File:** [`src/components/pageTabs/cryptocurrency/tab.cryptocurrency.Calculator.vue`](../../src/components/pageTabs/cryptocurrency/tab.cryptocurrency.Calculator.vue)  
**Lines:** script setup (~L165–L166)

---

## Symptom

The Vue + TypeScript language service reports **three errors on one import line**:

| Line | Diagnostic |
|------|-------------|
| L166 | `Module '"vue"' has no exported member 'ref'.` (ts-plugin) |
| L166 | `Module '"vue"' has no exported member 'reactive'.` (ts-plugin) |
| L166 | `Module '"vue"' has no exported member 'watch'.` (ts-plugin) |

Typical offending line:

```ts
import { ref, reactive, watch } from 'vue'
```

The application **still builds and runs** with Vite; this is tooling/type-resolution noise in many setups, not a missing runtime API.

---

## Why this is surprising

Vue 3 re-exports the composition API from the main `vue` entry. Official typings resolve `vue` to `dist/vue.d.mts` / `vue.d.ts`, which **re-export** `@vue/runtime-dom`. Those modules **do** declare `ref`, `reactive`, and `watch`. So either:

- TypeScript/Volar is resolving `vue` to **incomplete or wrong typings**, or  
- The checker is evaluating the `.vue` `<script>` block in a context where module resolution differs from plain `.ts` files.

---

## Likely contributing factors (this repo)

1. **`compilerOptions.types` is narrowed** in [`tsconfig.json`](../../tsconfig.json):

   ```json
   "types": ["vite/client"]
   ```

   That limits which **ambient** `@types/*` packages are pulled in automatically. It does **not** remove named exports from the `vue` package itself, but it can interact poorly with IDE plugins or older setups that expect broader ambient typing loading.

2. **`.vue` shims** — [`src/vite-env.d.ts`](../../src/vite-env.d.ts) declares `*.vue` as `DefineComponent` from `@vue/runtime-core`. That affects the **shape of importing a `.vue` file**, not importing from `'vue'` inside `<script setup>`. It is unrelated to whether `vue` exposes `ref`, but mismatched Vue type packages occasionally show up alongside these errors during upgrades.

3. **Vue / tooling version skew** — Mismatches between `@vue/typescript-plugin`, `vue-tsc`, `typescript`, and `vue` can produce stale or incorrect diagnostics until dependencies and the IDE Vue extension agree.

---

## Verification

- **`vue` typings really export composition API:**  

  ```text
  node_modules/vue/package.json → "types": "dist/vue.d.ts"
  vue/dist/vue.d.mts → export * from '@vue/runtime-dom';
  ```

- **Runtime:** `ref`, `reactive`, and `watch` are standard Composition API APIs; removing them would break the component immediately in the browser, not only in the editor.

---

## Mitigations (if the noise persists)

_ordered from low risk to heavier changes_

1. **Reload IDE / Vue (Volar) language server** and confirm `typescript` workspace version matches project `typescript`.

2. **Ensure a single Vue version** (`npm ls vue @vue/runtime-dom`) — duplicates can confuse resolution.

3. **Loosen ambient `types`** (evaluate trade-offs): widen or remove `"types": ["vite/client"]` and rely on `/// <reference types="vite/client" />` already present in [`src/vite-env.d.ts`](../../src/vite-env.d.ts) so Vite env types remain available without starving other resolution paths.

4. **Align with the official Vite + Vue TS template**: keep `vite-env.d.ts`, `verbatimModuleSyntax` / Vue recommended `tsconfig` snippets, and current `@vue/typescript-plugin` setup docs.

---

## Scope note

Renaming **`Calculator.vue` → `tab.cryptocurrency.Calculator.vue`** does **not** change how `'vue'` is resolved; the diagnostics are anchored to the **import from `'vue'`** in `<script setup lang="ts">`, not to the filename.
