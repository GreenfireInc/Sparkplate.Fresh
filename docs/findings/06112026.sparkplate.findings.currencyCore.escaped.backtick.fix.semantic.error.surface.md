**Scope:** `src/lib/cores/currencyCore/` (26 files edited); whole-project `vue-tsc` pass
**Tooling:** `npx vue-tsc --noEmit --skipLibCheck` (the first step of `npm run build:check`)
**Category:** TypeScript syntax repair → newly-surfaced semantic typing
**Status:** Partially resolved — the **parse failures are fixed**; a large **pre-existing semantic** error surface is now visible and remains **open** (out of scope for the backtick fix)

## Executive summary

The **26** `currencyCore` files flagged by the 05/15 finding (escaped-backtick / unterminated
template-literal parse failures) have been **repaired**. After the fix, `vue-tsc` no longer aborts on those
files.

Critically, repairing the parse errors **unmasked a large latent semantic error surface**. `vue-tsc` had been
**aborting its full semantic pass** at the catastrophic syntax errors; it only ever reported the 26 syntactic
files. With the syntax fixed, the compiler completes the pass and now reports **~1,930 errors across ~512
files** — none of which are backtick-related.

| State | Total `error TS…` | Distinct files | Nature |
|-------|-------------------|----------------|--------|
| Before fix (05/15 baseline, re-measured) | **193** | **26** | parser-level (TS1005/TS1127/TS1160 …) |
| After fix | **~1,930** | **~512** | semantic (barrel conflicts, missing deps, type mismatches) |

This was verified empirically by stashing the fix and re-running (193/26), then restoring it (1,930/512).

## What was fixed (the escaped-backtick corruption)

Two opposite corruption shapes, plus two unrelated outliers. All edits were confined to 26 files
(102 insertions / 103 deletions).

### Category A — oracle/DEX `example:` doc strings (re-escape stray backticks)

These properties are **real** template literals whose content is **intentionally escaped** code samples. A few
interior backticks had **lost their backslash**, prematurely closing the string. Fix = re-escape the stray
`` ` `` → `` \` ``.

- `oracles/XTZ.Tezos/dia.ts` (×2: `…Price:` and `…Supply:` log lines)
- `oracles/XTZ.Tezos/kaiko.ts` (`…Volume Data:` log line)
- `oracles/XLM.Stellar/reflector.ts` (`…Failed to get price for …:` log line)
- `oracles/ETC.EthereumClassic/coingecko.ts` (`OHLC data for last …:` log line)
- `oracles/LUNA.Terra/astroport.ts` (`…%) | ` + concatenation join)

### Category B — `distributionEngines/*` (unescape delimiters / real code)

These are **real** code (executable methods) or **source-as-string constants** (Solidity/Rust). Their template
literals had **escaped delimiters** (`export const X = \`` … `\`;`) and/or escaped interpolation (`\${…}`) in
executable code. Fix = unescape `` \` `` → `` ` `` and `\${` → `${` wholesale (the embedded Solidity/Rust
contains neither real backticks nor `${`, so this is safe). 18 files:

```
distributionEngines/{ATOM.Cosmos,BCH.BitcoinCash,BNB.Binance,BTC.Bitcoin,DOGE.Dogecoin,
DOT.Polkadot,ETC.EthereumClassic,ETH.Ethereum}/*.{escrow,rewards}.{manual,smartContract}.ts
```

### Outliers (different corruption classes, fixed so the build could progress)

- `oracles/ATOM.Cosmos/chainlink.ts` — identifier split by a space: `valueSec ured` → `valueSecured`.
- `oracles/BNB.BinanceCoin/api3.ts` — identifier split by a blank line: `quantifiableSecur\n\nity` →
  `quantifiableSecurity`.
- `DEXs/DOGE.Dogecoin/stealthex.DOGE.ts` — the `integrationExample` template literal was never closed and was
  followed by an invalid `@DataSources:` block. Closed the string and converted `@DataSources` into a valid
  `dataSources` array property (per the 05/15 finding's §3 recommendation).

After the fix, the dominant error codes (whole-project run) are:

| Code | Count | Meaning |
|------|-------|---------|
| TS2308 | 428 | barrel `index.ts` re-exports the same member from multiple modules |

| TS2307 | 233 | **missing optional deps** (`@cosmjs/cosmwasm-stargate`, `cashscript`, `@psf/bch-js`, `@polkadot/api-contract`, …) |

| TS2740 | 21 | `Uint8Array` not assignable to `Buffer` |

These are **pre-existing latent defects** in the broad `currencyCore` provider library (auto-generated/paste-in

were simply never reachable while `vue-tsc` aborted on the syntax wall.

## Impact on the Vuex→Pinia migration (Phase 5)

- **The migration scope is type-clean.** No errors appear in `src/stores/*`, the migrated components
  (`MarqueeTicker`, `tab.cryptocurrency.Indices`, `AddressBook` + address-book tabs,
  `dropdown.authentication`, `KeyboardShortcuts`), or the security/legacy files touched by the migration.
- **`npm run build:check` cannot reach green** until the `currencyCore` semantic surface is addressed — but
  this was **already true** before the migration (the build aborted at 193 syntax errors). Phase 5's
  "zero Vuex references" and migration-scoped typecheck are satisfied; full `build:check` acceptance is
  **blocked by this finding**, not by the migration.

**Decision (20260611):** treat Phase 5 typecheck as **migration-scope clean** and track full `build:check`
acceptance against this finding.

## Suggested remediation (future, separate effort)

1. **Decide the boundary of `currencyCore` in the typecheck.** Much of it is provider scaffolding that
   references uninstalled optional deps. Options: (a) `tsconfig` `exclude` (or a separate, looser project) for
   `currencyCore` provider trees so `build:check` reflects shippable app code; (b) install the missing optional
   deps and fix types; (c) convert source-as-string constants to externalized `?raw` artifacts.
2. **Barrel conflicts (TS2308/TS2305, ~837 combined).** The `*/index.ts` aggregators re-export colliding member
   names; namespace the re-exports or rename.
3. **Duplicate object keys (TS1117).** Quick, safe wins in `goracle.ts`, `band.ts`, `totle.ts`,
   `menuTranslations.ts`, `useMenuTranslations.ts`, `locales/menuTranslations.ts`.
4. **Add a `currencyCore`-scoped CI typecheck** (per the 05/15 finding) so paste-in regressions can't silently
   reappear.
