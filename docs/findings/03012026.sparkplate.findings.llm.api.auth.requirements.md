# Findings: LLM Provider API Authentication Requirements

**Date:** 2026-03-01  
**Context:** `src/components/modals/settings/apis/LlmModal.vue` — each LLM entity currently defines three credential fields: `apiKey`, `apiSecret`, and `apiPassphrase`. This document records whether any provider actually requires the second or third credential.

---

## Summary

**None** of the 11 LLM providers in `LLM_ENTITIES` require an API secret or passphrase in addition to the initial API key. All use single-key Bearer token authentication.

## Authentication Details

All providers use HTTP **Bearer token** authentication with a single API key:

```
Authorization: Bearer <API_KEY>
```

The exception is **Claude**, which uses the custom header `x-api-key` rather than the `Authorization` header, but still requires only one credential.

**Manus** uses the header `API_KEY: <key>` rather than `Authorization: Bearer`, but again, only one credential.

---

## Recommendation

The `apiSecret` and `apiPassphrase` storage fields in `LLM_ENTITIES` are **unused** by all current providers. The `apiKey.2` and `apiKey.3` slots in the modal serve no functional purpose for any of these LLMs.

Options to consider:
1. **Remove** the `apiSecret` and `apiPassphrase` fields from `LLM_ENTITIES` and the `apiKey.2` / `apiKey.3` slots from the modal entirely.
2. **Repurpose** the extra slots as user-defined custom fields (e.g. org ID, project ID, or a secondary key for a different environment such as sandbox vs. production).
3. **Keep as-is** for forward compatibility in case a future provider added to the list does require multi-credential auth.
