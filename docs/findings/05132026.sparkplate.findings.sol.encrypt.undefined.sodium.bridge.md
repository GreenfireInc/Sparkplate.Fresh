# Solana SPCK encrypt: `cannot read properties of undefined (reading 'sodium')`

**Date:** May 13, 2026  
**Component:** SPCK v2 (`encryptWithEd25519PublicKey` / Edwards→Montgomery for SOL & XTZ)  
**Primary file:** `src/lib/cores/cryptographyCore/encryption/encryption.crypto.PublicKey.ed25519.ts`

---

## Symptom

Encrypting a file to a **Solana** address (Base58-encoded 32-byte Ed25519 public key) surfaces a failure similar to:

> Failed to encrypt file: cannot read properties of undefined (reading **'sodium'**)

The stack points into the libsodium wrappers bootstrap (`libsodium-wrappers-sumo`), not into Base58 decoding or HKDF/WebCrypto layers.

## Root cause (hypothesis, verified by fix)

1. **Encrypt path** depended on **`libsodium-wrappers-sumo`** only for the standard map **Ed25519 public key → X25519 (Montgomery) public key** (`crypto_sign_ed25519_pk_to_curve25519`).
2. That package’s bundled bootstrap expects internal objects (often exposed as **`sodium`** on a module-scope holder) to be defined once `libsodium-sumo` has loaded.
3. Under some **Vite + Rollup/CommonJS interop** combinations—or when nested `require('libsodium-sumo')` does not attach the WASM bundle the wrappers expect—the runtime object accessed as `.sodium` is **`undefined`**, so the next property read throws.

This is orthogonal to Solana pubkey parsing: a valid Base58 key still triggers the libsodium bootstrap before ECDH proceeds.

---

## Fix (implemented in Sparkplate.fresh)

**Stop importing libsodium for SPCK v2 entirely** for the Edwards→Montgomery steps.

`@noble/curves` already exposes the same cryptography on the primitives used elsewhere in SPCK v2 (`x25519`):

| Operation | Noble API | Equivalent libsodium helper |
|-----------|-----------|-----------------------------|
| Ed25519 **pub** → X25519 **pub** | `ed25519.utils.toMontgomery(pub32)` | `crypto_sign_ed25519_pk_to_curve25519` |
| Ed25519 **seed** → X25519 **secret** | `ed25519.utils.toMontgomerySecret(seed32)` | `crypto_sign_ed25519_sk_to_curve25519` on expanded secret derived from seed |

Byte-for-byte agreement with libsodium **0.7.15** for random test vectors was validated in Node (same `pk → u` and `seed → clamped scalar` outputs).

Envelope layout, HKDF `info`, and AES-GCM remain unchanged; decrypt is compatible with ciphertext produced by the libsodium-backed implementation.

---

## Verification checklist

1. Encrypt a short payload to a **Solana** Base58 pubkey; confirm success (no libsodium bootstrap).
2. Decrypt with the matching **Solana seed** (32-byte secret or 64-byte `seed ‖ pubkey`).
3. **Regression:** Encrypt/decrypt round-trip for a **tz1 hex** pubkey and Tezos **`edsk...`** recipient key (same module path).
