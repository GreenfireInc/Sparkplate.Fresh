# Methodology: Resolving Electron Network / GPU / Zygote Crash Logs on Linux

**Date:** April 6, 2026  
**Related finding:** `docs/findings/04062026.sparkplate.findings.electron.network.gpu.zygote.linux.log.md`  
**Target:** `background/main/index.ts` (Electron main process)  
**Platform:** Linux (X11 and Wayland sessions)

---

## Problem

The following error triad appears in stderr during development and sometimes production runs:

```
Network service crashed, restarting service.
GPU process exited unexpectedly: exit_code=15
Failed to send GetTerminationStatus message to zygote
```

As documented in the findings doc, these three messages share a **single root cause** — they are a cascade: the GPU process is signaled (SIGTERM, code 15), which causes the network service subprocess to lose ground and restart, and the zygote IPC then breaks because the process tree is in a partial teardown state.

---

## Diagnosis checklist (run first)

Before making code changes, confirm which scenario applies:

| Check | Command | Benign if… |
|---|---|---|
| Logs appear only on window close | Observe in terminal | App exits cleanly afterward |
| `dmesg` GPU / OOM | `sudo dmesg \| grep -iE 'gpu\|oom\|killed'` | No recent GPU reset or OOM killer |
| GPU process signal | Note `exit_code` in log | `15` = SIGTERM (usually orderly) |
| Driver crash | `journalctl -b \| grep -i drm` | No DRM reset events |
| Wayland vs X11 | `echo $WAYLAND_DISPLAY $DISPLAY` | Identify display session |

**If logs only appear at shutdown/quit → they are benign and no code change is required.** Document them and proceed.  
**If they appear mid-session, on launch, or cause a blank window → continue with the resolution steps below.**

---

## Resolution 1 — Ordered, safe Chromium switches (apply always)

The existing `background/main/index.ts` already carries three GPU switches added during prior GPU troubleshooting:

```ts
// current (lines 35–37)
app.commandLine.appendSwitch('--disable-gpu-sandbox')
app.commandLine.appendSwitch('--disable-software-rasterizer')
app.commandLine.appendSwitch('--disable-gpu-compositing')
```

These should remain. Supplement them with switches that stabilise the GPU process lifecycle and the network sandbox on Linux **before** `app.whenReady()`:

```ts
// Add after existing GPU troubleshooting flags

if (process.platform === 'linux') {
  // Disable GPU sandbox isolation — prevents SUID sandbox failures on some distros
  app.commandLine.appendSwitch('no-sandbox')
  // Use Mesa / llvmpipe as software fallback instead of crashing when no GPU driver
  app.commandLine.appendSwitch('use-gl', 'swiftshader')
  // Prevent Chromium from sharing the GPU process (reduces cross-process kill cascade)
  app.commandLine.appendSwitch('in-process-gpu')
  // Suppress the zygote on systems that don't support it (e.g. Docker, some Flatpak envs)
  app.commandLine.appendSwitch('no-zygote')
}
```

> **Caution:** `no-sandbox` reduces Chromium isolation. Use it only if the app does not load untrusted remote content (which Sparkplate does not in its current architecture). Do not apply globally to non-Linux platforms.

> **`in-process-gpu`** moves the GPU process into the main browser process. This eliminates the "GPU process exited unexpectedly" message because there is no separate GPU subprocess. Performance is slightly lower but stability on commodity Linux hardware improves significantly.

> **`no-zygote`** disables the forking layer entirely. This is safe for desktop Electron apps that do not rely on sandbox isolation through the zygote.

## Resolution 2 — Graceful shutdown to prevent cascade on quit

The zygote and network-service messages often appear because the app exits without waiting for `webContents` to finish. Add a controlled teardown:

```ts
// In background/main/index.ts, replace or augment window-all-closed:

app.on('before-quit', (event) => {
  if (win && !win.isDestroyed()) {
    event.preventDefault()
    win.webContents.executeJavaScript('window.__sparkplateBeforeQuit && window.__sparkplateBeforeQuit()')
      .catch(() => {})
      .finally(() => {
        win?.destroy()
        app.quit()
      })
  }
})

app.on('window-all-closed', () => {
  win = null
  if (process.platform !== 'darwin') app.quit()
})
```

This gives the renderer a brief window to flush before Electron destroys subprocess handles, which removes the IPC-broken zygote message in most orderly-exit cases.

## Resolution 3 — Linux-specific GPU flags scoped by session type

For Wayland sessions, Chromium's GPU process behaves differently from X11. Add session detection to apply the right fallback:

```ts
import { execSync } from 'node:child_process'

function getLinuxDisplaySession(): 'wayland' | 'x11' | 'unknown' {
  const wayland = process.env.WAYLAND_DISPLAY
  const xdg = process.env.XDG_SESSION_TYPE
  if (wayland || xdg === 'wayland') return 'wayland'
  if (process.env.DISPLAY || xdg === 'x11') return 'x11'
  return 'unknown'
}

if (process.platform === 'linux') {
  const session = getLinuxDisplaySession()

  if (session === 'wayland') {
    // Native Wayland — use Ozone backend
    app.commandLine.appendSwitch('ozone-platform', 'wayland')
    app.commandLine.appendSwitch('enable-features', 'UseOzonePlatform,WaylandWindowDecorations')
  } else {
    // X11 or unknown — keep existing GPU compositing flags
    app.commandLine.appendSwitch('--disable-gpu-compositing')
  }

  app.commandLine.appendSwitch('no-sandbox')
  app.commandLine.appendSwitch('in-process-gpu')
  app.commandLine.appendSwitch('no-zygote')
}
```

Remove the existing unconditional `--disable-gpu-compositing` switch if this block is used (it would conflict on the Wayland path).

## Recommended implementation in `background/main/index.ts`

The consolidated replacement for the GPU block at lines 34–37 (and the new teardown) is:

```ts
// ─── Platform: Linux GPU + subprocess stability ───────────────────────────
if (process.platform === 'linux') {
  const isWayland =
    !!process.env.WAYLAND_DISPLAY || process.env.XDG_SESSION_TYPE === 'wayland'

  if (isWayland) {
    app.commandLine.appendSwitch('ozone-platform', 'wayland')
    app.commandLine.appendSwitch('enable-features', 'UseOzonePlatform,WaylandWindowDecorations')
  } else {
    app.commandLine.appendSwitch('--disable-gpu-compositing')
  }

  // Move GPU work into the browser process — eliminates "GPU process exited unexpectedly"
  app.commandLine.appendSwitch('in-process-gpu')
  // Disable zygote forking — eliminates zygote IPC errors on unsupported envs
  app.commandLine.appendSwitch('no-zygote')
  // Disable sandbox (safe for trusted desktop app with no remote web content)
  app.commandLine.appendSwitch('no-sandbox')
}

// Keep these on all platforms (existing GPU rendering workarounds)
app.commandLine.appendSwitch('--disable-gpu-sandbox')
app.commandLine.appendSwitch('--disable-software-rasterizer')
// ──────────────────────────────────────────────────────────────────────────
```

## Priority matrix

| Symptom | Primary resolution | Expected outcome |
|---|---|---|
| Messages only on app exit | No code change needed | Cosmetic only |
| Messages on every launch | Resolution 1 (`in-process-gpu`, `no-zygote`) | Eliminates subprocess crash logs |
| Blank window on launch | Resolution 1 + try `--disable-gpu` fallback (`app.disableHardwareAcceleration()`) | Forces software render |
| Messages during Wayland session | Resolution 3 (session detection + `ozone-platform`) | Proper Wayland path |
| Crashes on Docker / Flatpak / CI | Resolution 1 (`no-sandbox`, `no-zygote`) | Common container fix |

---

## What not to do

- **Do not** add `--disable-gpu` globally — it disables hardware acceleration on all platforms including Windows/macOS and degrades rendering.
- **Do not** use `app.disableHardwareAcceleration()` unconditionally — it must be guarded or scoped to specific conditions (e.g. Windows 7 as already in the file).
- **Do not** call `app.commandLine.appendSwitch` after `app.whenReady()` — switches must be set before the app is ready or they are ignored by Chromium.

---

## Testing

After applying changes, verify:

1. `npm run dev` — confirm no new errors in the terminal.
2. Check stderr: `GPU process exited unexpectedly` and `zygote` lines should be absent during normal use.
3. On exit: `Network service crashed` may still briefly appear; if it does not block exit it is acceptable.
4. If using Wayland: confirm window renders correctly (`echo $WAYLAND_DISPLAY` should be non-empty; check `ozone-platform=wayland` appears in Chromium internals).
