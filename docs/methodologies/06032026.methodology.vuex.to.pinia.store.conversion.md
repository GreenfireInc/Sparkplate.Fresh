# Vuex → Pinia Store Conversion Methodology

**Date**: June 3, 2026 (`06032026`)
**Reference codebase**: `greenery/` (Vue 2.7 + Vuex 3 + Electron)
**Target application**: Sparkplate.Fresh (Vue 3.5 + Vite 6 + Pinia + Electron 37, TypeScript)
**Related methodology**: `03252026.methodology.addressbook.page.integration.md`, `10192025.methodology.sqlite3.database.implementation.md`
**Related rules**: `.rules/migrationRules.md`, `.rules/vueMigrationTroubleshooting.md`, `.rules/vuePackageInstallationSteps.md`

---

## Executive Summary

Greenery is a cryptocurrency wallet whose entire application state lives in a **Vuex 3 store**: one root store plus **12 registered namespaced modules** (a 13th, `markets`, is dead code). The store persists a slice of root state to **SecureLS** via `vuex-persistedstate` and shares `settings` across windows via `vuex-shared-mutations`. Many actions reach into the Electron main process through `window.fs`, `window.cryptos`, `window.ipcRenderer`, `window.storage`, `window.notification`, `window.app`, and `window.geoip`.

Sparkplate.Fresh is **Vue 3 + Vite + TypeScript** with **no Pinia today**. Shared state currently lives in **module-level composables** (e.g. `useDashboardCurrencies.ts`, `useMenuState.ts`) and **localStorage-backed services** (`src/services/addressBook/*`). The renderer talks to Electron only through a narrow `contextBridge` (`window.ipcRenderer`, `window.app`) — **`window.fs` and `window.cryptos` do not exist**.

This document:

1. Inventories **every Vuex module** and **every Greenery view's store usage**.
2. Defines the **target Pinia architecture** for Sparkplate (setup-style stores, TypeScript, `pinia-plugin-persistedstate`).
3. Gives **mechanical conversion patterns** (state/getters/mutations/actions/`mapX` → Pinia).
4. Calls out the **Electron `window.*` boundary** each module crosses and how to satisfy it in Sparkplate.
5. Lays out a **phased migration plan** with a per-view checklist and acceptance criteria.

> This is a planning artifact. Implement per Sparkplate conventions in minimal-scope, build-verified PRs (`npm run dev` + `npm run build` after each phase, per `.rules/migrationRules.md`).

### 1.4 Global mixin (often missed)

`greenery/src/mixins.js` registers a **global mixin** that injects store-backed computeds into *every* component:

- `mapState` for `accounts`, `coinsData`, `coinsInfo`, and `contacts` (`state.contacts.list`)
- `loggedIn` / `loggedUserData` / `userCurrency` from `state.accounts.active`
- `getDisplayCurrencyFormat` reads `state.coinsMeta`

Views with **no** explicit Vuex import still depend on the store through this mixin. The mixin must become **composables** in Sparkplate (`useAccount()`, `useCoins()`, `useContacts()`), since Vue 3 + Pinia favors composables over global mixins (`.rules/migrationRules.md` §6).

## 3. Target architecture in Sparkplate.Fresh

### 3.1 What already exists

- **No Pinia, no `src/stores/`.** Shared state = composables (`src/composables/`) + services (`src/services/`).
- `useDashboardCurrencies.ts` is an explicit, documented re-creation of Greenery's `userSettings.visibilityToggles` using a module-level `ref` + localStorage. **It is the canonical "store-like" pattern already blessed in this repo.**
- Renderer↔Electron bridge is **narrow**: `window.ipcRenderer` (`on/off/send/invoke`) and `window.app` (`getPreloadData`, `getGPUInfo`, `getUsbDrives`, `getNetworkAdapters`, versions). Defined in `background/preload/index.ts`, typed in `src/types/background.d.ts` / `src/vite-env.d.ts`.
- **`window.fs`, `window.cryptos`, `window.storage`, `window.notification`, `window.geoip` do NOT exist.** Two legacy Greenery-era SFCs (`TwoFactorAuth.vue`, `MiscSecurity.vue`) still `import 'vuex'` and call `window.fs` — they are broken stubs and must be rewritten or excluded.

### 3.2 Decision: setup-style Pinia stores in TypeScript

Use **setup stores** (`defineStore(id, () => { ... })`) — they match Sparkplate's Composition API majority and the KeyForge reference store, and convert cleanly to/from the existing composables. (Options stores from `.rules/vueMigrationTroubleshooting.md` are acceptable but less consistent here.)

Mapping of Vuex concepts → Pinia setup store:

| Vuex | Pinia setup store |
|------|-------------------|
| `state: () => ({...})` | `const x = ref(...)` per key |
| `getters` | `const g = computed(() => ...)` |
| `mutations` | **deleted** — mutate `ref.value` directly inside actions/functions |
| `actions` | plain functions (may be `async`) |
| `rootState.foo` / `rootGetters['bar']` | `const other = useOtherStore()` then `other.x` |
| `commit('m', p)` | call the function / assign the ref |
| `dispatch('mod/a', p)` | `useModStore().a(p)` |
| `mapState`/`mapGetters` in component | `const s = useStore(); storeToRefs(s)` |
| `mapActions` in component | `const { a } = useStore()` |
| `vuex-persistedstate` | `persist: { key, pick: [...] }` (plugin) |
| `vuex-shared-mutations` | `pinia-plugin-persistedstate` + `storage` event (already used by `useDashboardCurrencies`) |

### 3.5 Persistence mapping

Greenery persisted only `settings`, `coinsData`, `coinsInfo`, `coinsMeta` (SecureLS). Map to:

| Greenery persisted path | Sparkplate store | `persist` config |
|-------------------------|------------------|------------------|
| `settings.menuType` | `useSettingsStore` | `persist: { key: 'sparkplate_settings', pick: ['menuType', ...] }` |
| `coinsData`/`coinsInfo`/`coinsMeta` | `useCoinsStore` | `persist: { key: 'sparkplate_coins' }` |

Keep service-owned data (address book, etc.) in its **existing localStorage services**; do not double-persist. Cross-window sync (formerly `vuex-shared-mutations`) is handled by the persistedstate plugin's `storage` event, mirroring `useDashboardCurrencies`.

## 4. Worked conversion example

Greenery `settingsModule.js` (abridged) → `useSettingsStore.ts`:

```typescript
// src/stores/useSettingsStore.ts
import { defineStore } from 'pinia'
import { ref } from 'vue'
import { settingsService } from '@/services/settings/service.settings'

export const useSettingsStore = defineStore('settings', () => {
  // state  (was: state: () => ({...}))
  const menuType = ref<'micro' | 'macro'>('micro')
  const visibilityToggles = ref<Record<string, boolean>>({})
  const networkSelection = ref<Record<string, string>>({})
  const notificationsEnabled = ref(true)

  // actions  (mutations folded in — mutate .value directly)
  function changeMenuType(type: 'micro' | 'macro') {
    menuType.value = type
  }

  async function loadSettings(userId: string) {
    const dbSettings = await settingsService.getUserSettings(userId)
    if (dbSettings) Object.assign({ menuType, notificationsEnabled }, dbSettings)
  }

  async function updateSetting(setting: string, value: unknown) {
    // local mutation + persistence side-effect
    ;(<Record<string, unknown>>{ menuType, notificationsEnabled })[setting] = value
    await settingsService.updateUserSettings(setting, value)
  }

  return { menuType, visibilityToggles, networkSelection, notificationsEnabled,
           changeMenuType, loadSettings, updateSetting }
}, {
  persist: { key: 'sparkplate_settings', pick: ['menuType', 'visibilityToggles', 'networkSelection'] },
})
```

Component usage (was `mapState`/`mapActions`):

```vue
<script setup lang="ts">
import { storeToRefs } from 'pinia'
import { useSettingsStore } from '@/stores/useSettingsStore'

const settings = useSettingsStore()
const { menuType, visibilityToggles } = storeToRefs(settings) // reactive state/getters
const { changeMenuType } = settings                            // actions destructure directly
</script>
```

Cross-store reference (was `rootGetters`/`rootState`) — e.g. `wallets.totalAssetsValue` reads coins + settings:

```typescript
// inside useWalletsStore setup
const coins = useCoinsStore()
const settings = useSettingsStore()
const totalAssetsValue = computed(() => /* uses coins.coinsData + settings.visibilityToggles */ 0)
```

## 5. Electron boundary — the hard part

Every Vuex action that calls a `window.*` API that **does not exist in Sparkplate** must be re-bridged. For each, add an `ipcMain.handle` in `background/main/index.ts`, a `contextBridge` method in `background/preload/index.ts`, and a type in `src/types/background.d.ts`.

| Greenery `window.*` | Used by modules | Sparkplate plan |
|---------------------|-----------------|-----------------|
| `window.fs.*` (read/write/checkFileExists) | root `fetchCoinsInfo`, exports | Add `ipcMain.handle('fs:read/write/exists')` + preload `window.fs`; or use existing `src/lib/cores/exportStandard` + localStorage where a file isn't required |
| `window.cryptos.*` | fees, wallets, transactions, paperWallet | Largest effort: port Greenery crypto layer to main process + IPC, or to `src/lib/cores/cryptographyCore`. Treat as its own methodology. |
| `window.storage.encryptBuffer` | contacts/wallets export | Bridge via IPC or reuse `cryptographyCore` |
| `window.notification.newNotification` | wallets `updateBalance` | Electron `Notification` via IPC, or web Notification API |
| `window.ipcRenderer.send(...)` | accounts `setAuthenticated` | Already exists as `window.ipcRenderer` — map channel names |
| `window.app.fetchIP` | accounts `fetchIP` | Add to existing `window.app` (`getPreloadData` pattern) |
| `window.geoip.lookup` | accounts `signup` | Add IPC handler or drop if not needed in v1 |

**Rule:** never assume a `window.*` API exists in the renderer — `nodeIntegration` is off on the main window. Audit each action before porting.

## 6. Defects to fix during the port (do not copy verbatim)

1. `activityModule` getter `value` and `settingsModule` getter `value` read non-existent state — drop them.
2. `invoicesModule.fetchInvoices` reads `rootState.active.id`; should be `accounts.active.id` → fix to `useAccountsStore().active?.id`.
3. `quickExchange` has `resetQuickExchangeHistoryState` but logout never calls it — add to the logout flow / `$reset()`.
4. Dashboard maps `paperWallet/addSelectedWallet` but never calls it — drop.
5. Home maps `getTotalMarketCap` which **isn't a store action** — keep it as a factory/composable call, don't recreate a phantom action.
6. `web3Connections.endAllSessions` references `state.walletBeacon.connector` while state uses `client` — fix the key.
7. `realMarketModule` (`markets`) is unregistered dead code — skip entirely.
8. `wallets` state is the coin map at the root of the module (`rootState.wallets[ticker]`, not `rootState.wallets.wallets`) — keep that shape when reading from other stores.

## 7. Phased migration plan

### Phase 0 — Prerequisites
- [ ] `npm install pinia pinia-plugin-persistedstate --save-exact`.
- [ ] Create `src/stores/index.ts`; wire `app.use(pinia)` in `src/main.ts` before router.
- [ ] Establish `src/services/` modules for the data layer each store needs (mirror `services/addressBook/*` pattern), or stub them.
- [ ] Baseline: `npm run dev` + `npm run build` green.

### Phase 1 — Foundational stores (no Electron crypto)
- [ ] `useSettingsStore` (replace/absorb `useDashboardCurrencies` + `useMenuState`).
- [ ] `useCoinsStore` (root coin data; `fetchCoinsInfo` via IPC `window.fs` or service).
- [ ] `useAccountsStore` (state + `user`; defer `signup/login` crypto to Phase 4).
- [ ] Convert the **global mixin** into composables (`useAccount`, `useCoins`, `useContacts`).

### Phase 2 — Domain data stores (service-backed, no `window.cryptos`)
- [ ] `useContactsStore`, `useInvoicesStore`, `useActivitiesStore`, `useQuickExchangeStore`, `useMnemonicPasswordsStore`, `useExchangesStore`.
- [ ] Back each with a localStorage/SQLite service (align with `10192025.methodology.sqlite3.database.implementation.md`).

### Phase 3 — Views
Convert each route view to `<script setup lang="ts">` + store hooks, in ascending complexity:
- [ ] About, Settings, Storage, QuickExchange, PaperWallet/Index, PaperWallet/Selection (shells — mostly child wiring).
- [ ] Transactions, Calculator, Home, Exchange, Web3Connect.
- [ ] AddressBook (contacts actions).
- [ ] **Dashboard** and **History** last (heaviest store consumers).

### Phase 4 — Electron crypto bridge + wallet/tx/fees/paperWallet/web3 stores
- [ ] Build `window.cryptos` / `window.fs` / `window.storage` / `window.notification` IPC bridges (own PRs, see §5).
- [ ] `useWalletsStore`, `useTransactionsStore`, `useFeesStore`, `usePaperWalletStore`, `useWeb3ConnectionsStore`.
- [ ] Rewrite/replace legacy `TwoFactorAuth.vue` and `MiscSecurity.vue`.

### Phase 5 — Persistence, cross-window, hardening
- [ ] Configure `persist` for settings + coins; verify cross-window sync via `storage` event.
- [ ] Add `store.$reset()` to the logout flow for every store (parity with Greenery `reset*State` mutations).
- [ ] Remove `vuex`, `vuex-persistedstate`, `vuex-shared-mutations` references; delete dead code.
- [ ] `npm run build` + Electron packaging smoke test.
