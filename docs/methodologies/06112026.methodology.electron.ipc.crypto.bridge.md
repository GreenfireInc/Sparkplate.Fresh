## 0. Why this exists

The Vuex→Pinia migration has finished every item that does **not** require the Electron crypto/filesystem
boundary (Phases 0–3 + the address-book Phase 2 store + Phase 5 reset/persistence). The remaining "live"
stores (`useTransactionsStore`, `useFeesStore`, `useWeb3ConnectionsStore`, `usePaperWalletStore`, and the
*spendable* half of `useWalletsStore`) all depend on `window.*` APIs that **do not exist** in Sparkplate's
renderer.

Greenery reached the main process through `window.fs`, `window.cryptos`, `window.storage`,
`window.notification`, `window.app`, and `window.geoip`. Sparkplate today exposes only **`window.ipcRenderer`**
and **`window.app`**. This methodology defines how to build the missing bridges as small, typed, independently
shippable PRs.

> **Security baseline (already correct):** the main window runs with `contextIsolation` on and
> `nodeIntegration` off (`background/main/index.ts`). All renderer access to Node/Electron **must** go through
> `contextBridge` + `ipcMain.handle`. Never expose raw `fs`/`ipcRenderer`/Node modules to the renderer.

## 1. The pattern already in the repo (extend, don't reinvent)

Sparkplate has one consistent IPC idiom. Every new bridge follows it exactly:

1. **Main handler** — `ipcMain.handle('<namespace>:<verb>', async (_evt, args) => …)` in
   `background/main/index.ts` (see `preloadAppData`, `getNetworkAdapters`, `getUsbDrives`).
2. **Preload bridge** — `contextBridge.exposeInMainWorld('<namespace>', { verb: (args) => ipcRenderer.invoke('<namespace>:<verb>', args) })` in `background/preload/index.ts` (see the existing `app` bridge).
3. **Renderer types** — augment `declare global { interface Window { … } }` in `src/types/background.d.ts`
   (see the existing `app` typing).
4. **Consumer** — a Pinia store/service calls `window.<namespace>.<verb>(...)`; never `ipcRenderer` directly.

**Conventions to standardize now (apply to all four bridges):**

- **Channel names:** `'<namespace>:<verb>'` lower-camel verb (`'cryptos:getBalances'`, `'fs:readFile'`).
- **Always `invoke`/`handle`** (request/response). Reserve `send`/`on` for main→renderer push (e.g. balance
  update events) and type those as typed event subscriptions.
- **Serializable boundary only:** no `Buffer`/class instances across IPC — use `base64`/`hex` strings or plain
  objects, decode on the receiving side. (Resolves the `Uint8Array`↔`Buffer` class of issues seen in
  `currencyCore`.)
- **Uniform result envelope:** handlers return `{ ok: true, data } | { ok: false, error: string }`; the preload
  unwraps or throws so store code can `try/catch` normally. No raw exceptions cross the boundary.
- **One file per namespace in main:** extract handlers into `background/main/ipc/<namespace>.ts` and register
  them from `index.ts` (keeps `index.ts` from ballooning; mirrors the existing modular `functions/utils/*`).

## 2. The four bridges (what each must expose)

Derived from methodology §5 and the wallet findings. **Scope each to the minimum the V2 features need** — do
not port Greenery's full surface speculatively.

### 2.1 `window.fs` — filesystem (smallest, lowest risk)
| Verb | Purpose | Consumers |
|------|---------|-----------|
| `readFile(path, enc?)` | read app data / exports | coins warm-data, import flows |
| `writeFile(path, data, enc?)` | write exports / backups | export flows, HD backup |
| `exists(path)` | guard reads | imports |
| `pickSave(opts)` / `pickOpen(opts)` | native save/open dialog | exports, paper-wallet backup |

> Many "fs" needs are already covered by `src/lib/cores/exportStandard` + localStorage. **Only bridge what
> truly needs disk** (native dialogs, large files, USB targets — note `getUsbDrives` already exists).

### 2.2 `window.notification` — native notifications (small)
| Verb | Purpose | Consumers |
|------|---------|-----------|
| `notify({ title, body, icon? })` | balance/tx alerts | `useWalletsStore.updateBalance`, tx confirmations |

> Implement with Electron `Notification` in main. Web `Notification` API is a fallback but loses parity on
> packaged builds.

### 2.3 `window.storage` — encrypted secret custody (medium, security-critical)
| Verb | Purpose | Consumers |
|------|---------|-----------|
| `encrypt(plaintext, key)` → ciphertext | encrypt seed/WIF/privKey before persistence | wallets, paperWallet, seed custody |
| `decrypt(ciphertext, key)` → plaintext | read secrets for signing | wallets, transactions |

> This is the bridge that unblocks **spendable** wallets. It pairs with the SQLite + field-encryption work in
> the login-auth finding and the SQLite methodology. **Decision required (see §5):** keep crypto in main vs
> reuse `src/lib/cores/cryptographyCore` in the renderer.

### 2.4 `window.cryptos` — blockchain crypto (largest, own sub-PRs)
| Verb | Purpose | Consumers |
|------|---------|-----------|
| `generateWallet({ seed, coinTicker, network, derivationIndex })` | deterministic HD address | `useWalletsStore` "From Mnemonic" |
| `generateBasicWallet({ coinTicker, network })` | random keypair | `useWalletsStore` "Throwaway" |
| `getBalances({ coinTicker, address, network })` | live balance | Dashboard footer, `useWalletsStore` |
| `txFee` / `getFeeData` | fee estimation | `useFeesStore` |
| `sendToAddress(...)` / `signTransaction(...)` | spend | `useTransactionsStore`, send flow |

> **This is its own methodology-sized effort.** It is the gating dependency for `useTransactionsStore`,
> `useFeesStore`, balances, and send. `background/functions/utils/blockchainApi/*` (infura/alchemy/blockchair/…)
> already exists in main and is the natural home for `getBalances`/fees. Address derivation can reuse
> `src/lib/cores/currencyCore`/`cryptographyCore` (renderer) **or** be moved to main per §5.

## 3. Security model decision (must precede `storage`/`cryptos`)

Two viable models; pick one before building 2.3/2.4 (the wallet finding §5 recommends the main-process model
for a real wallet):

| | A. Main-process crypto (Greenery parity) | B. Renderer crypto (current `cryptoGenerator`) |
|--|------------------------------------------|------------------------------------------------|
| Secrets location | Main process only; never in renderer heap | Renderer (exposed to any XSS/devtools) |
| Node-native libs | In main (allowed) | Must run in renderer (some won't) |
| Effort | Higher (full IPC surface) | Lower (already partly built) |
| Recommendation | **Preferred for spendable wallets** | Acceptable only for display-only addresses |

**Recommended:** Model A for `storage` + `cryptos` (secrets + signing in main); the renderer holds only public
material and ciphertext. This aligns with §5 of the methodology ("never assume a `window.*` API exists… derive
secrets out of the renderer").

## 5. Open decisions (resolve in P4.3 / before P4.6)

1. **Crypto location** — Model A (main) vs B (renderer). *Recommended: A for secrets/signing.*
2. **Persistence backend for secrets** — extend localStorage pattern vs adopt SQLite
   (`10192025…sqlite3…`). *Recommended: SQLite for encrypted secret fields; localStorage stays for public/UI.*
3. **Coin coverage** — fix the non-standard TRX/XTZ/LUNC encoders (finding §4 item 6) or gate those coins until
   real encoders land.
4. **Network selection** — add a settings-backed mainnet/testnet per coin (`useSettingsStore.networkSelection`
   already exists) and thread `network` through derivation/balances.
5. **`window.geoip` / `window.app.fetchIP`** — needed by Greenery `accounts.signup`/`fetchIP`; decide whether
   V2 needs geo/IP at signup or drop it (extend existing `window.app` if kept).

## 6. Acceptance criteria

- Each bridge: typed in `background.d.ts`, reachable via `window.<ns>.<verb>`, no raw `ipcRenderer`/Node in
  renderer, errors surfaced through the result envelope.
- `contextIsolation` remains **on**; no `nodeIntegration` regressions on the main window.
- Consuming stores type-check clean (migration-scope typecheck stays green).
- Dashboard footer shows real balances (post P4.7); "From Mnemonic" vs "Throwaway" behave distinctly
  (post P4.6); secrets are persisted **encrypted only** (post P4.4/P4.5).
- `npm run dev` green after each PR; full `npm run build:check` tracked separately against the `currencyCore`
  semantic-surface finding (`06112026…escaped.backtick.fix.semantic.error.surface.md`).
