# Methodology: Cryptocurrency-Aware Encrypt/Decrypt Tab in `Cryptography.vue`

**Date:** May 19, 2026
**Project:** Sparkplate.fresh
**Target view:** `src/views/Cryptography.vue`
**New tab component:** `src/components/pageTabs/cryptography/tab.cryptography.tool.CryptocurrencyEncryptDecrypt.vue`
**Source of behaviour:** `00.references/From.sparkplate.storage/src/App.vue`
**Currency reference:** `00.references/From.sparkplate.storage/src/lib/cores/currencyCore/currencies/`

---

## 1. Overview

This methodology documents the integration of the cryptocurrency-aware file
encryption / decryption workflow from `From.sparkplate.storage`'s `App.vue` into
Sparkplate.fresh as a brand-new tab on the **Cryptography** page. It also
records the small but real correctness deltas brought over from the reference's
`currencyCore/currencies/` modules and the broken-import repairs that surfaced
once the new tab started exercising the SPCK decrypt path.

The new tab is intentionally narrower in scope than the original reference: it
focuses purely on **cryptocurrency-keyed encryption / decryption** (SPCK v1 +
v2 with an OpenPGP armored fallback), driven by a `%ticker%://%privateKey%`
URI. The PGP-only validation / generation flows already exist in the
neighbouring tabs (`PGP/GPG Keys`, `Generate`) and were left untouched.

## 3. New tab structure

`tab.cryptography.tool.CryptocurrencyEncryptDecrypt.vue` is a single-file
component built with the project's standard Radix Vue primitives
(`Label`, `Separator`, `Tooltip*`) and the existing Tailwind-flavoured
Sparkplate.fresh scoped-class convention (`cced__*`).

Three logical panes in a responsive 1-column / 2-column grid:

1. **Derive wallet** (left, wider column)
   - URI input + JSON import button (`fileImports.privateKey`).
   - Reactive readouts: detected ticker (+ SPCK curve hint),
     derived public key, derived wallet address, error message.
2. **Encrypt file** (top-right)
   - File picker, public-key textarea, `Pull public key`, `Encrypt`.
3. **Decrypt file** (bottom-right)
   - File picker, private-key textarea, `Pull private key`, `Decrypt`.

### Cryptography ladder (mirrors `From.sparkplate.storage/src/App.vue`)

```
encryptFile(trimmedKey)
├─ isHexCryptoPublicKey   → encryptWithCryptoPublicKey   (SPCK v1, secp256k1)
├─ isEd25519PublicKey     → encryptWithEd25519PublicKey  (SPCK v2, X25519)
├─ ticker-mismatch hint   → user-visible "use PGP" message (no encrypt)
└─ fallback               → encryptBinaryWithArmoredPublicKey (.gpg)

decryptFile(privateKey)
├─ tryDecryptSpckFile (binary "SPCK" magic + version byte detect)
│   ├─ v1 secp256k1     → decryptWithCryptoPrivateKey
│   └─ v2 X25519        → decryptWithEd25519PrivateKey
└─ fallback             → decryptArmoredMessageWithPrivateKey (armored OpenPGP)
```

`resolveEncryptedExtension` + `buildEncryptedFilename` keep the
`<basename>.<TICKER>.spck` naming convention identical to the reference and to
`card.WalletAddress.vue`.

## 4. Library imports — path remapping reference → fresh

The reference used flat module paths under `cryptographyCore/`. Sparkplate.fresh
keeps the modules in versioned subfolders (`encryption/`, `decryption/`,
`encryption.gpg/`). All imports were remapped accordingly and re-exported
through the `@/` alias.

| Concern | Reference `App.vue` | Sparkplate.fresh |
| --- | --- | --- |

| Private-key file import | `./lib/cores/importStandard/fileImports.privateKey` | `@/lib/cores/importStandard/fileImports.privateKey` |
| SPCK v1 (secp256k1) encrypt | `./lib/cores/cryptographyCore/encryption.crypto.PublicKey.general` | `@/lib/cores/cryptographyCore/encryption/encryption.crypto.PublicKey.general` |
| SPCK v2 (Ed25519/X25519) encrypt | `./lib/cores/cryptographyCore/encryption.crypto.PublicKey.ed25519` | `@/lib/cores/cryptographyCore/encryption/encryption.crypto.PublicKey.ed25519` |
| OpenPGP armored encrypt | `./lib/cores/cryptographyCore/encryption.openPGP.PublicKey.general` | `@/lib/cores/cryptographyCore/encryption/encryption.openPGP.PublicKey.general` |
| OpenPGP armored decrypt | `./lib/cores/cryptographyCore/decryption.openPGP.PublicKey.general` | `@/lib/cores/cryptographyCore/decryption/decryption.openPGP.PublicKey.general` |

## 5. Pre-existing broken imports surfaced and repaired

The new tab is the first consumer in `Sparkplate.fresh` that pulls in *both*
`tryDecryptSpckFile` and `resolveEncryptedExtension` from the existing core
libraries. That exercised three module paths that had been written as if the
folders were still flat (the reference layout) and had silently never been
imported elsewhere. They are now fixed:

| File | Was | Now |
| --- | --- | --- |
| `src/lib/cores/exportStandard/fileExports.Encrypted.ts` | `../cryptographyCore/encryption.crypto.PublicKey.general` | `../cryptographyCore/encryption/encryption.crypto.PublicKey.general` |
| `src/lib/cores/cryptographyCore/decryption/decryption.crypto.PublicKey.general.ts` | `../bridge/bridge.Cryptography.Via.Currency` | `../../bridge/bridge.Cryptography.Via.Currency` |
| `src/lib/cores/cryptographyCore/decryption/decryption.crypto.PublicKey.general.ts` | `./encryption.crypto.PublicKey.general` *(and `.ed25519`)* | `../encryption/encryption.crypto.PublicKey.general` *(and `.ed25519`)* |

After these fixes `npx vite build` completes cleanly (`✓ built in 2m 13s`).

## 6. `currencyCore/currencies` deltas pulled from reference

Comparing each ticker module against the same path in
`00.references/From.sparkplate.storage/` produced five differing files. Each
delta was applied because the reference version reflects a strictly newer
runtime API (or stricter TS hygiene) that the rest of the new tab transitively
depends on.

### 6.1 `@noble/ed25519` v3 hash-wiring (correctness fix)

`@noble/ed25519` v3 removed the writeable `etc.sha512Sync` slot (the `etc`
object is now frozen) and exposes a `hashes.sha512` property instead. Old
v2-style wiring throws at runtime:

```
Cannot add property sha512Sync, object is not extensible
```

Both `SOL.Solana.ts` and `XTZ.Tezos.ts` were updated:

- `SOL.Solana.ts`
  ```ts
  const ed25519Module = await import('@noble/ed25519');
  if (!ed25519Module.hashes.sha512) {
    ed25519Module.hashes.sha512 = sha512;
  }
  ```
- `XTZ.Tezos.ts` — same idiom applied at its ed25519 derivation point.

This matters for the new tab: the moment a user pastes `SOL://...` or
`XTZ://tz1...`, `useCryptocurrencyEngine` triggers exactly these derivations to
produce the SPCK-eligible public key.

### 6.2 `libsodium-wrappers-sumo` ESM `default`-export shim (XTZ)

`XTZ.Tezos.ts` now resolves the `default` export explicitly before calling any
`crypto_*` helper:

```ts
const sodiumMod = await import('libsodium-wrappers-sumo');
// eslint-disable-next-line @typescript-eslint/no-explicit-any
const sodium: any = (sodiumMod as any).default ?? sodiumMod;
```

The reference’s in-line comment + reference to its own
`docs/findings/20260511.findings.xtz.encrypt.libsodium.crypto.functions.on.default.export.md`
were preserved verbatim so future readers can trace why the shim exists. Vite's
strict-ESM dev server does not forward unknown property access from a module
namespace to `.default`, so without this shim only the formatter helpers
(`ready`, `to_hex`, …) work and `crypto_pwhash` is `undefined`.

### 6.3 `verbatimModuleSyntax`-style `type` imports

Three files received the explicit `type` keyword on type-only imports — a
no-op at runtime, but it removes the warning under TypeScript's
`verbatimModuleSyntax` and keeps the two trees byte-identical on the import
line shape:

- `currencies/BTC.Bitcoin.ts` — `type BitcoinAddressFormat`
- `currencies/LTC.Litecoin.ts` — `type LitecoinAddressFormat`
- `currencies/ext/multiFormatAddresses.tsx` — both of the above re-exports

### 6.4 What was *not* changed

The other 41 `currencies/*` files matched byte-for-byte and were left
untouched. `currencyByTicker` / `DerivedInfo` / `currencyData.ts` did not
change, so no consumer needed updating.

## 7. UX behaviour the new tab preserves

- **Pull public key** copies the URI-derived `cryptoPublicKey` straight into
  the Encrypt panel and labels its curve (`SPCK v1 / secp256k1`,
  `SPCK v2 / X25519`) or warns when the ticker is not SPCK-compatible.
- **Pull private key** parses `TICKER://…`, strips the scheme, and drops the
  raw key into the Decrypt panel — exactly as the reference does.
- **JSON import** (`Upload` button next to the URI label) uses
  `parsePrivateKeyFile` from `fileImports.privateKey`, which encodes the
  ticker from the file's `<owner>.<ticker>.<address>.json` segment and rebuilds
  the canonical URI.
- **Filename convention** for encrypted output:
  - Recognized ticker → `<basename>.<TICKER>.spck`
  - Bare hex/Ed25519 key → `<basename>.spck`
  - Armored PGP → `<basename>.gpg`
- Inactive tabs unmount (`v-if`), so dialog/tooltip portals from neighbouring
  tools never leak across.

- `npx vite build` — clean build, no errors, no new warnings introduced.
  (Large-chunk warnings for `libsodium-wrappers`, `stellar-base`, etc. are
  pre-existing.)
- Linting on the new SFC, the view file, and each edited TS file — no errors.

  `useCryptocurrencyEngine` is the same composable used elsewhere
  (`card.WalletAddress.vue`), the SPCK encrypt/decrypt routines remain the same
  files used by the existing wallet card, and the file-naming helper is the
  unique source of `.spck` extensions across the app.
