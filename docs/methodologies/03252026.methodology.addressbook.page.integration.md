# Address Book Page Integration Methodology

**Date**: March 25, 2026 (`03252026`)  
**Reference codebase**: `00.references/from.AddressbookUpdate/`  
**Target application**: Sparkplate.Fresh (Vue 3 + Vite + Electron)  
**Related methodology**: `10192025.methodology.sqlite3.database.implementation.md` (persistence alignment)

---

## Executive Summary

The reference project implements a self-contained **Address Book** feature: contact CRUD, wallet addresses per contact, multi-tab views (Contacts, Exchanges, Wallets, Companies), import from VCF/CSV/spreadsheets, export hooks, and rich modals. This document inventories that functionality, calls out inconsistencies in the reference, and proposes a **phased plan** to land an equivalent **Address Book** page in Sparkplate (`/addressbook`) with routing, navigation, styling aligned to the existing app (radix-vue / Tailwind patterns used elsewhere), and a **single coherent persistence strategy** suitable for Electron.

## 3. Technical gaps in the reference (fix during integration)

1. **Dual persistence**: `databaseService` (sql.js) vs `contactService`/`walletService` (localStorage JSON). **Pick one** for Sparkplate; for Electron, prefer **SQLite via main process** or a single renderer store with IPC, as in `10192025.methodology.sqlite3.database.implementation.md`.
2. **`addWallet` usage**: `walletService.addWallet` expects an object `{ contactId, coinTicker, address, … }`; some call sites may pass positional args—**normalize** during port.
3. **Export button**: Not implemented—plan real CSV/VCF (and optionally JSON) export.
4. **Static tabs**: Exchanges/Wallets need product decisions (persist vs mock).
5. **`ActionsDropdown` import path**: Uses a fragile path to `contactService`—fix to `@/services/...` or relative.

## 4. Proposed Sparkplate architecture

### 4.1 Route and view

- **Path**: `/addressbook`
- **View**: `src/views/AddressBook.vue` (thin wrapper: layout, title, passes props if needed)
- **Feature components**: `src/components/addressBook/` (or `src/components/pages/addressBook/`) to mirror the reference tree:
  - `AddressBookPage.vue` (port from `AddressBook.vue`)
  - Subfolders: `modals/`, `tabs/`, `structure/`, `import/` as in reference

### 4.2 Router

- Add route in `src/router/index.ts` alongside existing views.
- Lazy-load if bundle size is a concern: `() => import('../views/AddressBook.vue')`.

### 4.3 Navigation

- **SideNav.vue**: New `router-link` with icon (e.g. `BookUser` or `NotebookTabs` from `lucide-vue-next`) and `pathColor('/addressbook')`.
- **KeyboardShortcuts.vue**: Insert `addressbook` in `pageOrder` + `pathMap` (e.g. `/addressbook`).
- **menuTranslations** (`locales/*.ts`): Add label keys for “Address Book” / macro menu text.

### 4.4 Styling

- Replace ad hoc gray table CSS with **existing Sparkplate patterns**: radix-vue `Separator`, cards, spacing tokens consistent with `Settings` / `Techstack` views.
- Keep responsive table behavior (`overflow-x`, sticky header) as in reference.

### 4.5 Persistence (recommended direction)

| Option | Pros | Cons |
|--------|------|------|
| A. Renderer **localStorage** (port as-is) | Fastest | Not ideal for large data; no main-process backup |
| B. **SQLite** (Electron `userData`) | Durable, queryable, aligned with methodology doc | Requires IPC + migration work |
| C. **sql.js** in renderer, single key | Middle ground | Still large blobs in storage; backup story weaker than B |

**Recommendation:** Phase 1 port with **localStorage services** (minimal friction), then **Phase 2** migrate to SQLite + IPC matching `10192025.methodology.sqlite3.database.implementation.md`.

## 5. Implementation phases

### Phase 0 — Prerequisites

- [ ] Confirm/add npm dependencies: `papaparse`, `xlsx` (and types if needed) if not present.
- [ ] Decide persistence: Phase 1 localStorage vs immediate SQLite.

### Phase 1 — Shell + routing

- [ ] Add `src/views/AddressBook.vue` and feature components under `src/components/addressBook/`.
- [ ] Register `/addressbook` route; wire SideNav, keyboard shortcuts, i18n.
- [ ] Smoke test: empty state, navigation from other pages.

### Phase 2 — Core contacts + wallets

- [ ] Port `contactService`, `walletService`, `companyService` (or merge into `src/services/addressBook/`).
- [ ] Port `AddressBook.vue` → `AddressBookPage.vue`; fix `initDatabase` vs storage: **either** remove unused sql.js **or** use SQL for all CRUD.
- [ ] Port `addContact`, `ContactDetailsModal`, `AddCurrencyModal`, `ConfirmModal`, `WalletsTab`.

### Phase 3 — Import / export

- [ ] Port `useContactParser`, `ImportButton`, `ImportConfirmModal`, `WalletImportConfirmModal`.
- [ ] Wire `importWallet.json.ts` into add-contact flow or import dropdown.
- [ ] Implement **Export** (CSV minimum; VCF optional; JSON for wallets optional).

### Phase 4 — Secondary tabs + polish

- [ ] Port `ExchangeTab`, `WalletTab` (mock or real data).
- [ ] Port `CompaniesTab` + `companyService` delete cascade.
- [ ] Implement QR generation (PNG/SVG) using shared utilities.
- [ ] Wire `ActionsDropdown` exports and delete/edit flows end-to-end.

### Phase 5 — Hardening

- [ ] Electron: file paths, security (no arbitrary path writes without user consent).
- [ ] Large contact lists: virtualize table if needed.
- [ ] Accessibility: focus trap in modals, keyboard nav for tabs.
- [ ] Tests: unit tests for parsers + services; smoke E2E for add/import flow.

## 7. Acceptance criteria (v1)

- User can open **Address Book** from the side nav and keyboard shortcut.
- User can **add**, **edit**, **delete** contacts and **attach wallet addresses** per contact.
- User can **import** CSV/VCF/XLSX and **confirm** before merge.
- User can **search**, **sort**, **paginate**, and **bulk delete** contacts.
- **Companies** tab shows aggregated companies; delete behavior matches reference intent.
- **Export** produces at least one downloadable format (CSV).
- No duplicate conflicting persistence layers in production code path.
