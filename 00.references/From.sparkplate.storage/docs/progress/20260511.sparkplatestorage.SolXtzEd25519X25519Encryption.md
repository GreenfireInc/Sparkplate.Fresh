## Overview

Extended the application's native cryptocurrency file-encryption pipeline so it accepts **Solana (SOL)** and **Tezos (XTZ `tz1`)** public keys as encryption recipients. Both chains sign with **Ed25519** — a curve the prior SPCK / ECIES envelope (`v1`, secp256k1-only) does not handle. The implementation adds a **second envelope variant (SPCK `v2`)** that converts Ed25519 keys to their **X25519** equivalents via the standard Edwards→Montgomery birational map and runs an analogous ECIES (X25519 ECDH + HKDF-SHA256 + AES-256-GCM) over them.

The work landed in two stages:

| Stage | Scope | Status before this session | Status after this session |
|-------|-------|----------------------------|----------------------------|
| 1 — UI guard | Stop OpenPGP's "Misformed armored text" when a non-secp256k1 derived key is pasted into the encrypt panel. | Shipped — `App.vue` already detected `!isSpckCompatible` and surfaced a clear status message. | Unchanged behavior; status copy refined to mention both curves. |
| 2 — Real Ed25519 encryption | Make SOL and XTZ `tz1` actually encryptable / decryptable inside the app via SPCK v2 (X25519 ECIES). | Proposal only (see findings doc). | **Implemented, tested, type-clean.** |

## Problem Description

### Errors encountered

Pasting a SOL or XTZ derived public key into the **Encrypt File** panel and clicking **Encrypt** produced one of two failures, depending on which Stage was in place:

- **Pre-Stage 1**: OpenPGP fallback was reached and threw `Error: Misformed armored text` because a Base58 (SOL) or 64-char hex (XTZ tz1) Ed25519 pubkey is neither SEC1 hex nor PGP armor.
- **Stage 1**: The guard in `App.vue` caught the `!isSpckCompatible` case and showed:
  > `<TICKER> is not compatible with native SPCK/ECIES in this app …` plus advice to use PGP.

### Root cause

The native encryption module **`encryption.cryptoPublicKey.ts`** implements ECIES over **secp256k1** only. The recipient predicate `isHexCryptoPublicKey()` requires SEC1 hex (`02|03 + 64 hex` compressed, or `04 + 128 hex` uncompressed). Ed25519 chains expose their public keys in completely different shapes:

| Chain | Derived pubkey format | SPCK v1 (secp256k1) eligible? |
|-------|------------------------|--------------------------------|
| SOL   | Base58 (32-byte raw Ed25519 point) | No |
| XTZ `tz1` | hex (32-byte raw Ed25519 point, 64 chars) | No |

Both can be used as encryption recipients only by switching to a curve that *is* an ECDH curve. Ed25519 itself can be safely converted to **X25519** (Curve25519's Montgomery form) using one-line helpers in `libsodium-wrappers-sumo`, after which the rest of an ECIES pipeline is straightforward.

## Solution Implemented (Stage 2)

### Curve & envelope design

Picked **X25519 ECDH + HKDF-SHA256 + AES-256-GCM**, mirroring the structure of SPCK v1 line-for-line:

```
encrypt(recipientEdPub, plaintext):
  xPub      = libsodium.crypto_sign_ed25519_pk_to_curve25519(recipientEdPub)
  eph       = x25519.keygen()                              // @noble/curves
  shared    = x25519.getSharedSecret(eph.secretKey, xPub)  // 32 B
  aesKey    = HKDF-SHA256(shared, salt=eph.publicKey,
                          info="sparkplate-x25519-v1", 32)
  iv        = randomBytes(12)
  ct, tag   = WebCrypto subtle.encrypt AES-GCM(aesKey, iv, plaintext)
  envelope  = "SPCK" || 0x02 || eph.publicKey (32 B)
                     || iv (12 B) || ct || tag
```

| Bytes | v1 (existing, secp256k1) | v2 (new, X25519) |
|-------|---------------------------|------------------|
| 0–3   | `"SPCK"` magic            | `"SPCK"` magic   |
| 4     | `0x01`                    | `0x02`           |
| 5–37  | 33 B compressed secp256k1 ephemeral pub | 32 B X25519 ephemeral pub (raw) |
| next 12 | AES-GCM IV              | AES-GCM IV       |
| rest  | AES-GCM(ciphertext ‖ tag) | AES-GCM(ciphertext ‖ tag) |

KDF `info` strings differ (`"sparkplate-ecies-v1"` vs. `"sparkplate-x25519-v1"`) so a shared point cannot be reused across formats.

### Code changes

#### New module — `src/lib/cores/cryptographyCore/encryption.ed25519PublicKey.ts`

```ts
export const SPCK_VERSION_X25519 = 0x02;
export const X25519_PUBKEY_LENGTH = 32;
export const ED25519_SEED_LENGTH = 32;

export const isBase58Ed25519PublicKey: (value: string) => boolean;
export const isHexEd25519PublicKey:    (value: string) => boolean;
export const isEd25519PublicKey:       (value: string) => boolean;

export const normalizeEd25519PublicKey: (value: string) => Uint8Array;     // 32 B
export const normalizeEd25519Seed: (value: string | Uint8Array) => Uint8Array; // 32 B

export const ed25519PubToCurve25519:    (edPub: Uint8Array) => Promise<Uint8Array>;
export const ed25519SeedToCurve25519Priv: (seed: Uint8Array) => Promise<Uint8Array>;

export const hasSpckX25519Header: (bytes: Uint8Array) => boolean;

export const encryptWithEd25519PublicKey: (
  recipientPublicKey: string | Uint8Array,
  plaintext: Uint8Array,
) => Promise<{ envelope: Uint8Array; ephemeralX25519PublicKey: Uint8Array; iv: Uint8Array }>;

export const decryptWithEd25519PrivateKey: (
  recipientPrivateKey: string | Uint8Array,
  envelope: Uint8Array,
) => Promise<Uint8Array>;
```

`normalizeEd25519Seed` recognizes all input shapes the project already accepts elsewhere:

| Input | Decoded length | Seed used |
|-------|----------------|-----------|
| Tezos `edsk…` (Base58check, standard) | 40 | bytes `[4, 36)` |
| Tezos `edsk…` extended variant | 68 / 72 | bytes `[4, 36)` |
| Solana Base58 (32-byte secret) | 32 | as-is |
| Solana Base58 (64-byte `seed ‖ pubkey` keypair) | 64 | `[0, 32)` |
| Solana JSON-array keystore | 32 / 64 | `[0, 32)` |
| Raw hex (64 chars) | 32 | as-is |
| Raw hex (128 chars) | 64 | `[0, 32)` |

#### Bridge — `src/lib/cores/bridge/bridge.Cryptography.Via.Currency.ts`

Added a curve discriminant so the UI can dispatch unambiguously without re-detecting from strings:

```ts
export type SpckCurve = 'secp256k1' | 'x25519';

export interface CryptocurrencyDerivation {
  // … existing fields …
  spckCurve: SpckCurve | null;
  isSpckCompatible: boolean; // true iff spckCurve !== null
}

export const resolveSpckCurve: (publicKey: string) => Promise<SpckCurve | null>;
```

Resolution rules:

- SEC1 hex (`02|03 + 64 hex` / `04 + 128 hex`) → `'secp256k1'`
- Raw 32-byte hex (64 chars, no SEC1 prefix) → `'x25519'`
- Base58 decoding to exactly 32 bytes → `'x25519'`
- Anything else → `null`

#### Dispatcher — `src/App.vue`

Encrypt path replaced the single `isHexCryptoPublicKey` check with a curve-aware ladder:

```ts
if (isHexCryptoPublicKey(trimmedKey)) {
  // SPCK v1 / secp256k1
} else if (isEd25519PublicKey(trimmedKey)) {
  const { envelope } = await encryptWithEd25519PublicKey(trimmedKey, fileData);
  // SPCK v2 / X25519
} else if (cryptoTickerSymbol.value && !isSpckCompatible.value && …) {
  // truly incompatible curve — show explanation, no OpenPGP
} else {
  // PGP fallback (only for actual armored input)
}
```

Decrypt path branches on the envelope's **version byte** rather than the key string, so v1 and v2 envelopes are handled by the right primitive automatically.

Status copy refined throughout to identify which envelope/curve was used (e.g. "*File encrypted with SOL public key (SPCK v2 / X25519 ECIES)!*").

## Future Work (out of scope for this report)

- **sr25519 / Ristretto255** (Polkadot, DOT) — would need a `0x03` envelope version and a separate curve adapter (no birational map to X25519).
- **Tezos `tz2` / `tz3`** — `tz2` is secp256k1 and could plug into v1 once derivation exposes SEC1 hex; `tz3` is P-256 and requires a `0x04`-style version.
- **Hardware-wallet decryption** of v2 envelopes — most devices don't expose `ed25519 → x25519` ECDH today.
- **Replace the redundant `Buffer.from(uint8Array)` wrappers** in `BTC.Bitcoin.P2TR.ts` and the LTC SegWit modules (cosmetic; the polyfill landed earlier today makes them functionally unnecessary).
