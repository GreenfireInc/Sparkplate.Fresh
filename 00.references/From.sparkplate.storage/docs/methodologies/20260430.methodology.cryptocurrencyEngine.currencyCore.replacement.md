# Methodology: `cryptocurrencyEngine.ts` vs `currencyCore/currencies` and a Path to Replace the Engine

**Date:** April 30, 2026  
**Project:** Sparkplate Storage  
**Primary sources:**  
`src/lib/cores/engine/cryptocurrencyEngine.ts`  
`src/lib/cores/currencyCore/currencies/` (notably `index.ts`, `currencyData.ts`, per-ticker `*.ts` modules)

---

## 1. Executive Summary

Today, **wallet derivation for the Native Cryptocurrency tab** is implemented in a **small, Vue-coupled engine** (`cryptocurrencyEngine.ts`) with a fixed list of six tickers and hand-written derivation rules.

Separately, **Sparkplate already maintains a large, data-driven currency registry** under `currencyCore/currencies`: each asset exposes optional `deriveFromPrivateKey`, rich metadata, and lookups like `currencyByTicker`.

**`currencyCore/currencies` can replace `cryptocurrencyEngine.ts`** for the derivation responsibilities that the UI needs (URI parse → ticker + private key → public key + display address), *provided* you add a thin adapter that:

- Parses the same `TICKER://privateKey` URI the UI uses today.
- Resolves the ticker through `currencyByTicker` (or `PROTOCOL_PREFIXES` / `NETWORKS`).
- Calls the asset’s `deriveFromPrivateKey` when present.
- Normalizes `DerivedInfo` into the three strings the app currently expects (`ticker`, `publicKey`, `walletAddress`).
- Decides which tickers are allowed in this screen if you still tie file encryption to **secp256k1 SEC1** public keys (see §7).

The **Vue composable** (`useCryptocurrencyEngine`) can stay as a thin wrapper around the new adapter, or move next to the adapter in a small module that imports `currencyCore` but does not duplicate derivation math.

## 2. What Each Module Is Responsible For

### 2.1 `cryptocurrencyEngine.ts` (current)

| Aspect | Implementation |
|--------|----------------|
| **Role** | Parse `%ticker%://%privateKey%`, derive **public key** and **one wallet address**, surface errors. |
| **Vue** | Exports `useCryptocurrencyEngine(input: Ref<string>)` with `watch(input, recompute)`. |
| **Derivation API** | Sync function `deriveCryptocurrency(input: string): DerivationResult`. |
| **Supported tickers** | `SUPPORTED_TICKERS`: `BTC`, `ETH`, `LTC`, `DOGE`, `BCH`, `DASH`. |
| **ETH path** | `ethers.Wallet` → `wallet.signingKey.publicKey`, `wallet.address`. |
| **Bitcoin-like path** | `@noble/secp256k1` compressed pubkey → `0x` + hex; P2PKH-style address via `SHA-256` + `RIPEMD-160` + `bs58check` with per-coin version bytes (`P2PKH_VERSIONS`). |
| **Consumers** | `src/App.vue` only (`useCryptocurrencyEngine(cryptoInput)`). |

The engine does **not** know about keystores, RPC endpoints, or multi-format Bitcoin/Litecoin routing. It encodes one opinionated address per UTXO coin (classic P2PKH version byte only).

### 2.2 `currencyCore/currencies` (registry)

| Aspect | Implementation |
|--------|----------------|
| **Role** | Per-currency **catalog**: `CurrencyData` with `basicInfo`, `technicalInfo`, explorers, optional `deriveFromPrivateKey`, keystores, import/export, etc. |
| **Entry points** | `currencyByTicker`, `allCurrencies`, `NETWORKS`, `PROTOCOL_PREFIXES` in `currencies/index.ts`. |
| **Derivation API** | Optional `deriveFromPrivateKey(privateKey, ...options) => Promise<DerivedInfo>` on each record (`currencyData.ts`). |
| **Derived shape** | `DerivedInfo`: `publicKey`, `address`, plus optional multi-format fields (`p2pkhAddress`, `p2wpkhAddress`, `formattedDisplay`, …). |
| **Bitcoin / Litecoin** | Routers can return **multiple address formats** (e.g. BTC `P2PKH` / `P2SH` / `P2WPKH` / `P2TR`) while picking a **primary** `address` for backward compatibility. |
| **Breadth** | Many more tickers than the engine (see `SupportedTicker` union in `currencies/index.ts`). **DASH** appears in the engine’s `SUPPORTED_TICKERS` but **does not** appear in `currencyByTicker` as of this writing — a parity gap if you switch without adding a `DASH` currency module. |

`currencyCore` as a whole also contains exchanges, web3Connect, walletProviders, etc.; **this document focuses on `currencies/`** as the replacement for derivation, because that is what mirrors the engine’s job.

## 3. Side-by-Side Comparison

| Topic | `cryptocurrencyEngine.ts` | `currencyCore/currencies` |
|-------|---------------------------|----------------------------|
| **Coupling to Vue** | Yes (`ref`, `watch`, `Ref`) | No — plain data + async functions |
| **`deriveFromPrivateKey`** | Implicit inside `deriveCryptocurrency` | Explicit per `CurrencyData` |
| **Async** | Sync derivation | Typically **async** (`Promise<DerivedInfo>`) |
| **Ticker list** | Fixed array of 6 | `currencyByTicker` + `NETWORKS` (larger set) |
| **URI parsing** | Built in (`/^([A-Za-z0-9]+):\/\/(.+)$/`) | Not centralized — you add one parser in an adapter |
| **BTC address policy** | Single P2PKH-style base58 address | Configurable formats; default primary often P2PKH-like but implementation is in `BTC.Bitcoin.ts` router |
| **ETH public key shape** | `ethers` signing key string | Noble secp256k1 **uncompressed** hex via `toHex(pub)` in `ETH.Ethereum.ts` (differs from engine — see §6) |
| **Keystore / import** | Not supported | Many assets implement `decryptKeystore` / `importWallet` (optional future UI) |
| **Reusability** | Tied to this screen | Intended as the shared “currency brain” for the app |

## 4. How `currencies/` Can Replace the Engine (Architecture)

### 4.1 Target behavior to preserve

`App.vue` needs, from `cryptoInput`:

- **`cryptoTickerSymbol`** — resolved ticker string for labels.
- **`cryptoPublicKey`** — string that is either accepted by `isHexCryptoPublicKey` for **Encrypt file**, or must be normalized to that form (secp256k1 SEC1 hex).
- **`cryptoWalletAddress`** — human-readable address for display / explorer links.
- **`cryptoErrorMessage`** — user-visible errors for bad URI, unsupported ticker, or derivation failure.

None of that *requires* the logic to live in `cryptocurrencyEngine.ts`; it only requires a stable contract.

### 4.2 Replacement pattern: adapter + optional thin composable

```text
cryptoInput (Ref<string>)
       │
       ▼
┌──────────────────────────────────────┐
│  parseUri(input) -> { ticker, sk }   │  // same regex / rules as today
└──────────────────────────────────────┘
       │
       ▼
┌──────────────────────────────────────┐
│  currency = currencyByTicker[ticker] │
└──────────────────────────────────────┘
       │
       ▼
┌──────────────────────────────────────┐
│  derived = await currency             │
│    .deriveFromPrivateKey?.(sk, opts) │
└──────────────────────────────────────┘
       │
       ▼
┌──────────────────────────────────────┐
│  mapDerivedToUi(derived, ticker)     │  // normalize publicKey + address
└──────────────────────────────────────┘
```

- **`parseUri`**: Lift the regex and empty-input behavior from `deriveCryptocurrency` in the engine, or reuse `PROTOCOL_PREFIXES` to validate that `ticker` is a known network.
- **Lookup**: `currencyByTicker[tickerUpper]` (with explicit “not found” error matching today’s `Unsupported ticker` message).
- **Call**: Guard `if (!currency.deriveFromPrivateKey)` → clear error.
- **Map**: Set `walletAddress` from `derived.address` (or a chosen field for multi-format coins — see §5).

### 4.3 Vue integration

`useCryptocurrencyEngine` today runs **synchronously** on every `input` change. After the switch:

- Use `watch` with an **async** callback, or `watch` + `void (async () => { ... })()` with a simple generation counter to ignore stale results.
- Keep the same returned ref names so `App.vue` changes are minimal.

## 5. Address and Public-Key Semantics (Multi-Format Coins)

For **BTC** and **LTC**, `currencyCore` may return several addresses (`p2pkhAddress`, `p2wpkhAddress`, etc.) and a **primary** `address`.

The engine today always produced a **single** legacy-style address for UTXO coins. When migrating:

- **Display address (`cryptoWalletAddress`)**: Recommend using **`derived.address`** (the module’s declared primary) for consistency with `currencyCore` docs, *or* pick a fixed format (e.g. native SegWit) if product requirements differ — but then document that choice.
- **Pull public key for encryption**: Must still produce a string that satisfies `isHexCryptoPublicKey` in `encryption.cryptoPublicKey.ts` (compressed `02|03…` or uncompressed `04…`, optional `0x`). If any `deriveFromPrivateKey` returns hex **without** `0x`, prepend `0x` when needed.

## 6. ETH Public Key: Engine vs `ethereumData` (Important)

The current engine uses **`ethers.Wallet`’s `signingKey.publicKey`** for ETH.

`ethereumData.deriveFromPrivateKey` builds the public key with **`@noble/secp256k1` in uncompressed form** and `toHex(pub)`:

```478:489:src/lib/cores/currencyCore/currencies/ETH.Ethereum.ts
  deriveFromPrivateKey: async (privateKey: string): Promise<DerivedInfo> => {
    // Import dependencies when needed to avoid loading them if not used
    const secp = await import('@noble/secp256k1');
    
    const wallet = new Wallet(privateKey.startsWith("0x") ? privateKey : `0x${privateKey}`);
    // wallet.publicKey is not typed in ethers v6 typings; compute from private key via noble
    const sk = Uint8Array.from((wallet.privateKey.slice(2)).match(/.{1,2}/g)!.map((b) => parseInt(b, 16)));
    const pub = secp.getPublicKey(sk, false);
    const addressString: string = String(wallet.address);
    
    return { publicKey: toHex(pub), address: addressString };
  }
```

Both are valid secp256k1 points; **hex strings may differ** (format, casing, compressed vs uncompressed). For **file encryption**, `isHexCryptoPublicKey` accepts uncompressed `04…` **and** compressed `02/03…`. After migration, regression-test **encrypt → decrypt** for ETH with the same private key to ensure the UI-fed public key still matches what `encryptWithCryptoPublicKey` expects.

## 7. SPCK / ECIES Constraint vs “Many Tickers”

File encryption in `App.vue` only uses the cryptocurrency path when the pasted key passes **`isHexCryptoPublicKey`** — that is **secp256k1 SEC1**, not ed25519, not sr25519, etc.

```48:51:src/lib/cores/cryptographyCore/encryption.cryptoPublicKey.ts
export const isHexCryptoPublicKey = (value: string): boolean => {
  const h = stripHexPrefix(value);
  return /^(02|03)[0-9a-fA-F]{64}$/.test(h) || /^04[0-9a-fA-F]{128}$/.test(h);
};
```

`currencyCore` includes assets whose `deriveFromPrivateKey` is **not** based on a raw 32-byte secp256k1 secret in the same way (e.g. Polkadot’s sr25519/ed25519 routing, Solana ed25519-style handling).

**Implication:** Expanding `NETWORKS` in the URI field does **not** automatically mean every ticker is safe to advertise with “32-byte hex private key” copy or with **SPCK** encryption. A replacement engine should either:

- **Gate** the Encrypt button / hints to a **documented allowlist** (secp256k1-friendly tickers), or  
- Derive and display addresses for all tickers, but **disable** crypto-pubkey file encryption when the displayed public key is not SEC1 secp256k1.

This is a product + safety decision, not just a refactor.

## 8. Parity Gaps to Close Before Deleting the Engine

| Gap | Notes |
|-----|--------|
| **DASH** | In engine `SUPPORTED_TICKERS`; no `currencyByTicker.DASH` in `currencies/index.ts` today. Add a `DASH` currency module or keep a tiny special-case. |
| **Supported list in UI** | `App.vue` imports `SUPPORTED_TICKERS` from the engine for “Supported: …” hint text. After migration, source that list from `NETWORKS`, a filtered allowlist, or `Object.keys(currencyByTicker)`. |
| **`SUPPORTED_TICKERS` type** | Engine exports `SupportedTicker` union; `currencies/index.ts` exports a different `SupportedTicker`. Consolidate naming to avoid confusion (e.g. `CurrencyRegistryTicker`). |
| **Sync → async** | All call sites of derivation must tolerate `Promise`; avoid assuming instant updates in unrelated code. |
| **Tests** | Add golden-vector tests per ticker you claim to support for the tab (URI → address + SEC1 pubkey). |

---

## 9. Suggested Migration Phases

1. **Introduce** `deriveCryptocurrencyFromRegistry(input: string)` (name as you prefer) in e.g. `src/lib/cores/engine/` or `currencyCore/` that implements §4.2 and maps to the existing `DerivationResult` shape from the engine file.
2. **Unit test** the adapter against current engine outputs for BTC/ETH/LTC/DOGE/BCH (and DASH once parity exists).
3. **Switch** `useCryptocurrencyEngine` to call the adapter internally (or replace imports in `App.vue` with a renamed composable).
4. **Remove** duplicated `hexToBytes` / P2PKH logic from `cryptocurrencyEngine.ts` once unused, then delete the file if nothing else imports it.
5. **Revisit** UI copy (“32-byte hex private key”) for any newly enabled tickers per §7.
