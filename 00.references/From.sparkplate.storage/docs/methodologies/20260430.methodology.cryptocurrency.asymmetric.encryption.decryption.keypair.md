# Methodology: Cryptocurrency Asymmetric Encryption and Decryption Using a privateKey/publicKey Keypair

**Date:** April 30, 2026
**Project:** Sparkplate Storage
**Target:** `src/lib/cores/cryptographyCore/encryption.cryptoPublicKey.ts`
**Consumer:** `src/App.vue`
**Related engine:** `src/lib/cores/engine/cryptocurrencyEngine.ts`

---

## Overview

This methodology documents how Sparkplate Storage performs **asymmetric encryption and decryption** using a cryptocurrency `privateKey` / `publicKey` keypair (the same secp256k1 keypair that controls a wallet on Bitcoin, Ethereum, Litecoin, Dogecoin, Bitcoin Cash, or Dash).

The scheme is **ECIES** (Elliptic Curve Integrated Encryption Scheme) over secp256k1, using:

- An ephemeral secp256k1 keypair generated per message (forward-secrecy hygiene).
- ECDH key agreement between the ephemeral keypair and the recipient's wallet keypair.
- HKDF-SHA256 to derive an AES-256 key from the ECDH shared secret.
- AES-256-GCM as the bulk cipher (authenticated encryption).
- A self-describing binary envelope tagged with the magic header `SPCK`.

The end-user mental model is identical to PGP: **encrypt with the recipient's public key, decrypt with the matching private key**. The internal protocol differs because the keys are wallet keys, not PGP keys.

## 2. Terms and Definitions

This section defines every term used in the rest of the document. Definitions are scoped to their meaning **in this codebase**, not generic textbook usage.

### 2.1 General Cryptography Terms

| Term                              | Definition                                                                                                                                                                                                                                          |
| --------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Symmetric encryption**          | Encryption where the same secret key encrypts and decrypts. Fast, but requires the parties to share a secret out of band.                                                                                                                            |
| **Asymmetric encryption**         | Encryption where one key (the **public key**) is used to encrypt and a different, mathematically related key (the **private key**) is used to decrypt. Also called **public-key encryption**.                                                       |
| **Hybrid encryption**             | Combining the two: an asymmetric step establishes a one-time symmetric key, which then encrypts the actual payload. Both PGP and the SPCK / ECIES path are hybrid.                                                                                  |
| **Authenticated encryption (AEAD)** | A cipher that simultaneously provides confidentiality **and** integrity. AES-256-GCM (used here) is AEAD: a 16-byte authentication tag is appended to the ciphertext and verified during decryption. Tampered ciphertext fails to decrypt.        |
| **KDF** (Key Derivation Function) | A deterministic function that turns one secret value into one or more cryptographically independent keys. Required because raw ECDH output is not directly suitable as an AES key.                                                                  |
| **HKDF**                          | *HMAC-based Extract-and-Expand KDF*, RFC 5869. Takes `(IKM, salt, info, length)` and produces `length` pseudorandom bytes. This codebase uses HKDF-SHA256 with `info = "sparkplate-ecies-v1"`.                                                       |
| **IKM** (Input Keying Material)    | The "raw" secret fed into a KDF. Here it's the X coordinate of the ECDH shared point (`sharedX`).                                                                                                                                                   |
| **IV / Nonce**                    | A per-message non-secret value that ensures the same plaintext encrypts to different ciphertext under the same key. AES-GCM uses a 12-byte IV. Reusing an IV with the same AES key is catastrophic.                                                 |
| **Auth tag**                      | The 16-byte MAC produced by AES-GCM. Bundled at the end of the ciphertext by Web Crypto. Verified during decryption; mismatch causes `subtle.decrypt` to throw.                                                                                     |
| **AES-256-GCM**                   | The bulk cipher used to encrypt the file payload. 256-bit key, 12-byte IV, 16-byte tag, AEAD construction. Provided by the platform's Web Crypto (`crypto.subtle`).                                                                                  |
| **Forward secrecy**               | A property where compromising a long-term key does **not** compromise past session keys. Per-message ephemeral keypairs improve this, but ECIES is still **not** post-compromise secure if `recipientPriv` itself leaks (see §12.5).                |
| **Replay attack**                 | Re-sending a captured valid ciphertext. AES-GCM does not protect against replay on its own; mitigations live above the envelope (e.g. timestamps inside plaintext).                                                                                 |
| **Constant-time code**            | Code whose execution time does not depend on secret inputs, defending against timing side channels. `@noble/secp256k1` and platform AES-GCM are constant-time on all supported runtimes.                                                            |

### 2.2 Elliptic Curve / secp256k1 Terms

| Term                                     | Definition                                                                                                                                                                                                                                  |
| ---------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **secp256k1**                            | The elliptic curve standardized by SEC 2 and used by Bitcoin, Ethereum, Litecoin, Dogecoin, Bitcoin Cash, and Dash. Defined by parameters `(p, a, b, G, n, h)` over a 256-bit prime field.                                                  |
| **Curve point**                          | A pair `(x, y)` satisfying the curve equation. Points form a mathematical group under "point addition".                                                                                                                                     |
| **Generator point** `G`                  | A fixed, public, agreed-upon curve point. Every public key is `d · G` for some private scalar `d`.                                                                                                                                          |
| **Curve order** `n`                      | The number of distinct points reachable as integer multiples of `G`. Private keys must lie in `[1, n-1]`. Validated here by `secp.utils.isValidSecretKey`.                                                                                  |
| **Scalar multiplication**                | `k · P` = adding the curve point `P` to itself `k` times. The asymmetric primitive: easy to compute forwards, infeasible to invert (the *Elliptic Curve Discrete Log Problem*).                                                              |
| **Private key (`d`)**                    | A 32-byte scalar in `[1, n-1]`. Encoded as 64 hex characters (with optional `0x` prefix).                                                                                                                                                    |
| **Public key (`Q`)**                     | The curve point `Q = d · G`. Encoded per SEC1 as compressed (33 bytes) or uncompressed (65 bytes).                                                                                                                                          |
| **Keypair**                              | The tuple `(d, Q)`. Two public keys derived from the same private key are identical; this is what lets the UI render the public key automatically once a private key is pasted.                                                              |
| **SEC1 compressed encoding**             | `0x02 ‖ x` if `y` is even, `0x03 ‖ x` if `y` is odd. 33 bytes total. Default form everywhere in this codebase. Recoverable to `(x, y)` because the curve equation has at most two `y` values per `x`.                                       |
| **SEC1 uncompressed encoding**           | `0x04 ‖ x ‖ y`. 65 bytes total. Accepted as input by `encryptWithCryptoPublicKey` but never produced by it.                                                                                                                                  |
| **ECDH (Elliptic Curve Diffie–Hellman)** | Key-agreement primitive: given keypair `(a, A)` and counterparty's public key `B`, computes `a · B`. By scalar-multiplication commutativity, the counterparty independently computes `b · A` and gets the same point. See §8 for the proof. |
| **Shared secret point**                  | The curve point produced by ECDH (`a · B`). 33 bytes when SEC1-compressed.                                                                                                                                                                  |
| **`sharedX`**                            | The 32-byte X coordinate of the shared secret point, with the SEC1 prefix byte stripped. ECIES specifies X-only as the input to the KDF.                                                                                                    |
| **Ephemeral keypair**                    | A throwaway secp256k1 keypair `(eph_d, eph_Q)` generated fresh per encryption. The private half is destroyed after use; the public half is shipped in the envelope so the recipient can re-derive the same shared secret.                  |
| **Wallet keypair**                       | The user's *long-term* secp256k1 keypair — the same one that controls funds at a Bitcoin/Ethereum/etc. address. Reused across messages. Conceptually plays the same role as a PGP keypair here.                                              |

## 3. Conceptual Model

| Role      | Knows               | Action                                                                                    |
| --------- | ------------------- | ----------------------------------------------------------------------------------------- |
| Sender    | `recipientPub`      | Calls `encryptWithCryptoPublicKey(recipientPub, plaintext)` and ships the envelope.        |
| Recipient | `recipientPriv`     | Calls `decryptWithCryptoPrivateKey(recipientPriv, envelope)` and recovers the plaintext.   |
| Network   | The envelope only   | Cannot derive plaintext without `recipientPriv` (assuming secp256k1 hardness + AES-GCM).  |

The recipient's private key never crosses the wire and never leaves the recipient's machine. This is exactly the invariant PGP provides — only the underlying primitives differ.

## 4. The Keypair: secp256k1 Fundamentals

The "wallet keypair" is a standard secp256k1 keypair:

- **Private key:** A 32-byte (256-bit) integer `d ∈ [1, n-1]` where `n` is the curve order.
- **Public key:** The curve point `Q = d · G`, where `G` is the secp256k1 generator. SEC1-encoded as either:
  - **Compressed (33 B):** `0x02 || x` if `y` is even, `0x03 || x` if `y` is odd.
  - **Uncompressed (65 B):** `0x04 || x || y`.

`isHexCryptoPublicKey()` accepts both forms (with optional `0x` prefix):

```48:51:src/lib/cores/cryptographyCore/encryption.cryptoPublicKey.ts

  const h = stripHexPrefix(value);
  return /^(02|03)[0-9a-fA-F]{64}$/.test(h) || /^04[0-9a-fA-F]{128}$/.test(h);
};
```

`cryptocurrencyEngine.ts` derives the compressed public key from a hex private key for BTC-family tickers:

```112:131:src/lib/cores/engine/cryptocurrencyEngine.ts

```

Ethereum uses the same curve but a different address-derivation rule (Keccak-256 of the uncompressed pubkey, last 20 bytes); ECIES works identically for ETH because it consumes the **raw curve point**, not the address.

> **Implication.** Any keypair the engine accepts can be used as an ECIES recipient. The address format is irrelevant to the encryption protocol.

## 5. The SPCK Envelope Format

Every encrypted file produced by `encryptWithCryptoPublicKey` is a **single self-describing binary blob**:

```
Offset  Length  Field
------  ------  ----------------------------------------------------------
0       4       Magic        : ASCII "SPCK" (0x53 0x50 0x43 0x4B)
4       1       Version      : 0x01
5       33      ephemeralPub : SEC1-compressed secp256k1 point
38      12      iv           : Random AES-GCM nonce
50      N       ciphertext   : AES-256-GCM(ciphertext || 16-byte auth tag)
```

Total header length is `HEADER_LENGTH = 4 + 1 + 33 + 12 = 50` bytes.

```15:24:src/lib/cores/cryptographyCore/encryption.cryptoPublicKey.ts
export const SPCK_MAGIC = new Uint8Array([0x53, 0x50, 0x43, 0x4b]);
export const SPCK_VERSION = 0x01;

/** File extension recommended for ECIES-encrypted output. */
export const CRYPTO_PUBKEY_FILE_EXTENSION = 'spck';

const HKDF_INFO = new TextEncoder().encode('sparkplate-ecies-v1');
const IV_LENGTH = 12;
const COMPRESSED_PUBKEY_LENGTH = 33;
const HEADER_LENGTH = SPCK_MAGIC.length + 1 + COMPRESSED_PUBKEY_LENGTH + IV_LENGTH;
```

Properties of this layout:

- **Self-identifying.** A consumer can tell SPCK apart from PGP / arbitrary binary by inspecting the first 4 bytes.
- **Self-contained.** Everything needed to decrypt is in the envelope, except the recipient's private key.
- **Version-tagged.** The `0x01` byte allows future format evolution without ambiguity.
- **No recipient identity leaked.** The recipient's public key is **not** stored in the envelope. Only the per-message ephemeral public key is. This makes SPCK envelopes anonymous-recipient by default (you cannot tell who can decrypt them just by looking).

## 6. Encryption Pipeline

Implemented in `encryptWithCryptoPublicKey()`:

```107:179:src/lib/cores/cryptographyCore/encryption.cryptoPublicKey.ts
export const encryptWithCryptoPublicKey = async (
  recipientPublicKey: string | Uint8Array,
  plaintext: Uint8Array,
): Promise<CryptoPublicKeyEncryptionResult> => {
  const subtle = requireSubtleCrypto();
  const cryptoApi = requireGetRandomValues();

  const recipientBytes =
    typeof recipientPublicKey === 'string'
      ? hexToBytes(recipientPublicKey)
      : recipientPublicKey;

  if (
    recipientBytes.length !== 33 &&
    recipientBytes.length !== 65
  ) {
    throw new Error(
      'Recipient public key must be a SEC1-encoded secp256k1 key (33 bytes compressed or 65 bytes uncompressed)',
    );
  }
  if (!secp.utils.isValidPublicKey(recipientBytes, recipientBytes.length === 33)) {
    throw new Error('Recipient public key is not a valid secp256k1 point');
  }

  const ephemeralPrivateKey = secp.utils.randomSecretKey();
  const ephemeralPublicKey: Uint8Array<ArrayBuffer> = Uint8Array.from(
    secp.getPublicKey(ephemeralPrivateKey, true),
  );

  // ECDH returns a SEC1-encoded point; drop the 1-byte prefix to get the
  // 32-byte X coordinate, which is what ECIES uses as input keying material.
  const sharedPoint = Uint8Array.from(
    secp.getSharedSecret(ephemeralPrivateKey, recipientBytes, true),
  );
  const sharedX = sharedPoint.slice(1);

  // Mix the ephemeral public key into the salt so each message produces a
  // distinct AES key even with deterministic plaintext.
  const aesKeyBytes = hkdf(sha256, sharedX, ephemeralPublicKey, HKDF_INFO, 32);

  const iv = new Uint8Array(IV_LENGTH);
  cryptoApi.getRandomValues(iv);

  const aesKey = await subtle.importKey(
    'raw',
    Uint8Array.from(aesKeyBytes),
    { name: 'AES-GCM', length: 256 },
    false,
    ['encrypt'],
  );

  const ciphertextBuffer = await subtle.encrypt(
    { name: 'AES-GCM', iv },
    aesKey,
    plaintext as BufferSource,
  );
  const ciphertext = new Uint8Array(ciphertextBuffer);

  const versionByte = new Uint8Array([SPCK_VERSION]);
  const envelope = concatBytes(
    SPCK_MAGIC,
    versionByte,
    ephemeralPublicKey,
    iv,
    ciphertext,
  );
  ...
```

### Step-by-step

1. **Validate recipient public key.**
   - Must be 33 (compressed) or 65 (uncompressed) bytes.
   - Must lie on the secp256k1 curve (`isValidPublicKey`).
2. **Generate an ephemeral keypair.**
   - `eph_d ← random secret in [1, n-1]`
   - `eph_Q = eph_d · G`, encoded as 33-byte compressed.
   - Fresh per message; never persisted.
3. **ECDH agreement.**
   - `sharedPoint = eph_d · recipient_Q` (a curve point).
   - `sharedX = sharedPoint.x` (32-byte X coordinate). The Y coordinate prefix byte is dropped — ECIES specifies using only X.
4. **Key derivation (HKDF-SHA256).**
   - `aesKey = HKDF(IKM=sharedX, salt=ephemeralPub, info="sparkplate-ecies-v1", length=32)`.
   - The ephemeral public key is mixed into the salt so identical plaintexts encrypted with identical recipient keys still produce different AES keys.
5. **Random IV.**
   - 12 bytes from `crypto.getRandomValues` — NIST-recommended size for AES-GCM.
6. **AES-256-GCM encrypt.**
   - Web Crypto's `subtle.encrypt({ name: 'AES-GCM', iv }, ...)` produces `ciphertext || tag` in a single buffer (16-byte tag).
7. **Assemble envelope.**
   - `SPCK_MAGIC ‖ SPCK_VERSION ‖ ephemeralPub ‖ iv ‖ ciphertextWithTag`.

## 7. Decryption Pipeline

Implemented in `decryptWithCryptoPrivateKey()`:

```185:244:src/lib/cores/cryptographyCore/encryption.cryptoPublicKey.ts
export const decryptWithCryptoPrivateKey = async (
  recipientPrivateKey: string | Uint8Array,
  envelope: Uint8Array,
): Promise<Uint8Array> => {
  const subtle = requireSubtleCrypto();

  if (envelope.length < HEADER_LENGTH) {
    throw new Error('Envelope is too short to be a valid SPCK message');
  }

  for (let i = 0; i < SPCK_MAGIC.length; i++) {
    if (envelope[i] !== SPCK_MAGIC[i]) {
      throw new Error('Envelope does not have the expected SPCK magic header');
    }
  }
  const version = envelope[SPCK_MAGIC.length];
  if (version !== SPCK_VERSION) {
    throw new Error(`Unsupported SPCK envelope version: ${version}`);
  }

  const ephemeralPublicKey = envelope.slice(
    SPCK_MAGIC.length + 1,
    SPCK_MAGIC.length + 1 + COMPRESSED_PUBKEY_LENGTH,
  );
  const iv = envelope.slice(
    SPCK_MAGIC.length + 1 + COMPRESSED_PUBKEY_LENGTH,
    HEADER_LENGTH,
  );
  const ciphertext = envelope.slice(HEADER_LENGTH);

  const privateBytes =
    typeof recipientPrivateKey === 'string'
      ? hexToBytes(recipientPrivateKey)
      : recipientPrivateKey;
  if (privateBytes.length !== 32) {
    throw new Error('Private key must be 32 bytes (64 hex chars)');
  }
  if (!secp.utils.isValidSecretKey(privateBytes)) {
    throw new Error('Private key is out of valid secp256k1 range');
  }

  const sharedPoint = secp.getSharedSecret(privateBytes, ephemeralPublicKey, true);
  const sharedX = sharedPoint.slice(1);
  const aesKeyBytes = hkdf(sha256, sharedX, ephemeralPublicKey, HKDF_INFO, 32);
  ...
```

### Step-by-step

1. **Length sanity check.** Reject anything shorter than `HEADER_LENGTH` (50 bytes).
2. **Magic check.** First four bytes must equal `"SPCK"`. If not, fail loudly — this is what guards against feeding a `.gpg`/random binary into the wrong decryptor.
3. **Version check.** Byte 4 must equal `SPCK_VERSION` (currently `0x01`).
4. **Slice the envelope.** Pull out `ephemeralPub`, `iv`, and the ciphertext-with-tag tail using the constants.
5. **Validate the recipient private key.**
   - Must be exactly 32 bytes.
   - Must be in `[1, n-1]` (`isValidSecretKey`).
6. **ECDH agreement.**
   - `sharedPoint = recipient_d · eph_Q`.
   - Same shape as encrypt step 3, but with the keys swapped: now we use **our** secret and the **ephemeral** public key from the envelope.
7. **HKDF (identical parameters).**
   - `aesKey = HKDF(IKM=sharedX, salt=ephemeralPub, info="sparkplate-ecies-v1", length=32)`.
   - Same `info` constant, same salt → same 32-byte AES key.
8. **AES-256-GCM decrypt.**
   - If the ciphertext or tag was modified, `subtle.decrypt` throws. AES-GCM gives us tamper detection for free.
9. **Return plaintext bytes.**

## 8. Why The Same AES Key Falls Out Of Both Sides (ECDH Math)

This is the single mathematical fact the entire scheme rests on.

Let `G` be the secp256k1 generator and `n` its order. Let:

- Sender computes `eph_Q = eph_d · G` (ephemeral public key).
- Recipient's wallet keypair is `(recipient_d, recipient_Q)` with `recipient_Q = recipient_d · G`.

**Encryption side computes:**

\[
\text{sharedPoint}_{\text{enc}} = eph\_d \cdot recipient\_Q = eph\_d \cdot (recipient\_d \cdot G) = (eph\_d \cdot recipient\_d) \cdot G
\]

**Decryption side computes:**

\[
\text{sharedPoint}_{\text{dec}} = recipient\_d \cdot eph\_Q = recipient\_d \cdot (eph\_d \cdot G) = (recipient\_d \cdot eph\_d) \cdot G
\]

Scalar multiplication on an elliptic curve is commutative, so:

\[
(eph\_d \cdot recipient\_d) \cdot G = (recipient\_d \cdot eph\_d) \cdot G
\]

⇒ both sides arrive at **the exact same curve point**, and therefore the same X coordinate (`sharedX`). HKDF is deterministic given equal inputs (`IKM`, salt, info, length), so both sides derive the same 32-byte AES-256 key. AES-GCM with that key + the IV from the envelope decrypts what AES-GCM with the same key + that IV encrypted.

An attacker observing the wire sees the envelope, which contains `eph_Q` and the ciphertext but not `eph_d` or `recipient_d`. To recover `sharedPoint` they would need to solve the **Computational Diffie–Hellman problem on secp256k1**, which is presumed intractable.

## 9. Integration With `App.vue`

`App.vue` is the only consumer of `cryptographyCore`. It branches on the **input format**, not on a UI selector — the same Encrypt and Decrypt buttons handle both PGP and ECIES.

### Encrypt branch (chosen by recipient pubkey shape)

```188:209:src/App.vue

      const { envelope } = await encryptWithCryptoPublicKey(trimmedKey, fileData);

        envelope,
        `${file.name}.${CRYPTO_PUBKEY_FILE_EXTENSION}`,
        'application/octet-stream',

      return;
    }
```

If the user pastes a hex secp256k1 public key, the ECIES path runs and the file is downloaded as `<name>.spck`. Otherwise the OpenPGP path runs.

The decrypt branch detects ECIES by the `SPCK` magic, not by the user's key format. This is correct: the file knows what it is, and the user should not have to remember which key produced it.

### Convenience: pulling keys into the panels

The Cryptocurrency tab parses a URI of the form `%ticker%://%privateKey%`. Two helper buttons hand-off the parsed pieces:

- **Pull publicKey** copies the derived `cryptoPublicKey` (or PGP `publicKey`) into the Encrypt panel's pubkey textarea.
- **Pull privateKey** strips the `BTC://` (or other ticker) prefix from `cryptoInput` and copies the raw 32-byte hex private key into the Decrypt panel's private-key textarea.

Both helpers are pure UI plumbing — no cryptography happens in them. The actual asymmetric operations are still gated behind `encryptFile` / `decryptFile`.

## 10. Comparison With PGP/GPG Path

`App.vue` ships both. They are conceptually identical (encrypt-with-pubkey / decrypt-with-privkey), but the primitives differ.

| Aspect                   | PGP/GPG path                                         | SPCK / ECIES path                                              |
| ------------------------ | ---------------------------------------------------- | -------------------------------------------------------------- |
| Library                  | `openpgp`                                            | `@noble/secp256k1`, `@noble/hashes`, Web Crypto                |
| Public-key algorithm     | RSA / ElGamal / ECC (per the PGP key)                | secp256k1 (Bitcoin/Ethereum curve)                             |
| Session-key establishment | Random session key wrapped to recipient pubkey       | ECDH between ephemeral keypair and recipient keypair           |
| Symmetric cipher          | OpenPGP cipher suite (AES-CFB / AES-GCM)             | AES-256-GCM via Web Crypto                                     |
| KDF                       | OpenPGP S2K / KDF                                    | HKDF-SHA256 with `info="sparkplate-ecies-v1"`                  |
| Container                | ASCII-armored text (`-----BEGIN PGP MESSAGE-----`)   | Binary `SPCK \x01 …`                                           |
| File extension            | `.gpg`                                               | `.spck`                                                        |
| Identity model            | Long-lived PGP keypair, possibly passphrase-protected| Wallet keypair (also controls funds), no passphrase            |
| Authentication of sender  | Optional via embedded signature                      | Not provided (encryption is anonymous)                         |
| Multi-recipient           | Yes                                                  | No (one recipient pubkey per envelope)                         |
| Format detection          | OpenPGP armor headers                                | First 4 bytes `SPCK`                                           |

The two paths coexist without conflict because:

- The **encrypt** branch is selected by `isHexCryptoPublicKey(trimmedKey)` — a hex-shape predicate. PGP-armored keys cannot match it.
- The **decrypt** branch is selected by `hasSpckMagic(fileBytes)` — a 4-byte file-magic check. PGP-armored messages start with `-----`, not `SPCK`.

## 11. End-to-End Test Vector

Using the `00.references/addressbook-contacts.csv.spck` workflow that motivated this document.

1. **Keypair (BTC):**
   - `privateKey = 2115da3db7aceff1ae89aaf62f3bb1f78d9a86ca3bb36872b6c3f2c0e1c97d07`
   - `publicKey  = 0x0213d0733380bfce12fc81d2f1e7429549906c050235b0834ecb22912c78d18694`
   - `cryptoInput` URI form: `BTC://2115da3db7aceff1ae89aaf62f3bb1f78d9a86ca3bb36872b6c3f2c0e1c97d07`

2. **Encrypt (`addressbook-contacts.csv` ⇒ `addressbook-contacts.csv.spck`):**
   - User pastes URI into Cryptocurrency tab → engine derives public key.
   - User clicks **Pull publicKey** in Encrypt panel → `encryptionPublicKey` is set to the hex pubkey.
   - User selects the CSV file and clicks **Encrypt**.
   - `App.vue` matches `isHexCryptoPublicKey` → calls `encryptWithCryptoPublicKey(...)`.
   - Engine generates `eph_d`, computes `sharedX = (eph_d · recipientPub).x`, derives AES key, AES-GCM encrypts.
   - Browser downloads `addressbook-contacts.csv.spck` — first 4 bytes `0x53 0x50 0x43 0x4B`.

3. **Decrypt (`addressbook-contacts.csv.spck` ⇒ `addressbook-contacts.csv`):**
   - User clicks **Pull privateKey** → URI prefix stripped → raw hex private key in Decrypt textarea.
   - User selects the `.spck` file and clicks **Decrypt**.
   - `App.vue` reads bytes, `hasSpckMagic` is true → calls `decryptWithCryptoPrivateKey(...)`.
   - Engine extracts `eph_Q`, `iv`, ciphertext from envelope. Computes `sharedX = (recipient_d · eph_Q).x`. By the math in §7 this equals the encryptor's `sharedX`.
   - Same HKDF inputs → same AES-256 key. AES-GCM decrypts; if the file was tampered with the auth tag fails and an error is surfaced.
   - Browser downloads `addressbook-contacts.csv`.

4. **Negative tests worth running:**
   - Encrypt with `pubkey_A`, decrypt with `priv_B` (different keypair) → AES-GCM auth tag fails → "Cipher job failed" / "OperationError".
   - Decrypt a `.gpg` file via the SPCK path: `hasSpckMagic` returns false, OpenPGP path runs (correct fallback).
   - Decrypt a `.spck` file via the OpenPGP path (e.g. someone hacks the magic check): `openpgp.readMessage` throws "Misformed armored text" — this is the bug class fixed by the magic-sniff.

## 12. Security Considerations

1. **Same key reuse risk.** The `recipientPriv` here is the same secret that controls the wallet's funds. A compromise of the file's recipient key is also a compromise of the wallet. Treat envelopes accordingly.
2. **No sender authentication.** Anyone with `recipientPub` can produce a valid `.spck` file claiming to be anyone. If sender-binding matters, layer a signature inside the plaintext or sign the envelope externally.
3. **No passphrase.** Unlike OpenPGP private keys, raw 32-byte hex secrets cannot be passphrase-protected at the protocol level. Protect the textarea/clipboard accordingly.
4. **Replay / freshness.** AES-GCM provides integrity for the ciphertext + tag, but the envelope as a whole has no timestamp or nonce binding. An attacker cannot modify content but **can** replay an envelope. Add a timestamp to the plaintext if you need replay resistance.
5. **No forward secrecy of past messages w.r.t. long-term key compromise.** The ephemeral keypair is fresh per message, but a leaked `recipient_d` decrypts every past envelope addressed to it. This is the same weakness PGP has.
6. **IV reuse.** AES-GCM is catastrophic under IV reuse with the same key. Because the AES key is freshly derived per message (HKDF salt = ephemeral pubkey), and the IV is independently random per message, the probability of the same `(key, iv)` pair recurring is negligible.
7. **HKDF info string is namespaced.** `info="sparkplate-ecies-v1"` ensures keys derived for this protocol cannot collide with keys derived by some other secp256k1+HKDF protocol that happens to reuse the same curve and inputs.
8. **Web Crypto availability.** `requireSubtleCrypto()` and `requireGetRandomValues()` throw early in environments lacking secure RNG / AES-GCM. Do not catch these as ordinary errors.
9. **Side-channel posture.** `@noble/secp256k1` is constant-time at the JavaScript level it can control. AES-GCM is delegated to the platform's Web Crypto, which is implemented in native constant-time code on every supported runtime.

## 13. Limitations and Future Work

- **Single recipient only.** Adding multi-recipient would mean either repeating the envelope per recipient or wrapping a content-encryption key with each recipient's pubkey (PGP-style). The latter would extend the envelope format and bump `SPCK_VERSION`.
- **No detached signatures.** A future v2 envelope could append a signature block (e.g. ECDSA / Schnorr from the sender's wallet key) covering the ciphertext and ephemeral pubkey, paired with a sender-pubkey field.
- **Version negotiation.** The single-byte `SPCK_VERSION` already supports forward evolution. New versions should keep the magic and version bytes in fixed positions so detection logic remains stable.
- **Streaming.** Both encrypt and decrypt currently buffer the entire plaintext / ciphertext in memory. Large-file streaming would require chunked AES-GCM (multiple IVs, or a streaming AEAD construction such as AES-GCM-SIV with chunk counters).
- **Passphrase wrap.** A future helper could wrap the raw 32-byte secret under a PBKDF2/Argon2-derived AES key for at-rest storage, mirroring OpenPGP's S2K behavior.
