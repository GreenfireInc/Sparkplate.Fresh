# Methodology: Relationship Between PGP/GPG and Cryptocurrency Keypair File Encryption

**Date:** April 30, 2026  
**Project:** Sparkplate Storage  
**Scope:** Explain whether PGP/GPG is used when encrypting or decrypting files with cryptocurrency keypairs, and document how the two flows coexist in the codebase.

---

## 1. Direct Answer

No. When this codebase encrypts or decrypts files with **cryptocurrency keypairs**, it does **not** use PGP/GPG.

The application has two separate file-encryption systems behind the same Encrypt / Decrypt panels:

| Flow | Key type | Code path | File format |
| --- | --- | --- | --- |
| **PGP/GPG** | OpenPGP armored public/private keys | `openpgp.encrypt` / `openpgp.decrypt` in `src/App.vue` | `.gpg`, ASCII-armored OpenPGP message |
| **Cryptocurrency keypair** | secp256k1 public/private key hex | `encryptWithCryptoPublicKey` / `decryptWithCryptoPrivateKey` in `src/lib/cores/cryptographyCore/encryption.cryptoPublicKey.ts` | `.spck`, binary SPCK ECIES envelope |

The two flows share UI state and buttons, but they do not share cryptographic primitives or file formats.

---

## 2. Relationship Summary

The relationship is best described as **parallel alternatives**:

- PGP/GPG is the fallback path for OpenPGP armored keys and OpenPGP messages.
- Cryptocurrency keypair encryption is an ECIES-style path over secp256k1.
- The app decides which path to use by inspecting input shape:
  - On encrypt: a hex SEC1 cryptocurrency public key triggers the crypto path.
  - On decrypt: an `SPCK` file magic header triggers the crypto path.
- If those checks do not match, the code falls back to PGP/GPG.

This means the app supports a PGP-like user workflow for cryptocurrency keys (public key encrypts, private key decrypts), but it is **not PGP internally**.

### 3.2 PGP encryption fallback

When the Encrypt button is clicked, the app first checks whether the provided public key is a cryptocurrency public key. If not, it treats it as OpenPGP:

```200:221:src/App.vue

    const cleanPublicKey = cleanArmoredKey(trimmedKey);
```

The key distinction is the `return` inside the cryptocurrency branch. Once `isHexCryptoPublicKey(trimmedKey)` is true, execution never reaches `openpgp.readKey` or `openpgp.encrypt`.

When decrypting, the app first checks whether the selected file begins with the SPCK magic header. If not, it treats the file as an OpenPGP armored message:

Again, the `return` inside the SPCK branch prevents the OpenPGP branch from running for cryptocurrency-encrypted files.

The algorithm is ECIES-like:

1. Generate ephemeral secp256k1 keypair.
2. Compute ECDH with recipient public key.
3. Derive AES key using HKDF-SHA256.
4. Encrypt file bytes using AES-256-GCM.
5. Write a binary `.spck` envelope.

The decryptor:

1. Validates the envelope has SPCK magic and supported version.
2. Extracts the ephemeral public key, IV, and ciphertext.
3. Validates the recipient private key as a 32-byte secp256k1 secret.
4. Computes ECDH with recipient private key and ephemeral public key.
5. Re-derives the AES key with HKDF-SHA256.
6. Decrypts with AES-256-GCM.

## 5. How The App Chooses Between Them

### 5.1 Encrypt selection: public-key shape

The encrypt path branches by public-key format:

```48:51:src/lib/cores/cryptographyCore/encryption.cryptoPublicKey.ts
export const isHexCryptoPublicKey = (value: string): boolean => {
  const h = stripHexPrefix(value);
  return /^(02|03)[0-9a-fA-F]{64}$/.test(h) || /^04[0-9a-fA-F]{128}$/.test(h);
};
```

If the public key looks like a SEC1-encoded secp256k1 key, the app uses the cryptocurrency path:

```200:208:src/App.vue
    if (isHexCryptoPublicKey(trimmedKey)) {
      const { envelope } = await encryptWithCryptoPublicKey(trimmedKey, fileData);
      downloadFile(
        envelope,
        `${file.name}.${CRYPTO_PUBKEY_FILE_EXTENSION}`,
        'application/octet-stream',

      encryptionStatus.value = 'File encrypted with cryptocurrency public key (ECIES)!';
      return;
    }
```

If it does not, the app assumes the user pasted an OpenPGP armored public key and uses `openpgp.encrypt`.

### 5.2 Decrypt selection: file magic

The decrypt path branches by file format:

```235:241:src/App.vue
const hasSpckMagic = (bytes: Uint8Array): boolean => {
  if (bytes.length < SPCK_MAGIC.length) return false;
  for (let i = 0; i < SPCK_MAGIC.length; i++) {
    if (bytes[i] !== SPCK_MAGIC[i]) return false;
  }
  return true;
};
```

The SPCK magic header is defined in the cryptocurrency crypto core:

```10:19:src/lib/cores/cryptographyCore/encryption.cryptoPublicKey.ts
/**
 * File-magic identifying envelopes produced by this module: ASCII "SPCK"
 * followed by a single version byte. Allows decryptors to detect the
 * Sparkplate Crypto-pubKey ECIES format and reject unrelated blobs.
 */
export const SPCK_MAGIC = new Uint8Array([0x53, 0x50, 0x43, 0x4b]);
export const SPCK_VERSION = 0x01;

/** File extension recommended for ECIES-encrypted output. */
export const CRYPTO_PUBKEY_FILE_EXTENSION = 'spck';
```

If the file starts with `SPCK`, the app uses `decryptWithCryptoPrivateKey`. If it does not, the app falls through to OpenPGP.

## 6. Why The Earlier “Misformed Armored Text” Error Happened

The error came from giving an SPCK file to the OpenPGP parser.

OpenPGP expects armored message text such as:

```text
-----BEGIN PGP MESSAGE-----
...
-----END PGP MESSAGE-----
```

SPCK files are binary and begin with:

```text
SPCK\x01...
```

If the app fails to route an SPCK file to `decryptWithCryptoPrivateKey`, then OpenPGP sees binary SPCK bytes and tries to parse them as armored PGP text. That produces errors like:

```text
Error: Misformed armored text
```

The fix is exactly the current design: detect `SPCK_MAGIC` before calling `openpgp.readMessage`.

## 7. Conceptual Similarity vs Implementation Difference

Cryptocurrency keypair encryption is **conceptually similar** to PGP/GPG:

- Public key encrypts.
- Matching private key decrypts.
- The actual file payload is encrypted with a symmetric cipher.

But it is **not implemented with PGP/GPG**:

| Concept | PGP/GPG path | Cryptocurrency path |
| --- | --- | --- |
| Long-term key | OpenPGP keypair | secp256k1 wallet keypair |
| Public-key operation | OpenPGP library-managed key wrapping | ECDH between ephemeral key and wallet key |
| Symmetric encryption | OpenPGP-selected cipher | AES-256-GCM |
| KDF | OpenPGP internals | HKDF-SHA256 |
| File container | OpenPGP message / `.gpg` | SPCK binary envelope / `.spck` |
| Code module | `openpgp` inside `src/App.vue` | `src/lib/cores/cryptographyCore/encryption.cryptoPublicKey.ts` |

The UI intentionally presents them side by side because both are asymmetric keypair workflows, but the code keeps the cryptographic formats separated.

## 8. Practical Rules For Maintainers

1. Do not call `openpgp.readMessage` on `.spck` files. Check `SPCK_MAGIC` first.
2. Do not call `decryptWithCryptoPrivateKey` on `.gpg` files. It expects an SPCK envelope.
3. Do not treat cryptocurrency private keys as OpenPGP armored keys.
4. Do not treat OpenPGP public keys as secp256k1 public keys.
5. Keep `.spck` and `.gpg` extensions distinct in UI labels and download filenames.
6. If adding tests, assert both routing decisions:
   - secp256k1 public key → `encryptWithCryptoPublicKey`, output `.spck`.
   - SPCK magic → `decryptWithCryptoPrivateKey`.
   - PGP public key → `openpgp.encrypt`, output `.gpg`.
   - non-SPCK file → OpenPGP decrypt fallback.
