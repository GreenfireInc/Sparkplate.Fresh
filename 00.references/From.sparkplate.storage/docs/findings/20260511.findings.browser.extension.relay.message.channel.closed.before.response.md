# Console noise: `relay.<hash>.js — message channel closed before a response was received`

**Date:** Monday, May 11, 2026 (`date +%Y%m%d` → `20260511`)
**Status:** Not a bug in this codebase — extension-injected content script.
**Severity:** Cosmetic. Does not affect application behavior, cryptography correctness, or any data flow.

---

## Symptom

DevTools console prints:

```
relay.49c93797.js:1 Uncaught (in promise) Error: A listener indicated an asynchronous
response by returning true, but the message channel closed before a response was received
```

The leading `relay.<hash>.js:1` filename pattern is the give-away: the error is logged from a script named `relay.*.js`, served from a `chrome-extension://<extension-id>/` URL (visible by clicking the filename in DevTools — it will open the file inside an extension's source tree, not under `localhost:5173`).

It is reproducible on any page (this Vite dev server, your bank's website, `about:blank`) as long as the offending extension is installed and active. It is **not** triggered by any code in `src/`. Verified:

```
$ rg -n 'chrome\.runtime|browser\.runtime|onMessage|sendMessage|window\.postMessage' src/
src/App.vue:59:const validationMessage = ref('');         ← only a substring match
src/App.vue:166:    validationMessage.value = …
…
```

No call site in the application uses Chrome's runtime-messaging API.

---

## Root cause

This is a documented warning from Chromium's WebExtension messaging layer. The relevant contract (Chrome docs, [`runtime.onMessage`](https://developer.chrome.com/docs/extensions/reference/api/runtime#event-onMessage)):

> If you want to asynchronously use `sendResponse`, add `return true;` to the `onMessage` event handler.

A typical content-script handler that wants to do async work:

```js
chrome.runtime.onMessage.addListener((msg, sender, sendResponse) => {
  doAsyncStuff(msg).then((result) => sendResponse(result));
  return true; // signal "I'll call sendResponse later, keep the channel open"
});
```

The message channel between the page's content script and the extension's background/service-worker is kept alive by the browser **only while both ends exist**. The browser will tear it down — and throw exactly this error on the page side — when any of these happen before the async handler resolves:

1. **The service worker goes idle and gets GC'd.** Manifest V3 extensions (almost all 2026-era extensions) run their background as a *service worker* that Chromium kills after ~30 s of inactivity. If the handler's promise hasn't resolved by then, the channel closes.
2. **The page navigates away** (SPA route change, location reload, tab close).
3. **The Vite dev server fires HMR-driven full reloads** — the page identity changes mid-handler.
4. **The extension is updated or reloaded** while a message is in flight.
5. **The user opens DevTools and the extension throttles**, or the OS suspends the renderer.

The reason this surfaces *prominently* on a crypto-focused page like the Sparkplate Storage tool is that Solana / Tezos / Ethereum wallet extensions (Phantom, Solflare, Backpack, MetaMask, Brave Wallet, Rabby, …) inject **provider relay** scripts onto every page so that web-3 dapps can call `window.solana.request(...)` or `window.ethereum.request(...)`. These relays maintain at least one `onMessage` listener for every loaded tab. When you reload during dev (Vite HMR fires multiple full-page reloads per session), the relay's in-flight handler gets the message-channel rug pulled out from under it, and you see this error.

### Likely culprits on this machine

The `relay.49c93797.js` filename matches the build artifact of several Solana / EVM wallet extensions. The hash is content-addressed and changes per extension version; checking which extension this is requires looking at the **origin** of the file in DevTools:

1. Open DevTools → **Sources** tab.
2. Find `relay.49c93797.js` in the file tree (it will live under a `chrome-extension://<extension-id>/` root).
3. Click the extension ID; Chromium will jump you to `chrome://extensions/?id=<extension-id>` which names the offending extension.

Common matches given the project's domain:

- **Phantom** (`bfnaelmomeimhlpmgjnjophhpkkoljpa`) — Solana / Ethereum / Sui wallet. Ships `relay.*.js` as its content-script bridge.
- **Solflare** (`bhhhlbepdkbapadjdnnojkbgioiodbic`) — Solana wallet. Similar relay shape.
- **Backpack** (`aflkmfhebedbjioipglgcbcmnbpgliof`) — multi-chain wallet.
- **MetaMask** (`nkbihfbeogaeaoehlefnkodbefgpgknn`) — content-script `inpage.js` proxy.
- **Brave Wallet** (built-in, no extension ID) — same pattern.

## Is anything actually broken?

No. The error is purely a console-level warning about the extension's *own* internal teardown handling — the extension's promise rejects, the page sees an unhandled `Uncaught (in promise)`, and Chrome logs it. Nothing in the page DOM, your app's reactive state, or the SPCK / SPCK v2 / PGP cryptography paths depends on or observes this channel.

A quick sanity check you can run from DevTools console:

```js
// Should be undefined (page-level globals aren't affected).
typeof globalThis.chrome?.runtime?.lastError
```

And the application's own error states are surfaced through Vue refs (`encryptionStatus`, `decryptionStatus`, `validationMessage`, `cryptoErrorMessage`) — none of them touch this channel.

## "Fix" — three options, in order of recommendation

### Option A (recommended): Filter the noise in DevTools

The most pragmatic choice: keep the extension installed, hide the noise.

1. Open DevTools → Console.
2. Click the **filter** icon (or use the URL filter box at the top of the Console).
3. Add a negative URL filter: `-url:chrome-extension://`

This hides all messages whose source is any Chrome extension, including this warning, without affecting messages from your own app. Chrome persists this filter per-origin.

For a more surgical filter (only this specific extension), use the extension's ID instead:

```
-url:chrome-extension://bfnaelmomeimhlpmgjnjophhpkkoljpa
```

### Option B: Disable the offending extension on `localhost`

If you don't need wallet injection on the dev server (the Sparkplate Storage tool does **not** consume `window.solana` / `window.ethereum` — it reads private keys from the textarea):

1. `chrome://extensions/` → click the extension → **Site access** → **On specific sites** → add `https://localhost:5173`.
2. Or toggle **Allow access to file URLs** off, and the relay won't inject on `https://localhost`.

This eliminates the noise without uninstalling.

### Option C: Update / reinstall the extension

Most wallet extensions have already fixed the underlying race in their service-worker keepalive logic. Confirm via the extension's own version page (`chrome://extensions/?id=<id>` → details → click the version number). If the version is more than a few months old, update through the Chrome Web Store and the warning often disappears.

### What this codebase does **not** need to change

- No code in `src/` listens on `chrome.runtime.onMessage`, calls `chrome.runtime.sendMessage`, or attaches a `window.message` listener.
- The fix is **entirely** on the browser side; touching application code would not silence this warning.
- Specifically, *do not* try to wrap a global `window.addEventListener('unhandledrejection', …)` to swallow it — that would also swallow legitimate promise rejections from `openpgp`, `libsodium`, `bitcoinjs-lib`, the bridge, etc., and would mask real bugs.

## Future signal

Any console error that meets **all** of these criteria can be triaged as "browser extension, not app code" and routed to one of the three options above:

1. Filename in the error is `<word>.<hex-hash>.js` with no folder prefix (no `/src/`, no `/node_modules/`, no `?v=…`).
2. Clicking the filename opens a `chrome-extension://...` URL in Sources.
3. Removing the extension makes the error disappear; touching `src/` does not.

Conversely, if the error names a file under `/src/`, `/node_modules/.vite/deps/`, or carries a `?v=<hash>` query, it's an in-app or Vite issue — see the other findings docs in this directory (notably `20260504.findings.currencyCore.vite.dev.integration.errors.md` for the prebundle-fetch family and `20260511.findings.sol.derive.failed.to.fetch.solana.web3.js.md` for the most recent example).
