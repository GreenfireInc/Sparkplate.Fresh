# 20260510 — Findings: Encrypting to Solana (Ed25519 / Base58) public keys

**Date:** Sunday, May 10, 2026 (`date +%Y%m%d` → `20260510` at write time)  
**Scope:** How to extend the app's native cryptocurrency-public-key encryption (currently **SPCK / ECIES over secp256k1**) to **Solana**, whose keys live on **Ed25519** and are exposed in **Base58**.  
**Status:** Proposal / design note. Not implemented yet.  
**Companion:** [`20260510.findings.sol.encrypt.misformed.armored.text.md`](./20260510.findings.sol.encrypt.misformed.armored.text.md) — explains why a SOL public key currently falls into the OpenPGP path and throws "Misformed armored text". This document picks up where that one ends and proposes a real Ed25519 encryption path so the SOL key is actually usable as an encryption recipient.

---

## 1. Why secp256k1 ECIES doesn't apply directly to Solana

The current envelope (`src/lib/cores/cryptographyCore/encryption.cryptoPublicKey.ts`):

```text
[4] "SPCK" magic
[1] 0x01 version
[33] ephemeral secp256k1 public key (compressed, SEC1)
[12] AES-GCM IV
[N]  AES-GCM ciphertext + 16-byte tag
```

uses **`secp.getSharedSecret()`** (ECDH on **secp256k1**) → **HKDF-SHA256** → **AES-256-GCM**. The recipient is detected by `isHexCryptoPublicKey()`, which only matches **SEC1 hex** (`02|03 + 64 hex` or `04 + 128 hex`).

A **Solana** public key is:

- **32 raw bytes** of an **Ed25519** point compression,  
- presented to the user as **Base58** (`SOL.Solana.ts:402` → `publicKey.toBase58()`),  
- the **address** is the same Base58 string.

Two structural reasons it cannot be plugged straight into the existing flow:

1. **Different curve.** Ed25519 (twisted Edwards) ≠ secp256k1 (short Weierstrass). The `secp.*` math will reject the bytes outright (`isValidPublicKey` would fail, even ignoring the encoding).
2. **Different encoding.** Even on the right curve, the bytes are Base58, not hex; the predicate selection in `App.vue` will not route a Base58 string into any crypto path.

---

## 2. The standard answer: convert **Ed25519 → X25519** and use ECIES on Curve25519

Ed25519 and X25519 are the **same underlying curve** (Curve25519), with different encodings tuned for signing vs. ECDH. There is a well-defined birational map between them, and every major crypto library exposes it:

| Library                              | Public-key conversion                                  | Private-key conversion                                  |
|--------------------------------------|--------------------------------------------------------|---------------------------------------------------------|
| **libsodium**                        | `crypto_sign_ed25519_pk_to_curve25519`                 | `crypto_sign_ed25519_sk_to_curve25519`                  |
| **TweetNaCl** (`tweetnacl`)          | not directly exposed — use `nacl.box.keyPair.fromSecretKey` after deriving the X25519 secret from the Ed25519 seed | same, via the seed |
| **@noble/curves**                    | `ed25519.utils.toMontgomery(edPub)`                    | derive X25519 priv from Ed25519 seed via SHA-512 + clamp|

Both **`libsodium-wrappers-sumo`** and **`tweetnacl`** are already declared dependencies of this project (see `package.json`); **`@noble/ed25519`** and **`@noble/curves`** are also present.

Once both ends are on **X25519**, the rest of the recipe is mechanically the same as today's SPCK:

1. **Generate** an ephemeral **X25519** keypair.  
2. **ECDH**: `shared = X25519(ephemeralPriv, recipientX25519Pub)` → 32 bytes.  
3. **KDF**: `aesKey = HKDF-SHA256(shared, salt = ephemeralPub, info = "sparkplate-x25519-v1", 32)`.  
4. **AEAD**: encrypt with **AES-256-GCM** (already used today via WebCrypto `subtle.encrypt`) under a 12-byte random IV.  
5. **Envelope**: write a header + ciphertext to disk.

### 2a. Or use libsodium's pre-built primitive: `crypto_box_seal`

`libsodium.crypto_box_seal(plaintext, recipientX25519Pub)` packages **all of the above** (X25519 ECDH + key derivation + XSalsa20-Poly1305 AEAD + ephemeral pubkey embedding) into one call. The output is a self-contained sealed box that anyone holding the recipient's X25519 secret key can open with `crypto_box_seal_open`.

Trade-offs:

| Approach                                | Pros                                                        | Cons                                                               |
|-----------------------------------------|-------------------------------------------------------------|--------------------------------------------------------------------|
| **Manual X25519 + HKDF + AES-GCM**      | Same code shape as existing SPCK; reuses WebCrypto AES-GCM. | More lines of ours to audit; need to define our own envelope.      |
| **`crypto_box_seal` (sealed box)**      | One-call, well-reviewed, smaller surface area, `XChaCha20-Poly1305`/`XSalsa20-Poly1305` AEAD already chosen. | Different AEAD than secp256k1 path; depends on libsodium runtime init. |

Either works; the recommendation in §4 is to use the **manual** path so the new format mirrors the existing `SPCK` envelope and the implementation reuses helpers already proven (`hkdf`, `subtle.encrypt`).

## 3. Envelope design: extend SPCK with a curve-aware version

We can keep one file format and one extension (`.spck`) and **bump the version byte** to encode the curve. That way a single decryptor can decide between secp256k1 and X25519 by reading byte 4.

| Byte         | Layout v1 (existing, secp256k1)               | Layout v2 (proposed, X25519/Solana)             |
|--------------|-----------------------------------------------|-------------------------------------------------|
| 0..3         | `"SPCK"` magic                                | `"SPCK"` magic                                  |
| 4            | `0x01`                                        | `0x02`                                          |
| 5..37        | 33 bytes — compressed secp256k1 ephemeral pub | 32 bytes — X25519 ephemeral pub (raw)           |
| next 12      | AES-GCM IV                                    | AES-GCM IV                                      |
| rest         | AES-GCM(ct + tag)                             | AES-GCM(ct + tag) (or sealed-box ciphertext)    |

Notes:

- Ephemeral pub field changes **size** between versions (33 → 32). The decryptor must branch on the version byte.
- Constants get version-scoped: keep `COMPRESSED_PUBKEY_LENGTH = 33` for v1 and add `X25519_PUBKEY_LENGTH = 32` for v2.
- Keep `info = "sparkplate-x25519-v1"` distinct from the secp256k1 string so the same shared-secret can never be re-used across formats by mistake.

## 4. Proposed code shape

### 4.1. New module `encryption.ed25519PublicKey.ts`

Mirrors `encryption.cryptoPublicKey.ts` but with X25519 math. Public surface (proposed):

```ts
export const SPCK_VERSION_X25519 = 0x02;
export const X25519_PUBKEY_LENGTH = 32;

export const isBase58Ed25519PublicKey: (value: string) => boolean;
export const ed25519PubFromBase58:    (b58: string) => Uint8Array;        // 32 bytes
export const ed25519ToX25519Public:   (edPub: Uint8Array) => Uint8Array;  // 32 bytes
export const ed25519SeedToX25519Priv: (seed: Uint8Array) => Uint8Array;   // 32 bytes

export const encryptWithEd25519PublicKey: (
  recipientEd25519PublicKey: string | Uint8Array,
  plaintext: Uint8Array,
) => Promise<{ envelope: Uint8Array; ephemeralX25519Pub: Uint8Array; iv: Uint8Array }>;

export const decryptWithEd25519PrivateKey: (
  recipientEd25519PrivateKeySeed: string | Uint8Array,  // 32-byte seed (Solana JSON gives 32)
  envelope: Uint8Array,
) => Promise<Uint8Array>;
```

Implementation choice for the `ed25519 → x25519` conversion (any **one** of the following — pick whichever is most stable in the WASM/Vite stack):

- **`@noble/curves/ed25519`**: `ed25519.utils.toMontgomery(edPub)` for the public side; private side derived as `clamp(sha512(seed)[0..32])`.
- **`libsodium-wrappers-sumo`**:
  ```ts
  await sodium.ready;
  const xPub  = sodium.crypto_sign_ed25519_pk_to_curve25519(edPub);
  const xPriv = sodium.crypto_sign_ed25519_sk_to_curve25519(edFullSecretKey64);
  ```
  Note: libsodium's `sk_to_curve25519` expects the **64-byte** Ed25519 secret key (seed ‖ pubkey). For Solana JSON whose `privateKey` is the 32-byte seed, build the 64-byte form first as `seed || ed25519.getPublicKey(seed)`.

The ECDH then runs on `@noble/curves/ed25519` `x25519.scalarMult(xPriv, peerXPub)` (or libsodium's `crypto_scalarmult`). HKDF-SHA256 + AES-GCM are already used in `encryption.cryptoPublicKey.ts` — reuse them verbatim.

### 4.2. Bridge: extend `isSpckCompatible`

`bridge.Cryptography.Via.Currency.ts` currently sets `isSpckCompatible` from `isSec1HexPublicKey(publicKey)`. Two clean options:

- **Single flag, broader meaning.** `isSpckCompatible = isSec1HexPublicKey(pub) || isBase58Ed25519PublicKey(pub)`. Simpler UI, but loses the curve hint.
- **Two flags.** Add `spckCurve: 'secp256k1' | 'x25519' | null`. UI / encryption module branches on this rather than re-detecting strings.

Recommendation: **two flags** so the encryption call site stays explicit (`encryptWithCryptoPublicKey` vs. `encryptWithEd25519PublicKey`).

### 4.3. `App.vue` dispatch

Replace the current single `if (isHexCryptoPublicKey(...))` with a small dispatcher:

```ts
const trimmedKey = encryptionPublicKey.value.trim();

if (isHexCryptoPublicKey(trimmedKey)) {
  // existing secp256k1 / SPCK v1 path
} else if (isBase58Ed25519PublicKey(trimmedKey)) {
  const { envelope } = await encryptWithEd25519PublicKey(trimmedKey, fileData);
  // reuse buildEncryptedFilename / resolveEncryptedExtension; ticker = SOL
  encryptionStatus.value = 'File encrypted with SOL public key (X25519/ECIES)!';
  return;
} else {
  // existing PGP fallback (now reached only for actual armored keys)
}
```

The decrypt side mirrors this by reading the **version byte** of the envelope rather than re-detecting from the key string.

### 4.4. UI cues from the bridge

The Solana refs in `useCryptocurrencyEngine` already include `cryptoTickerSymbol === 'SOL'` and `cryptoPublicKey` (Base58). Once `isBase58Ed25519PublicKey` exists:

- `pullPublicKey` no longer needs the "non-SPCK-compatible" warning for SOL — both `secp256k1` and `Ed25519` derived keys are pull-able.
- The encrypt status message changes from "use PGP" to "encrypted with SOL public key".

Keep the current guard for any **truly** unsupported curve (e.g., sr25519 keys from Polkadot if the bridge ever returns them as a non-X25519/non-secp256k1 string).

## 5. Decryption inputs from a Solana JSON wallet

The reference file `00.References/CoreyStedman.sol.<addr>.json.json` carries `{ "privateKey": "<base58>" }`. Two cases:

| Decoded length | Meaning                                     | Use as Ed25519 seed   |
|----------------|---------------------------------------------|-----------------------|
| **32**         | Raw Ed25519 **seed**                        | seed = decoded        |
| **64**         | Solana keypair (`seed ‖ pubkey`)            | seed = decoded[0..32] |

`SOL.Solana.ts` already normalizes both. The new `decryptWithEd25519PrivateKey` should accept the same inputs (Base58, JSON array, hex) and reduce to a 32-byte seed before calling `ed25519SeedToX25519Priv`.

## 6. Security notes specific to this approach

- **Convert before ECDH.** Never feed an Ed25519 public key directly to `x25519.scalarMult`; the encoding mismatch silently produces garbage. Always run the Edwards→Montgomery conversion first.
- **Distinct domain separators.** The HKDF `info` for v2 must differ from v1 ("`sparkplate-x25519-v1`" vs. "`sparkplate-ecies-v1`"). Otherwise an attacker who can adaptively choose curves could pivot between them.
- **Solana keys are signing keys.** Reusing the same key for signing **and** encryption (via the X25519 conversion) is widely accepted (this is exactly what libsodium's `crypto_sign_*_to_curve25519` exists for), but document it in the user-facing copy.
- **Authentication.** `crypto_box_seal` and AES-GCM both authenticate ciphertext. Do **not** drop the tag in any custom layout; without it the v2 envelope becomes malleable.
- **Constant-time Base58.** Decoding Base58 from user input should reject non-canonical lengths early (Solana pubkeys are exactly 32 bytes after decoding); accept only canonical 32-byte decodings to avoid ambiguity with truncated keystore arrays.

## 7. Test plan (when implemented)

1. **Round-trip**: encrypt an arbitrary file with the SOL Base58 pubkey, decrypt with the SOL Base58 / hex / JSON-array seed; bytes match.
2. **Cross-curve negative**: try to decrypt a v1 (`0x01`) envelope with the v2 path and vice versa — must fail with a clear error, not WebCrypto's generic "operation failed".
3. **Wrong-key negative**: decrypt with a different SOL seed; AES-GCM tag check must fail.
4. **JSON seed forms**: cover Base58 (32 and 64 decoded bytes), hex (32 and 64), and JSON-array (32 and 64). Match `SOL.Solana.ts`'s parsing matrix.
5. **Filename**: extension still `.spck`; ticker tag in the filename ("`<base>.SOL.spck`") via existing `resolveEncryptedExtension`.
6. **No regression**: BTC / ETH / DOGE round-trip on v1 still passes (must be unaffected by adding v2).

## 8. Out of scope / future work

- **Sr25519 / BLS / Schnorr-only keys.** Polkadot's sr25519 lives on Ristretto255; X25519 conversion does not apply, but a separate Ristretto-based ECIES could be added with a new SPCK version byte (e.g., `0x03`).
- **Hybrid recipients.** Encrypting once for **multiple** recipients (PGP-style) is not handled by SPCK today and is unrelated to this proposal.
- **Hardware wallets.** Decrypting SOL envelopes via Ledger/Trezor would require those devices to expose `ed25519 → x25519` ECDH; today they generally do not. Out of scope.
