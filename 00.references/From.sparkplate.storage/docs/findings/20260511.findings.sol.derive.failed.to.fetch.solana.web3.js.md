# SOL derive error: `Failed to fetch dynamically imported module: …/@solana_web3__js.js`

**Date:** Monday, May 11, 2026 (`date +%Y%m%d` → `20260511`)
**Status:** Resolved
**Severity:** Medium — blocks every Solana private-key derivation on first run after a fresh dev-server start; intermittently affects other currencies that route through the same dynamic-import chain.

---

## Symptom

Pasting a SOL private-key URI into the **Native Cryptocurrency** tab surfaced:

```
Error: Failed to fetch dynamically imported module:
https://localhost:5173/node_modules/.vite/deps/@solana_web3__js.js?v=619cb519
```

`@solana_web3__js.js` corresponds to the prebundled form of `@solana/web3.js`. It is already declared in `vite.config.ts`'s `optimizeDeps.include`, so the error is not "package was never prebundled" — it is the prebundle's `?v=` hash being **invalidated mid-request** by a parallel discovery rebundle.

---

## Root Cause

A short refresher on Vite's pre-bundling pipeline (full write-up in [Finding 6 of `20260504.findings.currencyCore.vite.dev.integration.errors.md`](20260504.findings.currencyCore.vite.dev.integration.errors.md)):

1. At dev-server start, Vite scans the **static** import graph reachable from `index.html` / `src/main.ts` and pre-bundles every discovered npm dep into `node_modules/.vite/deps/<pkg>.js?v=<hashA>`.
2. `optimizeDeps.include` lets you pre-declare extra packages so dynamic imports (`await import('pkg')`) don't have to wait for runtime discovery.
3. If a package is reached at runtime that the startup scan **didn't** know about, Vite re-runs the prebundle. Re-bundling bumps the hash on **every** `?v=<hash>` URL, so any in-flight request to a stale URL now 404s. The browser surfaces the generic *"Failed to fetch dynamically imported module"*.

In this session we added the new SPCK v2 encryption module:

```
src/lib/cores/cryptographyCore/encryption.ed25519PublicKey.ts
```

…which is statically imported by `src/App.vue` and brings four new top-level static `import` lines:

```ts
import { x25519 } from '@noble/curves/ed25519.js';
import { sha256 } from '@noble/hashes/sha2.js';
import { hkdf }   from '@noble/hashes/hkdf.js';   // ← NEW
import bs58 from 'bs58';
```

Cross-referencing against `vite.config.ts`'s `optimizeDeps.include`:

| Static import | In `optimizeDeps.include`? |
|---------------|-----------------------------|
| `@noble/curves/ed25519.js` | ✓ |
| `@noble/hashes/sha2.js`    | ✓ |
| `@noble/hashes/blake2.js`  | ✓ |
| `@noble/hashes/legacy.js`  | ✓ |
| `@noble/hashes/utils.js`   | ✓ |
| **`@noble/hashes/hkdf.js`** | **✗** |
| `bs58`                     | ✓ |

`@noble/hashes/hkdf.js` is the only entrant in the new static graph that the include list doesn't cover. Vite's startup scanner *should* still find it (it's a static import), but the trigger is the **prebundle cache**: if the dev server is started against a `node_modules/.vite/` cache that was built **before** the new static import was added, Vite resolves the import lazily during the first request, and that lazy resolution triggers the rebundle-with-hash-bump race.

### Why `@solana/web3.js` is the visible victim

`SOL.Solana.ts:327` does:

```ts
const solanaWeb3 = await import('@solana/web3.js');
```

It is one of the largest prebundled chunks (~2 MB) and takes meaningful time to stream from disk. Window of vulnerability:

```
t0   browser → GET /node_modules/.vite/deps/@solana_web3__js.js?v=619cb519
t1   Vite worker sees `encryption.ed25519PublicKey.ts` brought in
     `@noble/hashes/hkdf.js` (not in cache) → spawns rebundle
t2   Rebundle finishes, every prebundled dep is now stamped ?v=<NEW HASH>
t3   Vite GC removes the old ?v=619cb519 file
t4   Browser's t0 request 404s →
     "Failed to fetch dynamically imported module"
```

Other (less common) victims of the same race in this codebase would be `@stellar/stellar-base`, `xrpl`, `algosdk` — anything large enough to still be in flight when Vite finishes the rebundle. SOL is the most reliable trigger because it's the largest and because the SOL derivation path is the first one a typical user exercises after the encryption refactor.

## Fix

### 1. Pre-declare `@noble/hashes/hkdf.js` in `optimizeDeps.include`

```ts
// vite.config.ts
optimizeDeps: {
  include: [
    // …
    '@noble/hashes/blake2.js',
    '@noble/hashes/hkdf.js',      // ← NEW: added by encryption.ed25519PublicKey.ts (SPCK v2)
    '@noble/hashes/legacy.js',
    '@noble/hashes/sha2.js',
    '@noble/hashes/utils.js',
    // …
  ],
}
```

This guarantees `hkdf.js` is in the first prebundle pass at startup, so no runtime discovery → no rebundle → no `?v=` hash bump → no stale 404.

### 2. Invalidate the stale prebundle cache

```bash
rm -rf node_modules/.vite
```

Vite re-prebundles on the next `npm run dev`, this time with `hkdf.js` in the first pass.

### 3. (Recommended) Restart the dev server

Hot-reload alone does not regenerate `optimizeDeps.include` discovery. After the config edit + cache wipe, `npm run dev` from a clean shell.

## Future signal

The diagnostic recipe from the May-04 doc still applies — when *any* "Failed to fetch dynamically imported module: `.../node_modules/.vite/deps/<pkg>.js?v=…`" shows up:

1. The named `<pkg>` is **the victim**, not necessarily the cause.
2. The cause is a *different* package that's missing from `optimizeDeps.include` and was discovered mid-request.
3. To find it: search the dev-server console for the line  
   `[vite] new dependencies optimized: <pkg-a>, <pkg-b>, …`  
   immediately after the failed request — those are the dependencies whose discovery triggered the rebundle that 404'd the victim.
4. Add every name from that list to `optimizeDeps.include`, then `rm -rf node_modules/.vite && npm run dev`.

### Hygiene rule for this codebase

When adding a new module under `src/lib/cores/cryptographyCore/` (or any other module that's statically imported from `App.vue`), audit its top-level `import` lines against `optimizeDeps.include`. Any new bare-specifier import that doesn't appear in the list is a latent prebundle-race waiting for someone's first cold start. The cost of including is a single extra prebundle entry; the cost of omitting is "Failed to fetch" errors that are very hard to reason about because the error names a different package than the cause.
