# 20260510 — Findings: BTC P2TR derivation → "Buffer is not defined" in browser

**Date:** Sunday, May 10, 2026 (`date +%Y%m%d` → `20260510` at write time)  
**Scope:** Importing or pasting any **Bitcoin** (`BTC://`) private key (WIF or hex) and triggering derivation in the live Vite dev server. Surfaces during the multi-format pass (`P2PKH + P2SH + P2WPKH + P2TR`) as a `ReferenceError` thrown from the **P2TR** branch and caught by the BTC router.  
**Outcome:** The fix is a **single global polyfill** of Node's `Buffer` for the browser, since the npm package is already in `node_modules` and many code paths (BTC, BCH, several minting modules, and `bitcoinjs-lib` / `ecpair` internals) expect it to be defined globally.

---

## Symptom

```
BTC.Bitcoin.ts:496 [BTC ROUTER] Bitcoin derivation routing error: ReferenceError: Buffer is not defined
    at Object.deriveFromPrivateKey (BTC.Bitcoin.P2TR.ts:18:34)
    at async Promise.all (:5173/index 3)
    at async Object.deriveFromPrivateKey (BTC.Bitcoin.ts:461:67)
    at async deriveCryptocurrencyViaCurrency (bridge.Cryptography.….Currency.ts:118:18)
    at async recompute (bridge.Cryptography.….Currency.ts:193:22)
```

The line number in the stack (`18:34`) is from the source-mapped browser view — the source-of-truth `Buffer.from(...)` calls in the current `BTC.Bitcoin.P2TR.ts` are at **line 42** (`ECPairFactory.fromPrivateKey(Buffer.from(privateKeyBytes))`) and **line 52** (`Buffer.from(keyPair.publicKey)`). The class of error is the same regardless of the exact line.

---

## Root cause

`Buffer` is a **Node.js global**. It is **not** part of the Web platform and is **not** exposed by browsers. Vite (v3+) deliberately does **not** polyfill Node globals automatically.

The codebase was authored against a Node-style assumption — it references `Buffer` directly without ever `import`-ing it. A quick survey:

| Area                                                         | Uses bare `Buffer`? |
|--------------------------------------------------------------|---------------------|
| `currencyCore/currencies/ext.BTC.Bitcoin/BTC.Bitcoin.P2TR.ts` | **Yes** (lines 42, 52) — direct trigger of this error. |
| `currencyCore/currencies/BCH.BitcoinCash.ts`                 | Yes (lines 595, 614, 615, 653, 664, 665, 706, 708, 749, 763, 772–774). |
| `currencyCore/exchanges/coinbase.ts`                         | Yes (line 59).      |
| `currencyCore/minting/**` (XLM, XTZ, ALGO, AR, ETC, LUNA, ETH, XRP) | Yes (file/JSON buffer plumbing). |

Additionally, **`bitcoinjs-lib`** and **`ecpair`** ship code paths that internally expect `Buffer` to be available as a global on Node. While the v7+ APIs accept `Uint8Array` at the boundary, the ecpair `fromPrivateKey()` path the P2TR module uses is wrapped in `Buffer.from(privateKeyBytes)` by **our** code, which is what trips the `ReferenceError` first.

The `buffer` npm package (v6.0.3) is already present in `node_modules` as a transitive dependency, so the fix is cheap: import it once at app entry and assign it to `globalThis.Buffer`.

---

## Why this didn't show up earlier

- The error is in the **P2TR** branch only, which is one of four parallel calls in `BTC.Bitcoin.ts:461` (`Promise.all([...])`). When the BTC router runs in single-format mode that doesn't include P2TR, the codepath is skipped and the bug stays latent.
- Other tickers (`SOL`, `XTZ`, etc.) don't reach `Buffer` in the derivation path — they use `Uint8Array` end-to-end via `@noble/*`. So the failure is **BTC/BCH-specific** at first glance, but the polyfill is needed broadly because of the minting modules.
- In Node-side tests (`vue-tsc`, lint, etc.), `Buffer` exists, so `vite build` and type-checking don't surface it.

---

## Solution — Stage 1: global Buffer polyfill in `src/main.ts`

Single, minimal change. Put it **before any other crypto-touching import** so the global is set up by the time `currencyCore` modules dynamic-import their dependencies.

### Code change

**`src/main.ts`** (current 5 lines):

```ts
import { createApp } from 'vue'
import './style.css'
import App from './App.vue'

createApp(App).mount('#app')
```

**Proposed:**

```ts
import { Buffer } from 'buffer';
if (typeof globalThis.Buffer === 'undefined') {
  // Node's `Buffer` is not exposed by browsers, but the bitcoinjs-lib /
  // ecpair stack and a handful of currencyCore modules (BTC P2TR, BCH,
  // minting helpers) still call `Buffer.from(...)` and `Buffer.concat(...)`.
  // The `buffer` npm package is already a transitive dependency; binding it
  // to `globalThis` here is the smallest fix that keeps every Buffer.* call
  // working unchanged in the browser.
  globalThis.Buffer = Buffer;
}

import { createApp } from 'vue'
import './style.css'
import App from './App.vue'

createApp(App).mount('#app')
```

That is the entire functional fix.

### Pin `buffer` as a direct dev/runtime dep (recommended)

The package resolves transitively today, but explicit is better than implicit. Add to `package.json`:

```jsonc
"dependencies": {
  // ...existing
  "buffer": "^6.0.3"
}
```

`npm install buffer` is a no-op for `node_modules` since the version is already present; it just records intent in `package.json` so a future dependency dedupe or version bump cannot silently delete it.

### Vite consideration

Vite's prebundler should ship the `buffer` import without issue. If you start seeing slow first-load (rare), add it to `optimizeDeps.include` in `vite.config.ts`:

```ts
optimizeDeps: {
  include: [
    'buffer',
    // ...existing entries
  ],
}
```

This is **not** required for correctness — only for prebundle scheduling.

## Why polyfill instead of removing every `Buffer.*` call

Two choices were considered:

| Approach                                                | Pros                                                                                                                                  | Cons                                                                                                                                  |
|---------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------|
| **Global polyfill (chosen)**                            | One-line change; works for our code, our deps, and any future Node-style helper added to `currencyCore`. Zero behavior change.        | Adds ~50 KB gzipped to the bundle (the `buffer` package). Couples the project to a Node-style global at the edges.                    |
| **Replace `Buffer` with `Uint8Array` throughout**       | Smaller bundle; more web-native code.                                                                                                 | Touches dozens of files (BCH.BitcoinCash.ts alone has 15+ `Buffer.from` / `Buffer.concat` calls); some library APIs really do want a `Buffer` (e.g. `Buffer.concat([sigBytes, [hashType]])`); high churn for low payoff. |
| **`vite-plugin-node-polyfills`**                        | Handles `Buffer`, `process`, `global`, `crypto`, `stream`, etc. all at once.                                                          | Larger plugin surface; pulls in polyfills we don't need; harder to audit. Already have a working secp256k1 path without it.            |

Polyfill is the best ROI for this codebase right now. A future cleanup pass can replace `Buffer.*` with `Uint8Array.*` per file when those files are otherwise modified.

## Notes specific to `BTC.Bitcoin.P2TR.ts`

Two `Buffer` calls in this file are technically unnecessary even without a polyfill — `keyPair.publicKey` from `ecpair@^3` is **already a `Uint8Array`**, and `.slice()` works on `Uint8Array` directly:

```ts
// Current (relies on Buffer global):
const publicKeyCompressed = Buffer.from(keyPair.publicKey);
const xOnlyPubkey = publicKeyCompressed.slice(1, 33);

// Equivalent without Buffer:
const publicKeyCompressed = keyPair.publicKey;          // Uint8Array
const xOnlyPubkey = publicKeyCompressed.slice(1, 33);   // Uint8Array
```

And `ECPairFactory.fromPrivateKey()` accepts `Uint8Array` in the `ecpair@^3` typings:

```ts
// Current:
keyPair = ECPairFactory.fromPrivateKey(Buffer.from(privateKeyBytes));
// Equivalent:
keyPair = ECPairFactory.fromPrivateKey(privateKeyBytes);
```

These are **optional refactors** — once the global polyfill is in, both versions work. Apply when you're touching the file anyway. Same observation applies to `BCH.BitcoinCash.ts` for its `Buffer.from(uint8Array)` call sites; the `Buffer.concat([...])` and `.toString('hex')` calls there genuinely depend on `Buffer` semantics and should keep using the polyfill.

## Test plan (after the polyfill lands)

1. **Reload the dev server** (the polyfill runs at app boot, so a hot-replace alone may not pick it up; do a full reload).
2. **BTC private key (WIF)**: paste a mainnet WIF (5… / K… / L…); router must produce **all four** addresses (`1…`, `3…`, `bc1q…`, `bc1p…`) without throwing. Confirm via the formatted display.
3. **BTC private key (hex)**: paste a 64-char hex; same outcome as WIF.
4. **BCH**: derive a Bitcoin Cash address; confirm `Buffer.from(... , 'hex').toString('hex')` round-trip in `BCH.BitcoinCash.ts` no longer throws.
5. **Node test (sanity)**: `node -e "require('buffer').Buffer.from('beef','hex').toString('hex')"` — already worked, just confirms the package import shape used in `main.ts`.
6. **Build**: `npm run build` (vue-tsc + vite build) must complete without TypeScript errors. The `globalThis.Buffer` assignment is well-typed via `@types/node`, which is already a dev dependency.
7. **Bundle-size sanity**: spot-check the output bundle size before/after; expect ~+50 KB gzip.
