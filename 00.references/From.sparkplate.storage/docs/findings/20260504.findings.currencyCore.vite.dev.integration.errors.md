## Finding 1 — `Unsupported ticker: LUNC` (and other non-legacy tickers)

### Symptom

```
Unsupported ticker: LUNC. Supported: BTC, ETH, LTC, DOGE, BCH, DASH.
```

Surfaced as `cryptoErrorMessage` in the **Native Cryptocurrency** tab of `App.vue` whenever a non-legacy ticker (XTZ, LUNC, SOL, etc.) was entered or imported via the JSON-import button.

### Root cause

Two cryptocurrency engines coexisted in the codebase:

- **Legacy:** `src/lib/cores/engine/cryptocurrencyEngine.ts` — hardcoded `SUPPORTED_TICKERS` of `[BTC, ETH, LTC, DOGE, BCH, DASH]`, all using inline secp256k1 + bs58check derivation.
- **Bridge:** `src/lib/cores/bridge/bridge.Cryptography.Via.Currency.ts` — a drop-in replacement that delegates to `currencyByTicker` from `currencyCore/currencies/`, automatically supporting every currency present in the registry (currently 20).

`App.vue` was still importing `SUPPORTED_TICKERS` and `useCryptocurrencyEngine` from the **legacy** engine path, so the ticker check failed against the 6-coin allowlist before the bridge was ever consulted.

### Solution

Swapped the import path. Both modules export identically-named symbols, so destructuring at the call site stayed unchanged:

```ts
// Before
import {
  SUPPORTED_TICKERS,
  useCryptocurrencyEngine,
} from './lib/cores/engine/cryptocurrencyEngine';

// After
import {
  SUPPORTED_TICKERS,
  useCryptocurrencyEngine,
} from './lib/cores/bridge/bridge.Cryptography.Via.Currency';
```

Also updated the supported-ticker hint to drop the misleading *"(32-byte hex private key)"* caveat — accurate for the legacy engine only — to *"(each ticker accepts its native private-key format)"* since the bridge accepts XTZ `edsk…`, SOL base58, etc.

### Future signal

If a ticker that *should* be supported ever reports "Unsupported": confirm the offending error string is the bridge's (`Supported: ALGO, AR, ATOM, …`) rather than the legacy engine's (6-coin list). Mismatched-length supported lists are the tell that the wrong engine is being imported.

## Finding 2 — Vite cannot resolve the `@/` path alias

### Symptom

```
[plugin:vite:import-analysis] Failed to resolve import
"@/lib/cores/currencyCore/currencies/ext" from
"src/lib/cores/currencyCore/currencies/STX.Stacks.ts"
```

### Root cause

Many files under `src/lib/cores/currencyCore/currencies/` use the `@/` import prefix — a common Vue/Vite convention for "from the `src/` root". But this project's config had no alias registered:

- `vite.config.ts` had no `resolve.alias` entry for `@`.
- `tsconfig.app.json` had no `compilerOptions.paths` entry.

The target file `src/lib/cores/currencyCore/currencies/ext/index.ts` actually existed and re-exported `getExpectedStacksBitcoinAddress` (via `testFixtures.ts`); the toolchain just had no map from `@` → `./src`.

### Solution

Registered the alias in **both** Vite (runtime resolver) and TypeScript (compile-time resolver, so `vue-tsc` and the IDE agree):

```ts
// vite.config.ts
import { fileURLToPath, URL } from 'node:url'

export default defineConfig({
  // …
  resolve: {
    alias: {
      '@': fileURLToPath(new URL('./src', import.meta.url)),
    },
  },
})
```

```jsonc
// tsconfig.app.json
"compilerOptions": {
  // …
  "baseUrl": ".",
  "paths": {
    "@/*": ["./src/*"]
  }
}
```

### Future signal

- `@/components/ui/...` imports in `ext.DOT.Polkadot/DOT.Polkadot.ImportModal.tsx` still target a non-existent shadcn-vue UI tree. Nothing imports `ImportModal.tsx`, so it's outside the live build graph and Vite never analyzes it. If that file is wired in later, you'll need to either install shadcn-vue or rewrite the UI in this project's idiom.

## Finding 3 — `"ESM integration proposal for Wasm" is not supported`

### Symptom

```
[vite] Internal server error: "ESM integration proposal for Wasm" is not
supported currently. Use vite-plugin-wasm or other community plugins to
handle this. Alternatively, you can use `.wasm?init` or `.wasm?url`.
```

### Root cause

The **WebAssembly/ESM Integration Proposal** (Stage-3) lets `.wasm` files be imported as if they were JS modules — `import init from 'foo.wasm'`. Vite has no built-in handler for this; its default `.wasm` loader explicitly throws this error to point users at the right plugin.

In this project, `tiny-secp256k1@2.x` (a transitive dep of `bitcoinjs-lib`, which is statically imported by `ext.BTC.Bitcoin/router.BTC.Bitcoin.ts` and the LTC counterpart) ships:

```js
// node_modules/tiny-secp256k1/lib/wasm_loader.browser.js
import * as wasm from "./secp256k1.wasm";
export default wasm;
```

That `import * as ... from "*.wasm"` is the exact ESM-proposal syntax Vite refuses to handle natively.

### Solution

Two-part fix:

1. **Installed plugins:** `vite-plugin-wasm@3.6.0` and `vite-plugin-top-level-await@1.6.0` (the latter is required because WASM modules instantiate via top-level `await`, which esbuild's default lowering rejects).

2. **Wired them into `vite.config.ts` plus told Vite to skip prebundling the offender** so the request goes through the dev pipeline where `vite-plugin-wasm` is active:

```ts
import wasm from 'vite-plugin-wasm'
import topLevelAwait from 'vite-plugin-top-level-await'

export default defineConfig({
  plugins: [vue(), wasm(), topLevelAwait()],
  optimizeDeps: {
    exclude: ['tiny-secp256k1'],
  },
})
```

After updating, also `rm -rf node_modules/.vite` so the prebundle cache rebuilds with the new exclude list.

### Future signal

Diagnostic recipe when this error reappears:

1. `find node_modules -maxdepth 4 -name '*.wasm'` to locate the binary.
2. `grep -l "from \".*\.wasm\"" node_modules/<pkg>/**/*.{js,mjs}` to confirm the package uses the ESM proposal.
3. Add the package to `optimizeDeps.exclude`.
4. `rm -rf node_modules/.vite && npm run dev`.

## Finding 4 — `does not provide an export named 'BitcoinAddressFormat'`

### Symptom

```
Uncaught SyntaxError: The requested module
'/src/lib/cores/currencyCore/currencies/ext.BTC.Bitcoin/router.BTC.Bitcoin.ts'
does not provide an export named 'BitcoinAddressFormat' (at BTC.Bitcoin.ts:21:3)
```

Identical pattern with `LitecoinAddressFormat` for the LTC equivalent path.

### Root cause

`router.BTC.Bitcoin.ts` declares the format type as a **type-only** export:

```ts
export type BitcoinAddressFormat = 'P2PKH' | 'P2SH' | 'P2WPKH' | 'P2TR';
```

TypeScript erases that at compile time — there is no runtime export with that name. But the consuming file imported it as a plain value binding mixed in with real value imports:

```ts
import {
  bitcoinP2PKHData,
  bitcoinP2SHData,
  bitcoinP2WPKHData,
  bitcoinP2TRData,
  BitcoinAddressFormat,   // ← treated as a runtime binding
} from './ext.BTC.Bitcoin/router.BTC.Bitcoin';
```

`tsconfig.app.json` enables `"erasableSyntaxOnly": true`, which forbids the compiler from inferring "this binding is type-only based on usage" — every type binding must be explicitly marked. Esbuild therefore preserved `BitcoinAddressFormat` in the runtime import list, the browser's strict ESM loader looked for an actual export with that name, and threw.

### Solution

Inline `type` modifier (TypeScript 4.5+) on each affected binding so the compiler unambiguously erases it from the runtime import:

```ts
// BTC.Bitcoin.ts
import {
  bitcoinP2PKHData,
  bitcoinP2SHData,
  bitcoinP2WPKHData,
  bitcoinP2TRData,
  type BitcoinAddressFormat,
} from './ext.BTC.Bitcoin/router.BTC.Bitcoin';

// LTC.Litecoin.ts
import {
  litecoinP2PKHData,
  litecoinP2SHData,
  litecoinP2WPKHData,
  type LitecoinAddressFormat,
} from './ext.LTC.Litecoin/router.LTC.Litecoin';

// ext/multiFormatAddresses.tsx
import { bitcoinData, type BitcoinAddressFormat } from "@/lib/cores/currencyCore/currencies/BTC.Bitcoin";
import { litecoinData, type LitecoinAddressFormat } from "@/lib/cores/currencyCore/currencies/LTC.Litecoin";
```

### Future signal

Whenever you see `does not provide an export named 'X'` and `X` is `PascalCase` (likely a type), grep the source export for `export type X` — if found, switch the importer to inline `type` modifier. Under `erasableSyntaxOnly`, this is required, not optional.

## Finding 5 — `does not provide an export named 'bytes'` (`@polkadot/wasm-crypto-wasm/cjs/bytes.js`)

### Symptom

```
Uncaught SyntaxError: The requested module
'/node_modules/@polkadot/wasm-crypto-wasm/cjs/bytes.js?v=0dbcf048'
does not provide an export named 'bytes' (at bundle.js?v=0dbcf048:2:10)
```

### Root cause

A self-inflicted regression from the Finding 3 fix.

`@polkadot/wasm-crypto-wasm` ships a **dual build**: `cjs/bytes.js` (`module.exports = { bytes }`) and `esm/bytes.js` (`export const bytes = ...`). Polkadot's own `bundle.js` does `import { bytes } from './bytes.js'` and expects to be served the ESM variant.

In Finding 3 I had defensively added every `@polkadot/wasm-*` package to `optimizeDeps.exclude`. But Polkadot does **not** use the WASM ESM proposal — it instantiates WASM via inline byte arrays + `WebAssembly.compile()`. By excluding them, I bypassed Vite's prebundle step that would have converted the CJS `bytes.js` to a proper ESM shim with the named export. Vite resolved the package's `main` field instead (the CJS variant) and shipped it raw to the browser, which then threw.

### Solution

Tightened `optimizeDeps.exclude` to **only** the package that genuinely uses the WASM ESM proposal:

```ts
optimizeDeps: {
  exclude: [
    'tiny-secp256k1',   // confirmed offender — has `import * as wasm from "./*.wasm"`
    // DO NOT add `@polkadot/wasm-*` here. They use inline-bytes + compile().
  ],
}
```

Cleared `node_modules/.vite` so Vite re-prebundles cleanly.

### Future signal

Decision matrix for `optimizeDeps.exclude`:

| Package style | Action |
|---|---|
| Uses `import * as wasm from "./*.wasm"` | **Exclude** — let `vite-plugin-wasm` handle. |
| Loads WASM via inline bytes + `WebAssembly.compile()` | **Don't exclude** — let prebundle convert CJS→ESM. |
| Pure-JS, ESM-only | **Don't exclude** — default behavior is correct. |

Quick triage: errors pointing at `node_modules/.vite/...` mean *needs* prebundling; errors pointing at `node_modules/<pkg>/cjs/...` mean *was* incorrectly excluded.

## Finding 6 — `Failed to fetch dynamically imported module: …/bs58.js?v=…`

### Symptom

```
Error: Failed to fetch dynamically imported module:
http://localhost:5173/node_modules/.vite/deps/bs58.js?v=3d7f2848
```

(`bs58` was the first module to surface the problem; any package in the dynamic-import chain could.)

### Root cause

The currencies code loads almost every npm dep via `await import('pkg')` (lazy / dynamic). Vite's startup-time scanner only follows **static** imports to populate its prebundle, so all those packages are unknown at boot time. The first runtime call triggers a **dynamic discovery rebundle**:

1. `await import('bs58')` → Vite spins up rebundle, serves `bs58.js?v=3d7f2848`.
2. Microseconds later, `await import('bech32')` → another rebundle.
3. Each new rebundle **bumps the `?v=<hash>` query for every previously-bundled dep**, invalidating in-flight URLs.
4. The browser's pending request for `bs58.js?v=3d7f2848` becomes stale; Vite returns 504; the browser surfaces the generic *"Failed to fetch dynamically imported module"*.

This is a documented Vite footgun in the **Dependency Pre-Bundling** guide and is especially common in crypto/blockchain projects that lazy-load many heavy packages.

### Solution

Pre-declared every dynamically-imported package in `optimizeDeps.include` so Vite prebundles them in **a single pass at startup**, eliminating the race:

```ts
optimizeDeps: {
  include: [
    // Noble suite
    '@noble/curves/ed25519.js', '@noble/curves/secp256k1.js',
    '@noble/ed25519',
    '@noble/hashes/blake2.js', '@noble/hashes/legacy.js',
    '@noble/hashes/sha2.js', '@noble/hashes/utils.js',
    '@noble/secp256k1',
    // Scure suite
    '@scure/base', '@scure/bip32', '@scure/btc-signer',
    // Chain SDKs
    '@solana/web3.js', '@stacks/network', '@stacks/transactions',
    '@stellar/stellar-base', '@waves/ts-lib-crypto',
    'algosdk', 'arweave', 'ripple-keypairs', 'xrpl',
    // Encodings / key formats
    'bech32', 'bip39', 'bs58', 'bs58check', 'wif',
    // Crypto primitives
    'crypto-js', 'libsodium-wrappers-sumo', 'scrypt-js', 'tweetnacl',
    // Bitcoin tooling
    'bitcoinjs-lib', 'ecpair',
  ],
  exclude: ['tiny-secp256k1'],
}
```

Cleared `.vite` cache so the next start does a fresh single-pass prebundle.

### Future signal

Whenever a new ticker is added to `currencyCore/currencies/` and brings a new dynamic import, append the package to `optimizeDeps.include`. Diagnostic: a "Failed to fetch dynamically imported module" error pointing at `/node_modules/.vite/deps/<pkg>.js?v=…` means `<pkg>` is missing from the include list.

## Finding 7 — `Cannot add property sha512Sync, object is not extensible`

### Symptom

```
Error: Cannot add property sha512Sync, object is not extensible
```

Thrown by Tezos (`XTZ.Tezos.ts`) and Solana (`SOL.Solana.ts`) derivation paths when calling `ed25519.getPublicKey(...)`.

### Root cause

`@noble/ed25519` is a hash-less Ed25519 implementation: the consumer must wire in a SHA-512 function before sync APIs work. The wiring API **changed between v2 and v3**:

| Version | Wiring API |
|---|---|
| v2 | `ed25519.etc.sha512Sync = (...m) => sha512(ed25519.etc.concatBytes(...m))` |
| **v3** | `ed25519.hashes.sha512 = sha512` |

Two breaking changes in v3:

1. The `sha512Sync` slot **moved off `etc`** and onto a new `hashes` object (matching the same renaming `@noble/secp256k1` v3 did with `secp.hashes.sha256`).
2. **`etc` was frozen** so its TypeScript surface is fixed. Assigning to `etc.sha512Sync` now throws `Cannot add property sha512Sync, object is not extensible`.

Confirmed v3 shape from the installed type declaration:

```ts
declare const etc: {
    bytesToHex: (bytes: TArg<Bytes>) => string;
    hexToBytes: (hex: string) => TRet<Bytes>;
    concatBytes: (...arrs: TArg<Bytes[]>) => TRet<Bytes>;
    mod: typeof M;
    invert: typeof invert;
    randomBytes: (len?: number) => TRet<Bytes>;
};
declare const hashes: {
    sha512Async: (message: TArg<Bytes>) => Promise<TRet<Bytes>>;
    sha512: undefined | ((message: TArg<Bytes>) => TRet<Bytes>);
};
```

Both `XTZ.Tezos.ts` and `SOL.Solana.ts` were using the v2 API against a v3-installed library.

### Solution

Migrated both files to the v3 wiring point. The new `hashes.sha512` accepts a single `Bytes` argument, matching `sha512` from `@noble/hashes/sha2.js` directly — so the v2 `concatBytes(...m)` wrapper is no longer needed:

```ts
// XTZ.Tezos.ts
const ed25519 = await import("@noble/ed25519");
if (!ed25519.hashes.sha512) {
  ed25519.hashes.sha512 = sha512;
}

// SOL.Solana.ts
const ed25519Module = await import('@noble/ed25519');
if (!ed25519Module.hashes.sha512) {
  ed25519Module.hashes.sha512 = sha512;
}
```

The `if (!…)` guard makes the wiring idempotent so repeated `deriveFromPrivateKey` calls don't redundantly reassign.

### Future signal

When updating a `@noble/*` package across a major version, the surface area to inspect is the *injection points*. The current convention across the v3 noble suite is:

```ts
secp.hashes.sha256 = sha256;
secp.hashes.hmacSha256 = (key, msg) => hmac(sha256, key, msg);
ed.hashes.sha512 = sha512;
```

If a future v4 freezes `hashes` and moves the slot again, the symptom will be the same `Cannot add property X, object is not extensible` — re-check the upstream `index.d.ts`.

## Cross-cutting playbook

The seven findings cluster into three failure categories. When something breaks in this stack, work through them in this order:

### A) Vite resolver / prebundle layer

Symptoms appear at startup or first navigation, mention `node_modules/.vite/deps/...` URLs, or contain phrases like "Failed to resolve import" / "Failed to fetch dynamically imported module" / "ESM integration proposal".

1. Static import unresolved? → check `resolve.alias` (Finding 2).
2. WASM-load throw? → identify the WASM-importing package via `find node_modules -name '*.wasm'` and add to `optimizeDeps.exclude` (Finding 3).
3. Missing named export from a `cjs/...` path? → that package was wrongly excluded; remove from `exclude` (Finding 5).
4. Generic "Failed to fetch dynamically imported module"? → add the package name to `optimizeDeps.include` and clear `.vite` cache (Finding 6).

### B) TypeScript / esbuild ESM-output layer

Symptoms appear at runtime as `Uncaught SyntaxError: ... does not provide an export named 'X'`, where `X` is your own module's name (not from `node_modules`).

5. Type imported as value? → use inline `type` modifier (Finding 4). Required under `erasableSyntaxOnly: true`.

### C) Library API breakage layer

Symptoms appear at runtime as the library throwing — frozen-object errors, "function is not a constructor", missing methods.

6. `Cannot add property X, object is not extensible`? → upstream froze its API surface; re-read the package's `index.d.ts` and migrate to the new injection points (Finding 7).

### D) Application-layer integration

7. Wrong engine wired up? → grep `App.vue` and any other entry point for stale paths (Finding 1).
