# 20260510 — Findings: LTC P2SH / P2WPKH derivation → "Buffer is not defined" in browser

**Date:** Sunday, May 10, 2026 (`date +%Y%m%d` → `20260510` at write time)  
**Scope:** Importing or pasting any **Litecoin** (`LTC://`) private key (WIF or hex) and triggering derivation in the live Vite dev server. Surfaces during the multi-format pass (P2PKH + **P2SH** + **P2WPKH**) as nested `Error: Buffer is not defined` thrown from each SegWit-aware branch and caught by the LTC router.  
**Outcome:** Same class of bug, same fix as the BTC P2TR finding from earlier today. The single global polyfill in [`docs/findings/20260510.findings.btc.p2tr.buffer.is.not.defined.md`](./20260510.findings.btc.p2tr.buffer.is.not.defined.md) **also** resolves Litecoin P2SH / P2WPKH derivation. This note explains the LTC-specific surface so future maintainers don't re-investigate.

## Root cause

Same as the BTC finding — `Buffer` is a Node.js global, browsers don't expose it, and Vite (v3+) doesn't auto-polyfill it. Litecoin's per-format derivation modules call `Buffer.*` directly:

| File                                                                                  | Bare `Buffer.*` call sites |
|---------------------------------------------------------------------------------------|----------------------------|
| `src/lib/cores/currencyCore/currencies/ext.LTC.Litecoin/LTC.Litecoin.P2SH.ts`         | lines 61, 69, 74, 97, 98   |
| `src/lib/cores/currencyCore/currencies/ext.LTC.Litecoin/LTC.Litecoin.P2WPKH.ts`       | lines 61, 69, 74, 92, 93   |
| `src/lib/cores/currencyCore/currencies/ext.LTC.Litecoin/LTC.Litecoin.P2PKH.ts`        | (none — pure noble-curves path) |

Hot path that throws (`LTC.Litecoin.P2SH.ts:61`):

```ts
const privateKeyBytes = fromHex(privateKey);          // Uint8Array, OK
// ...
keyPair = ECPairFactory.fromPrivateKey(Buffer.from(privateKeyBytes));   // ← ReferenceError here
```

The next two lines (`Buffer.from(keyPair.publicKey).toString('hex')`, `bitcoin.payments.p2wpkh({ pubkey: Buffer.from(keyPair.publicKey), ... })`) would throw the same way if execution reached them.

The router (`LTC.Litecoin.ts:272`) fans out to all three formats via `Promise.all`. **P2PKH succeeds**, **P2SH and P2WPKH fail**, so the parallel call rejects and the router re-throws on line 303 with the wrapped message.

---

## Why P2PKH worked but the SegWit formats didn't

`LTC.Litecoin.P2PKH.ts` derives the public key with `@noble/secp256k1` and builds the P2PKH script manually using `Uint8Array` operations — no `bitcoinjs-lib` payments helper, no `Buffer` references. The two SegWit formats use `bitcoinjs-lib`'s `bitcoin.payments.p2wpkh` / `p2sh` and our wrapper code feeds it `Buffer.from(...)`. That branch is the one that needs `Buffer` to exist.

---

## Solution

**Apply the polyfill from [the BTC finding](./20260510.findings.btc.p2tr.buffer.is.not.defined.md).** Verbatim — there is nothing LTC-specific to add. After the four-line edit to `src/main.ts`:

```ts
import { Buffer } from 'buffer';
if (typeof globalThis.Buffer === 'undefined') {
  globalThis.Buffer = Buffer;
}
```

…all three Litecoin formats derive correctly:

- `L…` (P2PKH), `M…` (P2SH-P2WPKH), `ltc1q…` (P2WPKH).

If the polyfill has already been applied for BTC and Litecoin still fails, the failure is a different bug; do **not** re-investigate `Buffer is not defined`.

### Optional micro-cleanups (no behavior change once polyfill is in)

These mirror the cleanup notes in the BTC finding. `ecpair@^3` accepts `Uint8Array` natively, and `bitcoinjs-lib@^7` payments helpers do too:

```ts
// LTC.Litecoin.P2SH.ts:61 — current
keyPair = ECPairFactory.fromPrivateKey(Buffer.from(privateKeyBytes));
// equivalent without Buffer
keyPair = ECPairFactory.fromPrivateKey(privateKeyBytes);

// LTC.Litecoin.P2SH.ts:74 — current
const p2wpkh = bitcoin.payments.p2wpkh({ pubkey: Buffer.from(keyPair.publicKey), network: LTC_NETWORK });
// equivalent without Buffer
const p2wpkh = bitcoin.payments.p2wpkh({ pubkey: keyPair.publicKey, network: LTC_NETWORK });

// LTC.Litecoin.P2SH.ts:97-98 — current
const publicKeyCompressed = Buffer.from(keyPair.publicKey);
const publicKeyUncompressed = Buffer.from(keyPair.publicKey);
// equivalent without Buffer
const publicKeyCompressed = keyPair.publicKey;          // already Uint8Array
const publicKeyUncompressed = keyPair.publicKey;
```

`Buffer.from(uint8Array).toString('hex')` (line 69) genuinely needs `Buffer` semantics until you swap it for a `Uint8Array` → hex helper (the project's `toHex` from `../utils` is already available in this file).

Apply only when touching the file for other reasons — the polyfill makes them unnecessary.

## Reference: hot-path source

`LTC.Litecoin.P2SH.ts` lines 49–67 — the hex branch that fires the `ReferenceError`:

```ts
} else {
  // Hex format
  try {
    const privateKeyBytes = fromHex(privateKey);
    if (privateKeyBytes.length !== 32) {
      throw new Error(`Expected 32 bytes, got ${privateKeyBytes.length}`);
    }

    const ECPair = await import('ecpair');
    const tinysecp256k1 = await import('tiny-secp256k1');
    const ECPairFactory = ECPair.ECPairFactory(tinysecp256k1);
    keyPair = ECPairFactory.fromPrivateKey(Buffer.from(privateKeyBytes));   // ← throws here

    console.log('[LTC P2SH] Successfully parsed hex private key');
  } catch (hexError) {
    throw new Error(`Invalid Litecoin private key: ${hexError.message}`);   // wraps the ReferenceError
  }
}
```

`LTC.Litecoin.P2WPKH.ts` has an identical structure (line 61 is the same `Buffer.from(privateKeyBytes)` call), which is why both formats fail with the exact same wrapped message in the same `Promise.all`.
