# 20260510 — Findings: XTZ (Tezos) tz1 + file encrypt → "not compatible with native SPCK/ECIES" / "Misformed armored text"

**Date:** Sunday, May 10, 2026 (`date +%Y%m%d` → `20260510` at write time)  
**Scope:** Encrypting a file after importing `00.References/CoreyStedman.xtz.tz1h1MhMCE4oh39oqJDg5XD1tew6F4LuxUCn.json` (or any `XTZ://edsk…` URI whose derived address starts with **`tz1`**, i.e. **Ed25519**).  
**Outcome:** The user-facing warning is **accurate** for the current build, but it under-states the situation: this is the **same Ed25519-vs-secp256k1 mismatch** as Solana, just with a hex-encoded public key instead of Base58. The fix is the same Stage 2 X25519 path documented for SOL, with one extra predicate to catch **hex-encoded** Ed25519 keys.  
**Companions:**
- [`20260510.findings.sol.encrypt.misformed.armored.text.md`](./20260510.findings.sol.encrypt.misformed.armored.text.md) — original Ed25519/SPCK gap analysis and the Stage 2 X25519 design that this doc reuses.
- [`20260510.findings.sol.ed25519.x25519.publickey.encryption.proposal.md`](./20260510.findings.sol.ed25519.x25519.publickey.encryption.proposal.md) — deeper protocol/library notes for X25519.

---

## Symptom

After importing the `tz1…` JSON, switching to **Encrypt File**, **Pull publicKey**, selecting a file, and clicking **Encrypt**, the status panel shows:

```
XTZ is not compatible with native SPCK/ECIES in this app (public key must be
secp256k1 SEC1 hex). OpenPGP was attempted because the key is not hex SEC1,
which produces "Misformed armored text". Use PGP with an armored key, or
encrypt with a secp256k1 coin (e.g. BTC, ETH, DOGE).
```

(Wording from the Stage 1 guard in `App.vue` `encryptFile`.)

---

## Is the message true?

**Yes — for `tz1` addresses, in this build.** With three nuances:

1. **Tezos has three address families** with three different curves:

   | Prefix | Curve     | Pubkey format produced by `XTZ.Tezos.ts` | SPCK v1 (secp256k1)        |
   |--------|-----------|------------------------------------------|----------------------------|
   | `tz1`  | **Ed25519** (this user)                | 32-byte hex (64 chars, no `02/03/04` prefix) | **Not** compatible        |
   | `tz2`  | **secp256k1**                           | (currently not produced by the codebase as SEC1 hex) | Would be compatible with format work |
   | `tz3`  | **P-256** (a.k.a. secp256r1)            | (not produced)                                    | **Not** compatible (different curve) |

   The reference JSON derives a **`tz1…`** address → Ed25519. So the Ed25519 branch is the relevant one.

2. **The pubkey is hex, not Base58.** `XTZ.Tezos.ts` returns `pubHex = toHex(edPub)` — a **raw 32-byte Ed25519 point** as **64 hex characters**. SPCK's predicate `isHexCryptoPublicKey()` requires either:
   - `(02|03)[0-9a-fA-F]{64}` (compressed secp256k1, **66** chars), or
   - `04[0-9a-fA-F]{128}` (uncompressed secp256k1, **130** chars).

   A 64-char Ed25519 hex never matches either, so the SPCK branch is correctly skipped.

3. **"OpenPGP was attempted" is slightly misleading.** With the Stage 1 guard from the SOL fix in place, OpenPGP is **not** actually called for this case — the guard short-circuits before `openpgp.readKey`. The message describes what *would* happen without the guard (which is exactly the prior "Misformed armored text" failure mode). A copy refinement candidate, but the underlying logic is sound.

So: **the warning correctly tells the user that XTZ tz1 cannot use SPCK today**. It's just incomplete on *why* (Ed25519 vs. secp256k1) and *what to do about it long-term* (X25519 path).

---

## Root cause

Identical class of bug to SOL, with the encoding twist:

1. **SPCK / ECIES** (`encryption.cryptoPublicKey.ts`) is a **secp256k1**-only ECIES envelope (33-byte ephemeral pub + 12-byte IV + AES-GCM).
2. **Tezos `tz1`** uses **Ed25519** signing keys. `XTZ.Tezos.ts:483–500` derives the public key with `@noble/ed25519.getPublicKey(sk)` and serializes it via `toHex(edPub)` — a 32-byte raw Ed25519 point in hex.
3. **`isHexCryptoPublicKey()`** is a **SEC1**-shape detector and (correctly) rejects the 64-char Ed25519 hex.
4. **Stage 1 guard** then catches the `XTZ`/`!isSpckCompatible` combo and surfaces the "not compatible / Misformed armored text" warning instead of letting OpenPGP try to parse the hex as armor.

The JSON file content is fine (`{"privateKey":"edsk…"}` decodes cleanly to a 32-byte Ed25519 seed). The bug class is **wrong pipeline** — the only encryption pipeline available is secp256k1.

---

## Solution — Stage 1: stop the bogus armor error (already shipped)

The same `App.vue` guard introduced for SOL catches XTZ as well, because `bridge.Cryptography.Via.Currency.ts` flips `isSpckCompatible` to **`false`** whenever the derived public key fails `isSec1HexPublicKey`. The user gets the explanation above instead of `Error: Misformed armored text`.

What this **does not** fix: a `tz1` (Ed25519) public key still cannot be used as an encryption recipient in this app.

### Optional copy refinement

The Stage 1 message says "OpenPGP was attempted". A more accurate phrasing once the guard is in place would be:

> "XTZ tz1 addresses use Ed25519, which this app's native SPCK/ECIES (secp256k1-only) does not support. OpenPGP would not parse this key either ('Misformed armored text'). Use PGP with an armored key for now, or wait for the X25519 path."

(Pure UX polish; no behavior change.)

## Solution — Stage 2: a new crypto implementation for Ed25519 keys (XTZ tz1 + SOL share this path)

The same Ed25519 → X25519 conversion proposed for SOL applies verbatim to XTZ `tz1`. The only XTZ-specific addition is a **predicate for hex-encoded Ed25519 keys**, since SOL's predicate looks for Base58.

### Curve choice: convert Ed25519 → X25519, then ECDH

See [`20260510.findings.sol.encrypt.misformed.armored.text.md` §"Solution — Stage 2"](./20260510.findings.sol.encrypt.misformed.armored.text.md) for the full algorithm. Summary:

| Step       | Conversion                                                     | Library helper                                            |
|------------|----------------------------------------------------------------|-----------------------------------------------------------|
| Public     | `xPub = ed25519_pub_to_curve25519(edPub)`                      | `libsodium.crypto_sign_ed25519_pk_to_curve25519` / `@noble/curves/ed25519` |
| Private    | `xPriv = clamp(SHA-512(seed)[0..32])` (Tezos `edsk` → 32-byte seed) | `libsodium.crypto_sign_ed25519_sk_to_curve25519`         |
| ECDH       | `shared = X25519(xPriv, xPubPeer)`                              | `@noble/curves` `x25519.scalarMult` / `libsodium.crypto_scalarmult` |
| KDF        | `aesKey = HKDF-SHA256(shared, salt = ephemeralPub, info = "sparkplate-x25519-v1", 32)` | reuse from `encryption.cryptoPublicKey.ts` |
| AEAD       | AES-256-GCM, 12-byte IV                                         | WebCrypto `subtle.encrypt` (already used in v1)           |

### Envelope: SPCK v2 (same magic, version byte `0x02`)

Identical to the SOL proposal:

```
[4]  "SPCK"
[1]  0x02
[32] X25519 ephemeral public key (raw)
[12] AES-GCM IV
[N]  AES-GCM(ciphertext ‖ tag)
```

KDF info `"sparkplate-x25519-v1"` is domain-separated from v1's `"sparkplate-ecies-v1"`.

### XTZ-specific: an additional predicate `isHexEd25519PublicKey`

SOL's `isBase58Ed25519PublicKey` needs a sibling for hex:

```ts
// True for raw 32-byte Ed25519 public key in hex (64 chars, optional 0x prefix),
// AND not already matched by isHexCryptoPublicKey (which catches SEC1 02/03/04 prefixes).
export const isHexEd25519PublicKey = (value: string): boolean => {
  const h = value.replace(/^0x/i, '').trim();
  if (!/^[0-9a-fA-F]{64}$/.test(h)) return false;
  // Disambiguate from SEC1: SEC1 compressed is 02|03 + 64 hex (66 chars total, not 64),
  // and uncompressed is 04 + 128 hex (130 chars). A 64-char hex is unambiguously
  // a raw 32-byte point — only valid here as Ed25519.
  return true;
};
```

Notes:

- Length disambiguates from SEC1 cleanly: 64 chars ≠ 66 ≠ 130. No prefix collision is possible.
- A 32-byte hex *could* in principle be other curves' raw points (X25519 itself, or secp256k1 X-coordinate), but those are not values the bridge currently emits. If the bridge starts producing raw secp256k1 X-coords for any ticker, the bridge should also set `spckCurve` so the dispatcher trusts the curve hint over the predicate.

### Bridge change: communicate the curve

`bridge.Cryptography.Via.Currency.ts` should expose:

```ts
export interface CryptocurrencyDerivation {
  // ... existing ...
  spckCurve: 'secp256k1' | 'x25519' | null;
  isSpckCompatible: boolean; // true iff spckCurve !== null
}
```

`spckCurve` resolution per ticker:

| Ticker (example)                  | Derived public-key shape                  | `spckCurve`     |
|-----------------------------------|--------------------------------------------|------------------|
| BTC, ETH, DOGE, LTC, BCH, ETC     | SEC1 hex (66 / 130 chars)                  | `'secp256k1'`    |
| **XTZ (tz1)**                     | **Raw 32-byte Ed25519, hex (64 chars)**    | **`'x25519'`**   |
| SOL                               | Base58 32-byte Ed25519                     | `'x25519'`       |
| LUNA, LUNC, ATOM, BNB             | (depends on chain — verify with bridge)    | varies           |
| DOT, sr25519 chains               | sr25519 / Ristretto255                     | `null` (out of scope today) |
| TRX, XLM, XRP, ALGO, AR, WAVES, STX, XTZ-tz3 | other / not applicable for v1 or v2 | `null`           |

### `App.vue` dispatch

Replace the single `isHexCryptoPublicKey` check with a curve-aware ladder:

```ts
const trimmedKey = encryptionPublicKey.value.trim();

if (isHexCryptoPublicKey(trimmedKey)) {
  // existing v1 / secp256k1 path (BTC, ETH, ...)
} else if (
  isBase58Ed25519PublicKey(trimmedKey) || isHexEd25519PublicKey(trimmedKey)
) {
  const { envelope } = await encryptWithEd25519PublicKey(trimmedKey, fileData);
  const extensionResolution = resolveEncryptedExtension(
    trimmedKey, cryptoPublicKey.value, cryptoTickerSymbol.value,
  );
  downloadFile(
    envelope,
    buildEncryptedFilename(file.name, extensionResolution),
    'application/octet-stream',
  );
  encryptionStatus.value = `File encrypted with ${cryptoTickerSymbol.value || 'Ed25519'} public key (X25519 / SPCK v2)!`;
  return;
}

// PGP fallback (only reached for actual armored input)
```

`encryptWithEd25519PublicKey` accepts either Base58 (SOL) or hex (XTZ) for the recipient pubkey; internally it normalizes to a 32-byte buffer, then runs the SOL Stage 2 algorithm unchanged.

### Decryption inputs from a Tezos `edsk` JSON wallet

The reference file `00.References/CoreyStedman.xtz.tz1….json` carries `{"privateKey": "edsk…"}`. Decoding with `bs58` gives **40 bytes** (`4-byte prefix + 32-byte seed + 4-byte checksum`), matching `XTZ.Tezos.ts:428–442`. The 32-byte seed is exactly what the X25519 conversion needs — the same shape SOL accepts. No XTZ-specific decryptor is required; `decryptWithEd25519PrivateKey` should accept:

| Input                                              | Decoded length | Seed used         |
|----------------------------------------------------|----------------|-------------------|
| Tezos `edsk…` (Base58)                             | 40             | decoded[4..36]    |
| Tezos `edsk…` (Base58, extended)                   | 68 / 72        | decoded[4..36]    |
| Hex (XTZ users sometimes use raw 64-char hex)      | 32             | decoded           |
| Solana Base58                                      | 32 or 64       | decoded[0..32]    |
| JSON array                                         | 32 or 64       | decoded[0..32]    |

A small `normalizeEd25519Seed()` helper in the new module owns this matrix; it can call into the existing parsing logic in `XTZ.Tezos.ts` and `SOL.Solana.ts` rather than duplicating it.

### Security notes (XTZ-specific add-ons to the SOL Stage-2 list)

- **Reusing baking/spending keys.** A Tezos `tz1` key is the user's **operation-signing** key on-chain. Reusing it for encryption is the same `crypto_sign_*_to_curve25519` trade-off mentioned in the SOL doc and is widely accepted in the wider Curve25519 ecosystem; document it in user-facing copy.
- **Prefix collisions are impossible.** Tezos hex-encoded Ed25519 pubkeys are 64 hex chars; SEC1 compressed/uncompressed are 66/130. The dispatcher cannot ambiguously route a key — there is no overlap.
- **`tz2` and `tz3` are *not* covered by this proposal.** `tz2` is secp256k1 and could plug into v1 if the derivation exposed SEC1 hex (a separate feature request); `tz3` (P-256) needs a third version byte and curve.

### Test plan (when implemented)

1. **Round-trip XTZ tz1**: encrypt a file with the derived hex pubkey from `tz1h1MhMCE4oh39oqJDg5XD1tew6F4LuxUCn`; decrypt with the `edsk…` from the JSON; bytes match.
2. **Cross-encoding**: encrypt to the same recipient, once with the **hex** pubkey and once with that pubkey re-encoded to **Base58** — both envelopes must decrypt with the same key.
3. **Cross-curve negative**: try to decrypt a v2 (`0x02`) envelope with a secp256k1 SEC1 seed and vice versa — must fail with a clear error, not WebCrypto's generic "operation failed".
4. **Wrong-key negative**: decrypt with a different `edsk…`; AES-GCM tag check must fail.
5. **Seed-input matrix**: cover all six rows of the table above; specifically confirm the 40-byte `edsk…` and the 72/68-byte extended forms.
6. **No regression**: BTC / ETH / DOGE round-trips on v1 still pass; SOL round-trip on v2 still passes.
7. **`tz2` / `tz3` (when supported)**: separate test plan; `tz2` should round-trip via v1 with format conversion; `tz3` requires a v3 envelope.
