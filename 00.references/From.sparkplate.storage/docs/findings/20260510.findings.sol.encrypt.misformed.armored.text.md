# 20260510 — Findings: SOL JSON + file encrypt → OpenPGP "Misformed armored text"

**Date:** Sunday, May 10, 2026 (`date +%Y%m%d` → `20260510` at write time)  
**Scope:** Encrypting a file after importing `00.References/CoreyStedman.sol.3BdhtQJPZ6vhWzfj6qQTUqSPdVyVNQ1BnBjTSjdaQyLu.json.json` (or any `SOL://…` URI whose derived public key is Base58 / Ed25519).  
**Outcome:** UI now detects **non–SPCK-compatible** derived keys (bridge flag `isSpckCompatible`) and **does not** send them to `openpgp.readKey`; users get an explanation instead of a cryptic armor parse error.

---

## Symptom

```
Error: Misformed armored text
```

Thrown from OpenPGP.js while encrypting, after pulling the **Native Cryptocurrency** public key into **Encrypt File** and clicking Encrypt.

---

## Root cause

1. **Native crypto file encryption (SPCK / ECIES)** in `encryption.cryptoPublicKey.ts` is implemented for **secp256k1** recipients only. The UI selects that path when `isHexCryptoPublicKey(trimmedKey)` is true — i.e. the string looks like **SEC1 hex** (compressed `02`/`03` + 64 hex chars, or uncompressed `04` + 128 hex chars).

2. **Solana** uses **Ed25519**. `SOL.Solana.ts` exposes `publicKey` and `address` as **Base58** (same string for both in typical derivation), not SEC1 hex. So `isHexCryptoPublicKey` is **false**.

3. **`App.vue` fallback** when the hex check fails is **OpenPGP**: `openpgp.readKey({ armoredKey: cleanPublicKey })` expects ASCII-armored **PGP** material (`-----BEGIN PGP PUBLIC KEY BLOCK-----…`). A Base58 Solana pubkey is neither SEC1 hex nor PGP armor, so the parser throws **Misformed armored text**.

The JSON file content is only a 32-byte secret encoded in Base58 (`privateKey` field); derivation is correct. The bug class is **wrong pipeline** (PGP parser fed non-PGP, non-SEC1 data), not a bad wallet file.

---

## Solution — Stage 1: stop the bogus armor error (shipped)

- **`useCryptocurrencyEngine`** (bridge) already computes **`isSpckCompatible`** (`isSec1HexPublicKey` on the derived `publicKey`).
- **`App.vue`** now:
  - destructures **`isSpckCompatible`**;
  - after **Pull publicKey**, warns when the pulled key is the derived crypto pubkey and **`!isSpckCompatible`**;
  - in **`encryptFile`**, before `openpgp.readKey`, if the encryption textarea matches the current derived **`cryptoPublicKey`** or **`cryptoWalletAddress`** and **`!isSpckCompatible`**, returns a **clear status message** instead of calling OpenPGP.

This stops the **"Misformed armored text"** error and tells the user *why* the pipeline can't run, but **does not yet make SOL keys usable as encryption recipients** — they still aren't a workable input to any encryption path in the app.

---

## Solution — Stage 2: a new crypto implementation for Ed25519 / Base58 keys

To actually let a SOL key encrypt files, we add a **second**, curve-aware path next to the existing secp256k1 SPCK code. The two paths share the magic header but use different curves and slightly different envelopes; the version byte distinguishes them.

### Curve choice: convert Ed25519 → X25519, then ECDH

Ed25519 is a **signing** curve. Doing ECDH directly on Ed25519 points is non-standard and library-fragile. The well-trodden path is to use the **birational map** between Ed25519 (twisted Edwards) and **X25519** (Montgomery) — they're the same Curve25519 underneath, with different encodings tuned for signing vs. ECDH:

| Side    | Conversion                                             | Library helper                                   |
|---------|--------------------------------------------------------|--------------------------------------------------|
| Public  | `xPub  = ed25519_pub_to_curve25519(edPub)`             | `libsodium.crypto_sign_ed25519_pk_to_curve25519` / `@noble/curves/ed25519` `edwardsToMontgomery` |
| Private | `xPriv = clamp(SHA-512(seed)[0..32])` (Solana 32-byte seed) | `libsodium.crypto_sign_ed25519_sk_to_curve25519` (needs the 64-byte form `seed ‖ pubkey`) |

Both `libsodium-wrappers-sumo` and `tweetnacl` are already declared in `package.json`; `@noble/curves` and `@noble/ed25519` are too. Any of them can supply the conversion; the recommendation is **libsodium** because the project already initializes it for other code paths.

### New envelope: SPCK v2

Keep one file format and one extension (`.spck`). **Version byte `0x02`** signals the X25519 layout; the decryptor branches on it.

| Bytes  | v1 (existing, secp256k1)                       | v2 (new, X25519 / Solana)                      |
|--------|------------------------------------------------|------------------------------------------------|
| 0..3   | `"SPCK"` magic                                 | `"SPCK"` magic                                 |
| 4      | `0x01`                                         | `0x02`                                         |
| 5..    | 33 B compressed secp256k1 ephemeral pubkey     | 32 B X25519 ephemeral pubkey (raw)             |
| next 12| AES-GCM IV                                     | AES-GCM IV                                     |
| rest   | AES-GCM(ciphertext ‖ tag)                      | AES-GCM(ciphertext ‖ tag)                      |

KDF: **HKDF-SHA256(shared, salt = ephemeralPub, info = "sparkplate-x25519-v1", 32)**. The distinct `info` string from v1 (`"sparkplate-ecies-v1"`) is **mandatory** as a domain separator so a single shared point can never decrypt across formats.

### New module: `encryption.ed25519PublicKey.ts`

```ts
export const SPCK_VERSION_X25519 = 0x02;
export const X25519_PUBKEY_LENGTH = 32;

export const isBase58Ed25519PublicKey: (value: string) => boolean;

export const encryptWithEd25519PublicKey: (
  recipientEd25519PublicKey: string | Uint8Array,
  plaintext: Uint8Array,
) => Promise<{ envelope: Uint8Array; ephemeralX25519Pub: Uint8Array; iv: Uint8Array }>;

export const decryptWithEd25519PrivateKey: (
  recipientEd25519Seed: string | Uint8Array,
  envelope: Uint8Array,
) => Promise<Uint8Array>;
```

Algorithm (mirrors `encryption.cryptoPublicKey.ts` line for line, only the curve and ephemeral-key length change):

```
encryptWithEd25519PublicKey(recipientEdPubB58, plaintext):
    edPub      = base58Decode(recipientEdPubB58)            // 32 B, Ed25519 point
    xPubPeer   = ed25519PubToCurve25519(edPub)              // 32 B
    {xPriv, xPubE} = randomX25519KeyPair()                  // ephemeral
    shared     = X25519(xPriv, xPubPeer)                    // 32 B
    aesKey     = HKDF-SHA256(shared, salt=xPubE, info="sparkplate-x25519-v1", 32)
    iv         = randomBytes(12)
    ct, tag    = AES-256-GCM(aesKey, iv, plaintext)
    envelope   = "SPCK" || 0x02 || xPubE || iv || ct || tag
    return envelope

decryptWithEd25519PrivateKey(recipientSeed, envelope):
    assert envelope[0..4] == "SPCK"
    assert envelope[4]    == 0x02
    xPubE      = envelope[5..37]                            // 32 B
    iv         = envelope[37..49]                           // 12 B
    ctTag      = envelope[49..]
    seed       = normalizeSolanaSeed(recipientSeed)         // 32 B (handles base58 / hex / json-array, 32 or 64)
    edFullSk   = seed || ed25519GetPublicKey(seed)          // 64 B for libsodium
    xPriv      = ed25519SkToCurve25519(edFullSk)            // 32 B
    shared     = X25519(xPriv, xPubE)
    aesKey     = HKDF-SHA256(shared, salt=xPubE, info="sparkplate-x25519-v1", 32)
    plaintext  = AES-256-GCM-Decrypt(aesKey, iv, ctTag)
    return plaintext
```

`normalizeSolanaSeed` reuses the same input matrix `SOL.Solana.ts` already accepts:

| Input         | Decoded length | Seed used         |
|---------------|----------------|-------------------|
| base58        | 32             | decoded           |
| base58        | 64             | decoded[0..32]    |
| hex           | 32             | decoded           |
| hex           | 64             | decoded[0..32]    |
| JSON array    | 32             | decoded           |
| JSON array    | 64             | decoded[0..32]    |

### Bridge change: communicate the curve

`bridge.Cryptography.Via.Currency.ts` should expose **two flags** (cleaner than overloading one):

```ts
export interface CryptocurrencyDerivation {
  // ... existing fields ...
  spckCurve: 'secp256k1' | 'x25519' | null;
  isSpckCompatible: boolean;       // true iff spckCurve !== null
}
```

`spckCurve` is set from the derived public key:

- `isSec1HexPublicKey(pub)` → `'secp256k1'`
- `isBase58Ed25519PublicKey(pub)` → `'x25519'`
- otherwise `null`

### `App.vue` dispatch

Replace the single hex check with an explicit dispatcher:

```ts
const trimmedKey = encryptionPublicKey.value.trim();

if (isHexCryptoPublicKey(trimmedKey)) {
  // existing v1 / secp256k1 path
} else if (isBase58Ed25519PublicKey(trimmedKey)) {
  const { envelope } = await encryptWithEd25519PublicKey(trimmedKey, fileData);
  const extensionResolution = resolveEncryptedExtension(
    trimmedKey, cryptoPublicKey.value, cryptoTickerSymbol.value,
  );
  downloadFile(
    envelope,
    buildEncryptedFilename(file.name, extensionResolution),
    'application/octet-stream',
  );
  encryptionStatus.value = `File encrypted with ${cryptoTickerSymbol.value || 'Ed25519'} public key (X25519 / SPCK v2)!`;
  return;
}

// PGP fallback only reached for actual armored input now
```

Decrypt mirrors this by branching on the **version byte** of the envelope, not on the key string.

### Security notes specific to v2

- **Always convert before ECDH.** Feeding raw Ed25519 bytes into `x25519.scalarMult` is a silent failure mode — no error, garbage shared secret.
- **Domain-separate the KDF.** `info = "sparkplate-x25519-v1"` must differ from `"sparkplate-ecies-v1"` to prevent cross-format pivoting.
- **Reuse signing key for encryption.** This is the same trade-off `crypto_sign_*_to_curve25519` exists to support; document it in user-facing copy.
- **Authenticate.** AES-GCM (or libsodium's sealed box, if used as an alternative) is a must — never strip the tag.
- **Canonical Base58.** Reject any decoded length other than 32 bytes for SOL pubkeys.

### Test plan (when implemented)

1. Round-trip: encrypt a file with the SOL Base58 pubkey from the reference JSON, decrypt with the JSON's `privateKey` (32-byte seed); bytes equal.
2. Cross-curve negative: try to decrypt a v2 envelope with a secp256k1 seed (and vice versa) — must fail with a clear error, not WebCrypto's generic "operation failed".
3. Wrong-key negative: decrypt with a different SOL seed; AES-GCM tag check must fail.
4. Seed-input matrix: cover all six rows of the table above.
5. Filename: extension still `.spck`; ticker tag `<base>.SOL.spck` via existing `resolveEncryptedExtension`.
6. v1 regression: BTC / ETH / DOGE round-trips still pass.

### Out of scope

- **sr25519 / Ristretto255** (Polkadot) — would need its own version byte (`0x03`) and ECIES.
- **Hybrid multi-recipient** envelopes — orthogonal to curve choice.
- **Hardware-wallet decryption** of v2 — most devices don't expose `ed25519 → x25519` ECDH.
