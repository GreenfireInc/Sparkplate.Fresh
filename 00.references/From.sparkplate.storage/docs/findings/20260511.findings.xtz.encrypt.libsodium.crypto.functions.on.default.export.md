# XTZ encrypt error: `sodium.crypto_sign_ed25519_pk_to_curve25519 is not a function`

**Date:** Monday, May 11, 2026 (`date +%Y%m%d` → `20260511`)
**Status:** Resolved
**Severity:** High — blocks every SPCK v2 (X25519) encrypt/decrypt operation in the browser, even though all dependencies are installed correctly.

---

## Symptom

After the SPCK v2 / X25519 file-encryption pipeline landed (see [`docs/progress/20260511.sparkplatestorage.SolXtzEd25519X25519Encryption.md`](../progress/20260511.sparkplatestorage.SolXtzEd25519X25519Encryption.md)), encrypting a file with a Tezos `tz1` public key surfaced:

```
Error: sodium.crypto_sign_ed25519_pk_to_curve25519 is not a function
```

The same error is reachable for Solana (Base58 Ed25519 pubkey) — both code paths route through `encryptWithEd25519PublicKey()` → `ed25519PubToCurve25519()` → libsodium.

The Node round-trip script (`scripts/test-ed25519-roundtrip.mjs`) does **not** exhibit the failure because it accesses `sodiumMod.default.crypto_sign_ed25519_pk_to_curve25519`; the production module under `src/` was reaching for `sodiumMod.crypto_sign_ed25519_pk_to_curve25519`.

---

## Root Cause

`libsodium-wrappers-sumo@0.8.4` ships a tiny ESM wrapper (`dist/modules-sumo-esm/libsodium-wrappers.mjs`). Its **named exports** are only the small set of formatter / helper functions:

```js
export {
  s as ready,
  o as add,
  l as compare,
  x as from_base64, b as from_hex, v as from_string,
  p as increment, h as is_zero, i as memcmp, y as memzero,
  u as pad, d as unpad,
  c as symbols,
  E as to_base64, f as to_hex, g as to_string,
  T as output_formats, m as base64_variants,
  r as libsodium,
};
export default t;
```

Every `crypto_*` primitive — including the three we need for SPCK v2 (`crypto_sign_ed25519_pk_to_curve25519`, `crypto_sign_ed25519_sk_to_curve25519`, `crypto_sign_seed_keypair`) — is attached at runtime onto **`t` (the default export object)**, not re-exported as a named binding. The Emscripten loader walks the WASM module's symbol table and assigns each `_crypto_*` function to `t.crypto_*` once `ready` resolves.

When code does:

```ts
const mod = await import('libsodium-wrappers-sumo');
await mod.ready;                            // works — `ready` IS a named export
mod.crypto_sign_ed25519_pk_to_curve25519(…) // ✗ undefined → "is not a function"
```

the named-binding lookup misses (`crypto_*` is *not* in the export map), so:

- **Vite (strict ESM dev server / esbuild prebundle):** `mod.crypto_*` evaluates to `undefined`. **Error.**
- **Node ESM dynamic import:** same — `mod.crypto_*` is `undefined`.
- **Legacy bundlers with relaxed CJS↔ESM interop** (older Webpack `esModuleInterop`, some ts-jest configs): may silently forward unknown property access to `mod.default`, in which case the bug is hidden.

The correct invocation is:

```ts
const mod = await import('libsodium-wrappers-sumo');
const sodium = mod.default ?? mod;          // prefer default; fall back to ns
await sodium.ready;
sodium.crypto_sign_ed25519_pk_to_curve25519(…); // ✓
```

This matches:

- libsodium's own [README example](https://github.com/jedisct1/libsodium.js#usage-from-modules) (`const sodium = require('libsodium-wrappers-sumo'); await sodium.ready;` — CJS, so `require` returns the default object directly).
- Our `scripts/test-ed25519-roundtrip.mjs`, which already uses `sodiumMod.default`.

### Why the existing `XTZ.Tezos.ts` keystore decryption appeared to work

`src/lib/cores/currencyCore/currencies/XTZ.Tezos.ts:338-372` uses the same antipattern (`const sodium = await import('libsodium-wrappers-sumo'); await sodium.ready; sodium.crypto_pwhash(…)`). It is a **latent bug**: the path is gated behind uploading an encrypted Tezos JSON keystore, which appears to be untested under Vite. Static analysis says it will fail identically on `sodium.crypto_pwhash is not a function` the first time someone exercises it. We patch both call sites in this fix.

### Why the type system didn't catch it

The ESM type declaration `dist/modules-sumo-esm/libsodium-wrappers.d.mts` (2 783 lines) declares only the named-exported helpers. The CJS sibling `dist/modules-sumo/libsodium-wrappers.d.ts` (2 191 lines) *does* declare every `crypto_*` function as a named export, which is what Node-resolving TypeScript picks up under `"moduleResolution": "node"`. Result: `tsc` reports the type as having `crypto_sign_ed25519_pk_to_curve25519`, even though the **ESM runtime** does not. Bundler-resolution mode (`"moduleResolution": "bundler"`, which our `tsconfig.app.json` uses) would expose the inconsistency, but the project picks up `node`-style resolution for this package via its `main` field.

## Fix

### 1. `src/lib/cores/cryptographyCore/encryption.ed25519PublicKey.ts`

Make the lazy loader resolve the runtime object (default export first, namespace fallback) and type it explicitly with the three primitives we actually call. This frees us from depending on whichever `.d.ts` variant TypeScript chooses.

```ts
// Minimal runtime surface of libsodium-wrappers-sumo that this module uses.
// The ESM build of libsodium attaches every `crypto_*` function to the default
// export object after `ready` resolves; they are NOT re-exported as named
// bindings. Typing only the functions we touch keeps the surface honest.
interface SodiumRuntime {
  ready: Promise<void>;
  crypto_sign_ed25519_pk_to_curve25519: (pk: Uint8Array) => Uint8Array;
  crypto_sign_ed25519_sk_to_curve25519: (sk: Uint8Array) => Uint8Array;
  crypto_sign_seed_keypair: (seed: Uint8Array) => {
    publicKey: Uint8Array;
    privateKey: Uint8Array;
    keyType: 'ed25519';
  };
}

let sodiumPromise: Promise<SodiumRuntime> | null = null;
const loadSodium = async (): Promise<SodiumRuntime> => {
  if (!sodiumPromise) {
    sodiumPromise = (async () => {
      const mod = (await import('libsodium-wrappers-sumo')) as unknown as {
        default?: SodiumRuntime;
      } & Partial<SodiumRuntime>;
      // ESM build: `crypto_*` lives only on `mod.default`. Some bundlers/test
      // runners forward namespace access to default; we don't rely on that.
      const sodium = (mod.default ?? (mod as unknown)) as SodiumRuntime;
      await sodium.ready;
      return sodium;
    })();
  }
  return sodiumPromise;
};
```

### 2. `src/lib/cores/currencyCore/currencies/XTZ.Tezos.ts` (defensive — same latent bug)

Inside the JSON-keystore decryption block, do exactly the same dance:

```ts
const sodiumMod = await import('libsodium-wrappers-sumo');
// libsodium-wrappers-sumo's ESM build exposes `crypto_*` only on its default
// export; see docs/findings/20260511.findings.xtz.encrypt.libsodium.crypto
// .functions.on.default.export.md.
// eslint-disable-next-line @typescript-eslint/no-explicit-any
const sodium = (sodiumMod as any).default ?? (sodiumMod as any);
await sodium.ready;
// …existing crypto_pwhash / crypto_secretbox_open_easy calls unchanged
```

No call-site changes are needed inside the block — `sodium.crypto_pwhash(…)` etc. continue to work because `sodium` now points at the same object that holds every `crypto_*` function.

## Why we didn't catch this in the verification step

The Node round-trip script was deliberately written to mirror the *production primitives* (so its assertion is "the algorithm is correct"), but it used `sodiumMod.default` from the start — under the working hypothesis that the source module did the same. The source module did not. The lesson for next time:

- **The integration test must import the production module**, not re-implement it. Re-implementing the algorithm caught the cryptography, but not the wiring.
- For modules that depend on a package with WASM bootstrap and CJS/ESM interop quirks (libsodium, `tiny-secp256k1`, `@polkadot/wasm-*`), require a smoke-import test under the actual bundler.

When the SPCK v3 / sr25519 work lands, the integration test will instantiate the project's compiled module directly (via a Vite-built bundle or via `vitest` running through Vite's transform pipeline) so this class of "named export vs. default export" gap fails at CI time, not in the browser.
