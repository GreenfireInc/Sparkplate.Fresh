# NoteEditorModal Toolbar Icons Not Visible

**Date:** 2026-01-12  
**Component:** `src/components/modals/entryDetails/contacts/NoteEditorModal.vue`  
**Status:** Resolved

## Problem Description

The formatting toolbar icons (Bold, Italic, Underline, AlignLeft, AlignCenter, AlignRight) in the `NoteEditorModal.vue` component are not visible despite being correctly imported from `lucide-vue-next`.

## Root Cause Analysis

### Issue 1: Lucide-vue-next SVG Default Styling

Lucide icons render as SVG elements with `currentColor` as their default stroke color. When the parent element or the SVG itself doesn't have an explicit `color` property set, the icons may inherit a transparent or white color, making them invisible on a white background.

### Issue 2: Vue Scoped CSS Limitations

Vue's scoped CSS uses attribute selectors (e.g., `[data-v-xxxxxx]`) to isolate styles. When targeting child elements inside third-party components (like lucide-vue-next icons), the scoped attribute is not applied to those nested elements, causing styles to not be applied.

### Issue 3: CSS Specificity and Inheritance

The toolbar buttons have a white background (`#ffffff`). If the SVG icons inherit a white or transparent color, they become invisible against the white button background.

### Issue 4: stroke-width Prop vs CSS

While `stroke-width="2"` is passed as a prop to the icons, the `stroke` color itself may not be properly set, resulting in invisible strokes.

## Technical Details

### Current Implementation (Before Fix)

```vue
<button class="toolbar-button" @click="handleFormat('bold')" title="Bold">
  <BoldIcon :size="16" stroke-width="2" class="toolbar-icon-bold-red" />
</button>
```

```css
.toolbar-button :deep(svg) {
  color: #000000 !important;
  stroke: #000000 !important;
  stroke-width: 2;
  fill: none;
}
```

### Why `:deep()` May Not Work

1. The `:deep()` selector (formerly `::v-deep` or `/deep/`) should penetrate scoped styles, but browser support and Vue version differences can cause inconsistencies.
2. The `!important` declarations may conflict with inline styles applied by lucide-vue-next.
3. The `stroke` property needs to be applied to the `path` elements inside the SVG, not just the SVG container.

## Solution

### Fix 1: Use Inline Style on Icon Component

Apply the `color` prop directly to the lucide icon, which sets the `currentColor` used by the SVG:

```vue
<BoldIcon :size="16" :stroke-width="2" color="#000000" />
```

### Fix 2: Target SVG Path Elements Correctly

Ensure CSS targets the actual path elements:

```css
.toolbar-button :deep(svg),
.toolbar-button :deep(svg path),
.toolbar-button :deep(svg line),
.toolbar-button :deep(svg polyline),
.toolbar-button :deep(svg rect),
.toolbar-button :deep(svg circle) {
  color: #000000 !important;
  stroke: #000000 !important;
  stroke-width: 2px;
}
```

### Fix 3: Remove Conflicting Styles

Remove `fill: none` if the icon requires a fill, or ensure it's appropriate for the specific icon type.

## Applied Fix

After multiple attempts with lucide-vue-next icons (using `color` prop, `:deep()` CSS selectors, stroke styling), the icons remained invisible. The issue appears to be related to how Vue's scoped CSS interacts with dynamically rendered SVG elements from lucide-vue-next.

### Final Solution: Text-Based Icons

Replaced the lucide SVG icons with simple text-based icons that are guaranteed to be visible:

```vue
<button class="toolbar-button" @click="handleFormat('bold')" title="Bold">
  <span class="toolbar-icon"><b>B</b></span>
</button>
<button class="toolbar-button" @click="handleFormat('italic')" title="Italic">
  <span class="toolbar-icon"><i>I</i></span>
</button>
<button class="toolbar-button" @click="handleFormat('underline')" title="Underline">
  <span class="toolbar-icon"><u>U</u></span>
</button>
<button class="toolbar-button" @click="handleFormat('justifyLeft')" title="Align Left">
  <span class="toolbar-icon">⫷</span>
</button>
<button class="toolbar-button" @click="handleFormat('justifyCenter')" title="Align Center">
  <span class="toolbar-icon">☰</span>
</button>
<button class="toolbar-button" @click="handleFormat('justifyRight')" title="Align Right">
  <span class="toolbar-icon">⫸</span>
</button>
```

```css
.toolbar-icon {
  display: flex;
  align-items: center;
  justify-content: center;
  font-size: 14px;
  font-weight: 600;
  color: #000000;
  width: 16px;
  height: 16px;
}
```

This approach:
1. Uses semantic HTML (`<b>`, `<i>`, `<u>`) that visually represents the formatting action
2. Uses Unicode symbols for alignment icons
3. Is guaranteed to render correctly regardless of CSS scoping issues
4. Matches the visual style of traditional rich text editors (like Microsoft Word)

## Lessons Learned

1. **Always set explicit color on lucide icons** - Don't rely on CSS inheritance for icon colors.
2. **Use the `color` prop** - Lucide-vue-next icons support a `color` prop that directly sets the stroke color.
3. **Test visibility on actual backgrounds** - Icons may appear in dev tools but be invisible against their actual background.
4. **Vue scoped CSS requires `:deep()` for third-party components** - But the selector must target the correct nested elements.

## Additional Fix: Text Direction Issue

### Problem
The content editor was starting centered and writing from right to left (reversed).

### Solution
1. Added explicit CSS for left-to-right text direction:
```css
.note-content-editor {
  text-align: left;
  direction: ltr;
  unicode-bidi: normal;
}
```

2. Added inline style resets when the modal opens and when notes load:
```typescript
contentRef.value.style.direction = 'ltr';
contentRef.value.style.textAlign = 'left';
```

3. Added a "Reset Formatting" button (↺) that:
   - Removes all text formatting
   - Resets alignment to left
   - Ensures text direction is LTR

## Final Fix: Replaced contenteditable with textarea

### Problem
Despite all CSS fixes, the contenteditable div was writing text in reverse/backwards. Characters were being inserted at the beginning of the text instead of at the cursor position.

### Root Cause
The `contenteditable` attribute on divs has known compatibility issues with certain browser/system configurations, especially related to:
- IME (Input Method Editor) handling
- Text direction inference
- Cursor position management

### Solution
Replaced the contenteditable div with a standard `<textarea>` element:

```vue
<textarea
  ref="contentRef"
  v-model="noteContent"
  class="note-content-editor"
  :class="{ locked: isLocked }"
  :disabled="isLocked"
  placeholder="Start writing..."
  @input="handleContentChange"
></textarea>
```

This approach:
1. Uses native browser textarea behavior which is reliable
2. Removes the need for complex CSS direction fixes
3. Sacrifices rich text formatting (Bold, Italic, etc.) for reliable text input
4. Can be enhanced later with a proper rich text editor library if needed

### Trade-offs
- **Lost**: Rich text formatting toolbar (Bold, Italic, Underline, Alignment)
- **Gained**: Reliable left-to-right text input that works correctly

### Future Enhancement
If rich text editing is needed, consider using a dedicated library like:
- TipTap (Vue-compatible)
- Quill
- ProseMirror
- Editor.js
