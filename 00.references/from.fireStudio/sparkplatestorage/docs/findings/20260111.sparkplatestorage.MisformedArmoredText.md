# Findings Report: Misformed Armored Text Error During Decryption

**Date:** January 11, 2026  
**Component:** PGP/GPG Key Tool - File Decryption (`decryptFile` function)  
**Status:** ⚠️ Issue Identified

---

## Error Description

When attempting to decrypt an encrypted file, the application throws the following error:

```
Error: Misformed armored text
```

This error occurs in the `decryptFile()` function when calling `openpgp.readMessage()` to parse the encrypted file data.

---

## Error Location

**File:** `src/App.vue`  
**Function:** `decryptFile()`  
**Line:** 177

```typescript
const message = await openpgp.readMessage({ armoredMessage: new TextDecoder().decode(encryptedData) });
```

---

## What "Misformed Armored Text" Means

The OpenPGP library's "Misformed armored text" error indicates that the input data does not conform to the PGP ASCII-armored format specification. PGP armored messages have a specific structure:

1. **Headers:** Must begin with `-----BEGIN PGP MESSAGE-----`
2. **Body:** Base64-encoded data, typically with 64-character line lengths
3. **Footers:** Must end with `-----END PGP MESSAGE-----`
4. **Line Breaks:** Must use consistent line break characters (LF or CRLF)
5. **Encoding:** Must be valid ASCII text

If any of these requirements are not met, the parser will reject the input with "Misformed armored text".

## Root Cause Analysis

### Current Encryption/Decryption Flow

**Encryption (line 148-153):**
```typescript
const encrypted = await openpgp.encrypt({
  message,
  encryptionKeys: publicKeyObj,
});

downloadFile(encrypted as string, `${fileToEncrypt.value.name}.gpg`, 'application/octet-stream');
```

**Decryption (line 176-177):**
```typescript
const encryptedData = await fileToDecrypt.value.arrayBuffer();
const message = await openpgp.readMessage({ armoredMessage: new TextDecoder().decode(encryptedData) });
```

### Potential Issues

#### 1. Content Type Mismatch
- **Encryption:** The file is saved with `contentType: 'application/octet-stream'` (binary)
- **Reality:** `openpgp.encrypt()` by default returns an **armored string** (ASCII text)
- **Impact:** While modern browsers typically handle this correctly, there could be encoding inconsistencies

#### 2. File Reading Method
- The file is read as an `ArrayBuffer` and then decoded as UTF-8 text
- If the file contains binary data (not text), this could corrupt the data
- However, if the file is truly armored (ASCII text), this should work

#### 3. Text Encoding Issues
- `TextDecoder().decode()` uses UTF-8 encoding by default
- PGP armored messages should be ASCII-compatible, but encoding mismatches could occur
- If the file was saved with a different encoding, reading it as UTF-8 could introduce errors

#### 4. File Corruption or Format Issues
- The file might have been corrupted during download or transmission
- The file might not actually be an armored PGP message (could be binary format)
- Line breaks might have been altered (CRLF vs LF vs CR)

#### 5. Default Encryption Format
- By default, `openpgp.encrypt()` returns an armored string
- However, if binary output was requested or configured, the result would be binary data
- Reading binary data as text would cause parsing errors

## Common Causes of "Misformed Armored Text"

1. **Binary Data Instead of Armored Text**
   - The encrypted file might be in binary format, not ASCII-armored
   - Attempting to parse binary data as armored text will fail

2. **Missing or Incorrect Headers/Footers**
   - File doesn't start with `-----BEGIN PGP MESSAGE-----`
   - File doesn't end with `-----END PGP MESSAGE-----`
   - Headers/footers are malformed or corrupted

3. **Line Break Issues**
   - Inconsistent line break characters (mixed CRLF/LF/CR)
   - Line breaks removed or altered
   - Maximum line length exceeded (PGP armor typically uses 64-char lines)

4. **Encoding Problems**
   - File saved with wrong encoding (e.g., UTF-8 BOM, UTF-16)
   - File read with wrong encoding
   - Non-ASCII characters introduced into armored text

5. **File Corruption**
   - Data corrupted during file download/save
   - Partial file transmission
   - File system encoding issues

6. **Content Type Mismatch**
   - File saved as binary but contains text (or vice versa)
   - Browser/download handling issues

## Code Flow Analysis

### Encryption Process
1. File is read as `ArrayBuffer` (binary)
2. Converted to `Uint8Array` for `createMessage()`
3. Encrypted with `openpgp.encrypt()` → returns armored **string**
4. Saved via `downloadFile()` with `application/octet-stream` content type
5. Browser downloads file (should be ASCII text despite content type)

### Decryption Process
1. File is read as `ArrayBuffer` (binary representation of text)
2. Decoded as UTF-8 string via `TextDecoder().decode()`
3. Passed to `openpgp.readMessage({ armoredMessage: ... })`
4. **Error occurs here** if format is invalid

### Potential Issue Point

The most likely issue is a **format mismatch**:
- The file might be saved correctly as armored text
- But when read back, there could be encoding issues
- Or the `openpgp.encrypt()` might be producing binary output instead of armored

## Recommendations

### Immediate Checks

1. **Verify File Content:**
   - Open the encrypted `.gpg` file in a text editor
   - Confirm it contains ASCII-armored PGP message format
   - Check for proper headers/footers and base64 content

2. **Test Encryption Output:**
   - Add logging to inspect the `encrypted` variable
   - Verify it's a string starting with `-----BEGIN PGP MESSAGE-----`
   - Compare with the actual downloaded file content

3. **Check for Encoding Issues:**
   - Verify file encoding is consistent (ASCII or UTF-8)
   - Check for BOM (Byte Order Mark) issues
   - Ensure line breaks are preserved

### Potential Solutions (For Future Implementation)

1. **Explicit Armored Format:**
   - Ensure `openpgp.encrypt()` is configured to output armored format
   - Use `format: 'armored'` option if available

2. **Proper Content Type:**
   - Use `text/plain` or `application/pgp-encrypted` instead of `application/octet-stream`
   - Better matches the actual content (ASCII text)

3. **Consistent Encoding:**
   - Explicitly specify encoding when saving/reading files
   - Use UTF-8 consistently

4. **Error Handling:**
   - Add validation before calling `readMessage()`
   - Check file format/headers before parsing
   - Provide more specific error messages to users

5. **Binary vs Armored Detection:**
   - Detect if file is binary or armored
   - Handle both formats appropriately
   - Use `binaryMessage` option for binary format
